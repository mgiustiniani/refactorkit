# Migrating from v0.5.0 to v0.6.2

## Compatibility

API compatibility remains `0.2`. Existing documented Java CLI, daemon, LSP and
MCP workflows remain available. Journal recovery should be allowed to finish
before replacing an executable, as with previous releases.

## TypeScript/JavaScript semantic operations

`v0.6.2` publishes the completed `v0.6.0` feature scope and adds an experimental explicit-toolchain semantic adapter. It never
searches PATH, npm global storage or workspace `node_modules` unless the caller
selects the corresponding opt-in policy. CLI callers provide:

```text
--node <node executable>
--language-server-package <typescript-language-server package root>
--typescript-package <typescript package root>
```

The managed `v0.6.2` support row is Node 22.18.0,
`typescript-language-server` 5.1.3 and TypeScript 5.9.3. See
[`v0.6.0-support-matrix.md`](v0.6.0-support-matrix.md).

Example read operation:

```bash
refactorkit typescript search /workspace \
  --query UserService \
  --node /tools/node \
  --language-server-package /tools/node_modules/typescript-language-server \
  --typescript-package /tools/node_modules/typescript
```

Rename is preview-only unless `--apply` is supplied. Managed apply uses exact
compiler diagnostics, explicit authorization, WAL and rollback:

```bash
refactorkit typescript rename /workspace \
  --file src/UserService.ts --line 1 --character 13 --to AccountService \
  --node /tools/node \
  --language-server-package /tools/node_modules/typescript-language-server \
  --typescript-package /tools/node_modules/typescript

# Repeat with --apply only after reviewing the JSON preview.
refactorkit patch rollback <transaction-id> --root /workspace
```

## Symbol identifiers

Semantic TypeScript/JavaScript symbols use opaque `lsp-symbol-v1:<sha256>` IDs.
Do not parse them or persist the old prototype location-shaped IDs. Obtain current
IDs through `symbol.search`; IDs remain stable across sessions and unrelated line
movement under the same toolchain provenance, but change after declaration rename,
move, hierarchy or signature changes.

## Diagnostics and LSP ownership

Managed TypeScript diagnostics come from `typescript-compiler-exact-v1`, not from
unversioned `publishDiagnostics` inference. The provider uses a source/config
overlay, no-emit compiler analysis, bounded output and snapshot attestation.

The RefactorKit LSP server does not replace an editor's native TypeScript language
client. Standard TypeScript/JavaScript LSP edits remain client-managed and have no
RefactorKit WAL. Use CLI, daemon or MCP managed operations when rollback is required.

## Capability discovery

Clients should refresh `capabilities` and inspect per-operation backend, runtime,
evidence, extension set and mutation authority. TypeScript diagnostics now report
compiler evidence; semantic rename remains experimental and external-proposal
backed even though managed integration surfaces can apply accepted plans safely.

## Known migration boundaries

- Package-based `extends` remains refused without a hash-bound package trust policy.
- Dynamic/mixed JavaScript cannot obtain the managed diagnostics gate.
- File/type move, organize imports, change signature, extract/inline and migration
  recipes are post-`v0.6.0` scope.
- The qualified toolchain is external and must be installed separately without
  relying on RefactorKit to run a package manager.
- No journal schema conversion is required from `v0.5.0`.
