# v0.7.0 bidirectional Java/Kotlin public type rename acceptance

Status: local bidirectional packaged and four-platform native acceptance complete.
Kotlin-to-Java passed run `29457693861`; Java-to-Kotlin passed run `29482085130`.

## Kotlin declaration used by Java

A public top-level Kotlin type can be renamed when K2 compilation yields its exact
JVM binary name and disposable class output lets JDT resolve exact Java binding
uses. The output never leaves the immutable overlay.

## Java declaration used by Kotlin

A public top-level Java class can be renamed when:

- JDT proves its source declaration and Java references;
- bounded ECJ compilation runs externally with `-proc:none`, a 30-second deadline,
  bounded output, source levels 8–21 and no project build/plugin execution;
- deterministic classes are packaged as a temporary hash-bound JAR;
- K2 resolves exact Kotlin type and constructor tokens against that JAR;
- staged ECJ, K2 and JDT evidence resolves the new binary name;
- the Java source filename is renamed collision-free in the same plan.

Both directions require explicit `acceptExternalConsumerRisk=true` (CLI:
`--accept-external-consumer-risk`). Dynamic quoted names, generated ownership,
ambiguous identities, dirty compiler baselines and missing usages refuse.

## Managed transaction evidence

Daemon and MCP apply select the direction-specific combined diagnostics gate. All
Java/Kotlin text edits and optional Java filename rename pass through one
`PatchEngine` transaction with preview, explicit approval, WAL and rollback.
