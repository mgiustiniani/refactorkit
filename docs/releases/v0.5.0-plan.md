# RefactorKit v0.5.0 Platform Release Plan

Status: active.

Baseline: published `v0.4.0`; main `0.5.0-SNAPSHOT`; API `0.2` until an explicit
migration is accepted.

## Objective

`v0.5.0` begins the production multi-language kernel and expands the
self-contained runtime from Linux-only publication to native Windows and macOS
artifacts. Cross-platform packaging is not considered complete until managed
preview/apply/WAL/recovery/rollback semantics pass natively, not merely until
`--help` starts.

## P1 — Native self-contained runtime matrix

Target assets:

| Platform | Runner/build architecture | Asset suffix |
|---|---|---|
| Linux | x86_64 | `linux-x86_64` |
| Windows | x86_64 | `windows-x86_64` |
| macOS | Intel x86_64 | `macos-x86_64` |
| macOS | Apple Silicon arm64 | `macos-aarch64` |

Requirements:

- build each `jlink` image on a native GitHub runner;
- include only the packaged runtime and never require global Java at execution;
- provide native shell/batch launchers and path-with-spaces/Unicode tests;
- run full build/tests plus packaged version, signed definition/reference,
  format/apply/rollback and source-restoration smoke on every platform;
- generate a platform checksum and SPDX JSON SBOM;
- publish build-provenance and SBOM attestations per runtime;
- aggregate all assets into one immutable GitHub Release;
- document PowerShell, Intel macOS and Apple Silicon installation/verification.

## P2 — Filesystem and transaction parity

Before Windows or macOS is marked supported for writes:

- introduce platform-aware durability capability/strategy where Java NIO differs;
- prove atomic replacement and the qualified directory durability contract;
- cover POSIX, ACL, DOS attributes and unsupported-view behavior;
- detect filesystem case sensitivity and reject colliding/case-only paths safely;
- cover Windows drive/UNC/long paths, junctions/reparse points and open-handle
  replacement behavior;
- cover macOS APFS, default case-insensitive volumes, ACL/xattr and symlinks;
- add native process-kill/restart, journal corruption, rollback conflict and
  recovery acceptance;
- preserve explicit refusal when a filesystem cannot meet the reported contract.

Packaging may be published as preview/read-only for a platform only if the asset
and README say so explicitly. Full support requires managed-write parity.

## P2A — Java 21 Maven reactor and ownership relocation

Status: complete on main. Normative decision: ADR 0010.

- [x] Embedded, plugin-free Maven effective-model discovery, offline by default.
- [x] Inherited Java 21, BOM/dependency-management, reactor edge, local artifact,
  and main/test/generated source-set analysis with hash-bound evidence.
- [x] Concise typed `buildModel.unavailable`, `classpath.unavailable`, and
  `sourceLevel.unavailable` roots instead of default classpath cascades.
- [x] `moveSourceRoot` rename-only preview/apply/WAL/rollback across Maven modules.
- [x] Committed `samples/java-maven-reactor-21` and read-only Magrathea acceptance.
- [x] Library, CLI, daemon, and MCP exposure with portable paths and typed
  relocation refusals.

Acceptance evidence:

- implementation commit `a368f54970597c6821bd1eacd17d37b7e9a2ae7c`;
- portable CLI follow-up `7edcf36c213cf928ec0dea6379aa4d7b66db1c32`;
- 433 unit/integration tests and 23/23 golden cases;
- packaged Magrathea scan: five modules, Java 21 effective model/classpath
  available in every module, zero diagnostics, cross-module references, and no
  workspace mutation;
- native CI run `29205094718`: Linux, Windows x86_64, macOS x86_64, and macOS
  arm64 green.

Completion of P2A does not by itself qualify the full `v0.5.0` platform release.
P1, P2, P2B, P3, and P4 retain independent gates.

## P2B — JVM build-model productionization

Status: in progress; `0.5.x` release gate before deep JVM adapter expansion.

- [x] Add internal language-neutral core contracts for `BuildModelProvider`,
  discovery policy, `BuildModel`, modules, source sets, scoped module edges,
  outputs, generated roots, typed model diagnostics, and status.
- [x] Bind build-model projections into `ProjectSnapshot` hashing while retaining
  API `0.2` `Module` compatibility.
- [x] Project current Java discovery through explicit `maven-effective-v1`,
  `gradle-declarative-v1`, and `java-conventional-v1` providers and expose
  redacted daemon/MCP summaries.
- [x] Migrate JDT parser environments (main/test roots, generated roots,
  classpaths, source levels, and transitive module visibility) to provider source
  sets, with compatibility fallback for snapshots without build models.
- [ ] Migrate remaining ownership/planner consumers from compatibility `Module`
  fields incrementally.
- [x] Retain the SPI inside RefactorKit rather than creating another
  service/product.
- [ ] Harden Maven mediation/classifier/test-JAR/system-path/profile/custom-source-set
  coverage and define repository/checksum policy;
- [ ] Define a separate credential-safe design before supporting private or
  authenticated repositories; Maven settings remain unread by default;
- [ ] Select a Gradle effective-model strategy with an explicit arbitrary-code
  execution policy—no silent Tooling API/build-script execution;
- [ ] Model Gradle main/test/integration/custom source sets and generated outputs with
  parity to the Maven evidence boundary;
- [ ] Complete build-model capability schemas and contract snapshots so integrations can
  distinguish available, partial, offline-missing, and execution-refused models;
- [ ] Add native packaged Maven reactor scan/diagnostics and `moveSourceRoot`
  preview/apply/WAL/rollback acceptance on every supported OS;
- [ ] Prove case-folded relocation collisions on Windows and default
  case-insensitive macOS filesystems.

P2B must not broaden generated-source write authority or permit build-model
providers to write outside `PatchEngine`.

## P3 — Distribution trust

- evaluate macOS Developer ID signing/notarization and document Gatekeeper/
  quarantine behavior until signed assets exist;
- evaluate Windows Authenticode/SmartScreen behavior;
- keep ZIP publication available independently of installer work;
- verify architecture, archive layout, executable permissions, embedded modules,
  checksums, SBOM and attestations after independent download.

## P4 — Multi-language kernel

Continue the roadmap foundation in parallel:

- language adapter registry and mixed-language routing;
- internal Build Model SPI and source-set/module graph extracted from the P2A
  Maven reference implementation;
- generalized semantic evidence;
- bounded external semantic-process lifecycle;
- real structural parser bindings;
- untrusted external workspace-edit normalization;
- capability schemas, timeout/cancellation, process provenance and limits.

No language adapter inherits stable mutation authority from platform packaging.

## Acceptance

`v0.5.0` requires green native matrix jobs, platform-specific managed transaction
acceptance, completed P2B build-model gates, native packaged Maven reactor and
source-root relocation acceptance, all existing 23+ golden cases, Linux
regression smoke, reproducible per-platform archives, release
asset/SBOM/attestation verification, updated
README/packaging/migration documentation, and no hidden global runtime dependency.
