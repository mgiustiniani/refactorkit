## Tag workflow gates

The immutable `v0.4.0` tag workflow must still prove:

- branch and tag CI green;
- final implementation metadata equals `0.4.0`;
- runtime zip and checksum publication;
- SPDX JSON SBOM generation;
- GitHub build provenance and SBOM attestations;
- release classified as non-prerelease/latest;
- injected release tag, commit, asset URL and checksum metadata;
- independent download/checksum/extract/version/help/scan/format smoke;
- local deployment records the verified artifact hash.

A failure does not permit moving the tag. Correct on a new commit and publish a
new patch release if the tag already exists.

## Qualified limitations

- API remains beta baseline `0.2`; stable API `1.0` is deliberately deferred.
- Linux x86_64 is the packaged runtime target.
- Native LSP formatting is client-managed and not transaction-backed; managed
  formatting uses preview plus `refactorkit.applyPlan`.
- Whole-file Java formatting is implemented; range/changed-lines formatting is
  later pre-1.0 scope.
- Synchronous API `0.2` transports do not advertise in-flight cancellation.
- External build scripts are not silently executed.
- Reflection, strings/configuration, generated source, incomplete classpaths and
  external consumers retain documented warning/refusal boundaries.
