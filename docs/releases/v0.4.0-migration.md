# Migrating from RefactorKit v0.3.0 to v0.4.0

`v0.4.0` keeps integration API baseline `0.2`. Existing documented preview,
apply, rollback, scan, symbol, definition, reference, diagnostics, and Java
refactoring requests remain compatible.

## New functionality

- `refactorkit format-file <file> [--apply] [--root <path>]` formats one Java
  compilation unit through Eclipse JDT.
- Daemon/MCP generic preview supports `operation=formatFile` with
  `arguments.file`.
- LSP advertises `textDocument/formatting` for client-managed edits and
  `refactorkit.formatFile` for preview/managed-apply workflows.
- Java diagnostics now include stable compiler evidence/category/range fields.
- Plans expose structured refactoring evidence; lexical fallback is review-only.

## CLI boolean option correction

Boolean options are now parsed independently of following positional arguments:

```text
--apply --force --stdin --whole-word --case-insensitive
```

The previously documented import ordering now works:

```bash
printf 'public class Imported {}' |
  refactorkit java import-class --target-package com.example --stdin /workspace
```

Scripts that incorrectly supplied a value to a boolean option must be corrected;
that value is now interpreted as a positional argument as documented.

## Safety changes

- Managed apply requires explicit authorization and a diagnostics gate.
- Only `JDT_BINDING` and documented `STRUCTURAL` plans can apply; lexical
  fallbacks refuse before WAL.
- Snapshot evidence includes module relationships, source levels, classpaths,
  build descriptors, generated declarations, and Eclipse formatter preferences.
- Staged diagnostics resolve exact post-image overlays and block unapproved new
  errors before WAL.
- Transaction journals use the hardened versioned recovery contract documented
  in `docs/transactionality-audit.md`.

Old transaction records remain subject to the documented schema migration and
legacy-image limitations. Preserve `.refactorkit/transactions` when upgrading or
downgrading if rollback history is required.

## Protocol and resource behavior

- Machine-readable API inventory: `docs/api-0.2-contract.json`.
- NDJSON and LSP requests larger than 1 MiB are rejected.
- Sessions retain at most 128 pending plans; evicted IDs must be previewed again.
- The synchronous API `0.2` servers do not advertise in-flight cancellation.

## Packaging

The Linux x86_64 runtime remains self-contained and uses its embedded Java image.
Release assets add an SPDX JSON SBOM and GitHub build/SBOM attestations. Verify
the checksum and attestations before deployment.

## Rollback and downgrade

Workspace changes made by `v0.4.0` should be rolled back with the same runtime
before downgrading when practical:

```bash
refactorkit patch rollback <transaction-id> --root <workspace>
```

Downgrading the executable does not rewrite user projects, but older versions may
not understand newer journal metadata or formatter plans. Back up transaction
metadata and use the verified `v0.3.0` artifact if a downgrade is required.
