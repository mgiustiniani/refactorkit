# RefactorKit v0.6.0 TypeScript/JavaScript Release Plan

Status: implementation complete; release-candidate acceptance pending.

Baseline: published `v0.5.0`; release target `v0.6.0`; API `0.2` remains active.
The release scope is the managed TypeScript/JavaScript semantic rename foundation.
Advanced T5 refactorings and real file-name-changing operations are explicitly
scheduled for `v0.7.x` rather than silently broadening this release.

## Objective

Deliver the first deep non-Java semantic adapter for TypeScript and JavaScript on
top of the `v0.5.0` mixed-language kernel. TypeScript and JavaScript do not become
stable merely because packaged Tree-sitter outline/search exists. Stable mutation
requires solution-aware compiler evidence, exact normalized proposals, staged
diagnostics, approval, managed apply, WAL, rollback, bounded resources and native
packaged/toolchain-qualified acceptance.

Java depth continues concurrently through advanced signature, import and move
work. This release does not begin the later Objective-C or Swift implementation
bands; those are now mandatory pre-`1.0` targets in ADR 0009 and the supreme
roadmap.

## T1 — Toolchain and supply-chain boundary

- [x] Define explicit experimental provider `typescript-lsp-explicit-v1` and
  strict versioned discovery/provenance models.
- [x] Discover configured Node.js, TypeScript and language-server entry points
  without executing project package scripts, lifecycle hooks or arbitrary build
  code; the only probe executes the selected Node binary with constant
  `--version` through the bounded process manager.
- [x] Refuse project-local executable/package roots unless the caller explicitly
  enables workspace-toolchain trust; PATH discovery is also opt-in.
- [x] SHA-256 bind Node, package manifests, language-server entrypoint and
  `tsserver.js` evidence; project configuration graph binding remains T2 work.
- [x] Publish the supported toolchain matrix, package license/SBOM and network
  policy, and self-contained versus toolchain-qualified boundary. Managed support
  is qualified for Node 22.18.0, `typescript-language-server` 5.1.3 and TypeScript
  5.9.3; broader discovery ranges remain experimental, not supported claims.
- [x] Add hostile package/workspace, timeout, crash and process-tree acceptance,
  including symlink/shape/size refusals, bounded restart, forced real-server kill,
  and packaged kill at durable transaction state `APPLYING`.

## T2 — TypeScript/JavaScript project model

- [x] Add `typescript-config-declarative-v1` and project effective
  `tsconfig.json`/`jsconfig.json` roots, local extends, references,
  include/exclude/files, root/out/base directories, module/JSX modes, path
  aliases, package type and declaration/composite flags into the language-neutral
  Build Model SPI. Package-based extends remains an explicit refusal until its
  trust/resolution policy is implemented.
- [x] Parse bounded JSON/JSONC declaratively without executing package managers,
  bundlers, task runners, plugins or framework configuration.
- [x] Model monorepo/project-reference compile edges, custom referenced config
  files, and TypeScript/JavaScript mixed settings with cycle/missing-target
  refusal.
- [x] SHA-256 bind config/extends/package inputs and the deterministic effective
  projection into `ProjectSnapshot` hashing.
- [x] Emit concise typed `typescript.*` root causes for syntax, shape, limit,
  path, symlink, extends, reference and package failures.

## T3 — Semantic read operations

- [x] Complete cross-session stable semantic symbol keys. Because LSP exposes no
  portable compiler handle, RefactorKit derives opaque `lsp-symbol-v1` SHA-256 IDs
  from language, normalized declaration path, semantic parent/container hierarchy,
  kind, name and bounded signature detail; source ranges are excluded. Bounded exact
  document symbols and 200-result workspace-symbol search therefore retain identity
  across sessions and line movement under the same toolchain provenance.
- [x] Synchronize bounded full overlay documents with content hashes and strictly
  increasing versions (`didOpen`/`didChange`/`didClose`), resynchronizing the
  snapshot image before rename.
- [x] Validate LSP UTF-16 positions against exact snapshot text and refuse
  surrogate-pair splits; normalized returned edits still pass the core range
  simulator.
- [x] Report full TypeScript, `checkJs=true`, dynamic JavaScript and mixed-project
  completeness. Dynamic or mixed JavaScript remains proposal-only and its managed
  diagnostics gate refuses with `typescript.semanticCompletenessInsufficient`.
- [x] Complete the packaged crash and bounded-process policy. Request/frame/output
  limits, 500 diagnostics, 256 open documents, 4 MiB documents, a 512 MiB Node/V8
  old-space ceiling, provenance-preserving restart capped at three attempts per
  rolling 60 seconds, real process kill, and kill-during-write recovery are native
  acceptance gates. OS-level RSS sandboxing is explicitly not claimed in `v0.6.0`.

## T4 — Managed rename catalogue

- [x] Cover identifier-based semantic rename catalogue. Exact resolved
  class/interface/enum/function/method/property/field/variable/constant/type-alias/
  type-parameter/parameter/namespace symbols now combine language-server identity
  with native-AST ancestor classification and require an exact bounded prepare-rename range
  and accept safe non-reserved Unicode identifiers. Native AST distinguishes
  `module Identifier` from `namespace Identifier`; ambient external string-module
  names remain intentionally non-identifier/refused. Unresolved, constructor,
  package, unclassified and no-op targets refuse.
- [x] Qualify the release-scope real-toolchain identifier rename. Packaged native
  acceptance covers path aliases and a three-file class/re-export rename with exact
  compiler diagnostics, apply, WAL and rollback. JVM tests retain server-required
  file-rename normalization and a bounded 128-entry staged hash allowlist. A
  user-facing real file-name operation is deferred to T5 in `v0.7.x`.
- [x] Distinguish proven semantic references from bounded dynamic candidates.
  Dynamic/mixed JavaScript cannot obtain a managed gate; up to 50 exact quoted
  symbol-name candidates across computed names, decorators, reflection and
  framework strings refuse by default and require an explicit high-risk override.
- [x] Complete native packaged TypeScript managed acceptance. The
  `typescript-compiler-exact-v1` provider now produces request-correlated no-emit
  diagnostics from the hash-bound compiler API and refuses proposals with new
  errors. Packaged Linux/Windows/macOS qualification covers lock-time evidence,
  forced semantic-process termination, provenance-preserving restart, explicit
  authorization, `PatchEngine` apply, WAL, kill in durable `APPLYING`, startup
  recovery and rollback. OS RSS acceptance and broader toolchain versions remain open.
- [x] Refuse bounded-workspace incompleteness: exported symbols in package
  `exports`/`types`/`typings`, declaration, composite or project-reference library
  surfaces refuse by default with `typescript.externalConsumersUnknown`; an
  explicit override raises risk and records a warning. Generated targets,
  ambiguous ownership and stale versions remain fail-closed.

## T5 — Deferred advanced refactorings (`v0.7.x`)

The following are explicit post-`v0.6.0` scope, not release blockers:

- [ ] Move file/type/module with import/export and project-reference updates.
- [ ] Organize imports and deterministic project-style-aware formatting.
- [ ] Change signature with overload/override/call-site coverage.
- [ ] Add bounded extract/inline operations where control/data-flow proof exists.
- [ ] Add TypeScript/JavaScript migration recipes without running framework build
  code.

## T6 — Integrations and acceptance

- [x] Complete LSP language routing while preserving client-managed native
  TypeScript/JavaScript LSP ownership. RefactorKit LSP accepts mixed script documents,
  provides bounded in-process structural outlines for `.ts`/`.js`, suppresses false
  Java code actions/formatting/tokens/diagnostics, and publishes the ownership split
  in `experimental.refactorkitSemanticOwnership`. Native TypeScript/JavaScript
  semantics remain client-managed; rollbackable compiler-backed operations route
  through CLI, daemon or MCP, which preserve mixed source images and expose
  explicit-toolchain search/definition/references/diagnostics, rename preview,
  exact-gated apply, WAL and rollback. MCP EOF closes all owned process trees.
- [x] Expose layered capabilities through library, CLI, daemon, LSP and MCP.
  Each operation now identifies its own backend and runtime, distinguishing
  in-process Tree-sitter outline/search from external compiler/LSP semantics.
- [x] Add single-project, project-reference/monorepo, checked/dynamic JavaScript,
  and JavaScript/TypeScript mixed samples with project-model acceptance.
- [x] Add unit, golden, protocol, agent-simulation, stress, crash/restart and
  packaged/toolchain-qualified native acceptance on all supported platforms.
  Pinned Node 22.18.0 + `typescript-language-server` 5.1.3 + TypeScript 5.9.3
  packaged qualification now covers real search/definition/references and stable
  IDs on Linux, Windows and macOS CI. Exact compiler diagnostics, path-alias/
  re-export apply, WAL and rollback, real semantic process kill/restart, and
  packaged kill-during-write startup recovery are included.
- [x] Document support matrix, limitations, migration and rollback.

## Later mandatory native-language bands

The approved pre-`1.0` sequence now includes:

- `0.11.x`: C, C++, Objective-C and Objective-C++ through shared Clang
  infrastructure with separate capability matrices;
- `0.12.x`: Swift through compiler/SourceKit-LSP/SwiftSyntax plus SwiftPM and
  bounded Xcode models, including Objective-C interoperability;
- `0.13.x`: Groovy;
- `0.14.x`: C#;
- `0.15.x`: Rust;
- final `1.0` development: Clojure and global stabilization.

Removing Objective-C or Swift from the `1.0` target requires an explicit
superseding product decision and ADR update.

## Release gate

`v0.6.0` requires at least one non-trivial TypeScript semantic mutation to pass
preview, exact normalized edits, staged diagnostics, authorization, managed
apply, WAL, recovery and rollback on every supported native platform. Structural
outline or an LSP rename response alone is insufficient.
