# RefactorKit v0.6.0 TypeScript/JavaScript Release Plan

Status: active.

Baseline: published `v0.5.0`; main `0.6.0-SNAPSHOT`; API `0.2` until an explicit
migration is accepted.

## Objective

Deliver the first deep non-Java semantic adapter for TypeScript and JavaScript on
top of the `v0.5.0` mixed-language kernel. TypeScript and JavaScript do not become
stable merely because packaged Tree-sitter outline/search exists. Stable mutation
requires solution-aware compiler evidence, exact normalized proposals, staged
diagnostics, approval, managed apply, WAL, rollback, bounded resources and native
packaged/toolchain-qualified acceptance.

Java depth continues concurrently through advanced signature, import and move
work. This release does not begin the later Objective-C or Swift implementation
bands; those are now mandatory pre-`1.0` targets in ADR 0009 and the supreme
roadmap.

## T1 — Toolchain and supply-chain boundary

- [x] Define explicit experimental provider `typescript-lsp-explicit-v1` and
  strict versioned discovery/provenance models.
- [x] Discover configured Node.js, TypeScript and language-server entry points
  without executing project package scripts, lifecycle hooks or arbitrary build
  code; the only probe executes the selected Node binary with constant
  `--version` through the bounded process manager.
- [x] Refuse project-local executable/package roots unless the caller explicitly
  enables workspace-toolchain trust; PATH discovery is also opt-in.
- [x] SHA-256 bind Node, package manifests, language-server entrypoint and
  `tsserver.js` evidence; project configuration graph binding remains T2 work.
- [ ] Complete supported Node/TypeScript/language-server matrix, package
  license/SBOM, network policy and self-contained versus toolchain-qualified
  distribution acceptance. The initial bounded policy accepts Node 18–24 and
  TypeScript 5.x only.
- [ ] Add hostile package/workspace, timeout, crash and process-tree acceptance.

## T2 — TypeScript/JavaScript project model

- [x] Add `typescript-config-declarative-v1` and project effective
  `tsconfig.json`/`jsconfig.json` roots, local extends, references,
  include/exclude/files, root/out/base directories, module/JSX modes, path
  aliases, package type and declaration/composite flags into the language-neutral
  Build Model SPI. Package-based extends remains an explicit refusal until its
  trust/resolution policy is implemented.
- [x] Parse bounded JSON/JSONC declaratively without executing package managers,
  bundlers, task runners, plugins or framework configuration.
- [x] Model monorepo/project-reference compile edges, custom referenced config
  files, and TypeScript/JavaScript mixed settings with cycle/missing-target
  refusal.
- [x] SHA-256 bind config/extends/package inputs and the deterministic effective
  projection into `ProjectSnapshot` hashing.
- [x] Emit concise typed `typescript.*` root causes for syntax, shape, limit,
  path, symlink, extends, reference and package failures.

## T3 — Semantic read operations

- [ ] Complete workspace-wide compiler identity/search. Bounded exact LSP
  document symbols (nested `DocumentSymbol` and `SymbolInformation`), definition,
  references and ranged diagnostics now replace structural symbols whenever the
  semantic client is active; cross-project stable symbol keys remain open.
- [x] Synchronize bounded full overlay documents with content hashes and strictly
  increasing versions (`didOpen`/`didChange`/`didClose`), resynchronizing the
  snapshot image before rename.
- [x] Validate LSP UTF-16 positions against exact snapshot text and refuse
  surrogate-pair splits; normalized returned edits still pass the core range
  simulator.
- [ ] Report full typed versus `checkJs`/dynamic JavaScript completeness; current
  descriptors and rename risk distinguish TypeScript from JavaScript but do not
  yet prove compiler completeness.
- [ ] Complete restart and memory policy. Request/frame/output/process limits,
  500 diagnostics, 256 open documents and 4 MiB document limits are implemented.

## T4 — Managed rename catalogue

- [ ] Rename TypeScript declarations, functions, classes, interfaces, types,
  enums, namespaces, modules, properties and parameters where compiler proof is
  complete.
- [ ] Update imports, exports, re-exports, path aliases and file names when
  semantically required.
- [ ] Distinguish JavaScript dynamic properties, computed names, decorators,
  reflection and framework strings from proven references.
- [ ] Complete TypeScript staged diagnostics and managed apply/rollback
  acceptance. Server edits now normalize into approval-required experimental
  `PatchPlan` previews with `LANGUAGE_SERVER` evidence and no direct writes.
- [ ] Refuse incomplete external consumers, generated files, ambiguous project
  ownership and stale document versions.

## T5 — Move, imports, signature and extraction

- [ ] Move file/type/module with import/export and project-reference updates.
- [ ] Organize imports and deterministic project-style-aware formatting.
- [ ] Change signature with overload/override/call-site coverage.
- [ ] Add bounded extract/inline operations where control/data-flow proof exists.
- [ ] Add TypeScript/JavaScript migration recipes without running framework build
  code.

## T6 — Integrations and acceptance

- [ ] Route mixed Java/TypeScript/JavaScript snapshots through the shared registry.
- [ ] Expose exact capabilities through library, CLI, daemon, LSP and MCP.
- [ ] Add single-project, project-reference, monorepo, typed/dynamic JS and
  JavaScript/TypeScript mixed samples.
- [ ] Add unit, golden, protocol, agent-simulation, stress, crash/restart and
  packaged/toolchain-qualified native acceptance on all supported platforms.
- [ ] Document support matrix, limitations, migration and rollback.

## Later mandatory native-language bands

The approved pre-`1.0` sequence now includes:

- `0.11.x`: C, C++, Objective-C and Objective-C++ through shared Clang
  infrastructure with separate capability matrices;
- `0.12.x`: Swift through compiler/SourceKit-LSP/SwiftSyntax plus SwiftPM and
  bounded Xcode models, including Objective-C interoperability;
- `0.13.x`: Groovy;
- `0.14.x`: C#;
- `0.15.x`: Rust;
- final `1.0` development: Clojure and global stabilization.

Removing Objective-C or Swift from the `1.0` target requires an explicit
superseding product decision and ADR update.

## Release gate

`v0.6.0` requires at least one non-trivial TypeScript semantic mutation to pass
preview, exact normalized edits, staged diagnostics, authorization, managed
apply, WAL, recovery and rollback on every supported native platform. Structural
outline or an LSP rename response alone is insufficient.
