# RefactorKit v0.2.0-beta Release Plan

Status: draft plan for the second RefactorKit release with P1/P2, initial P3,
initial P4 daemon/MCP progress, hardened P4 LSP command-contract progress, P5
documentation and preview-warning review, P6 importer
output-contract/refusal-guidance, P7 packaging/release-verification, and P8 draft
release-documentation plus installation/smoke-test instruction progress
recorded. This document is planning prose only; it does not imply that a release
tag or release artifact already exists.

Progress snapshot, 2026-07-10:

- P1/P2: 22 golden cases document shipped patch-producing operations, including
  framework-warning assertions, and sample coverage progress.
- P3: PatchEngine safety tests now cover stale snapshots, outside-workspace
  paths, overlapping edits, and rollback restoration for modify, create, rename,
  and delete edits.
- P4: Initial daemon JSON-RPC and MCP contract tests cover selected
  preview/apply/rollback and refusal/error contracts. LSP command tests now
  cover preview metadata, pending-plan apply, transaction-backed rollback,
  unknown transaction refusal, and `safeDelete` `PLAN_REFUSED` behavior. Daemon
  and MCP importer output tests also cover provenance/license fields.
- P5: Operation docs now cover rename class/member, move class, organize imports,
  safe delete, extract method, change signature, and external import success
  conditions, refusals, warnings, and rollback expectations. Runtime preview
  warning wording has been reviewed against those docs for the major shipped and
  experimental operations. Safe-delete docs explicitly call out build files,
  generated code, reflection, framework config, tests, downstream consumers, and
  forced-mode risk.
- P6: External importer hardening tests now cover provenance warning details,
  GPL high-risk handling, helper-type preservation, multi-public-type splitting
  with package/import preservation, non-Java Markdown fence stripping,
  CLI/JSON-RPC/MCP output exposure for provenance/license fields, and refusal
  guidance for unknown-license blocks and naming conflicts.
- P7: CI now builds and uploads `refactorkit-runtime.zip` plus
  `refactorkit-runtime.zip.sha256`, verifies the checksum, smoke-tests the
  packaged launcher with `JAVA_HOME` unset across Maven, Gradle, Spring, JPA,
  and multi-module samples, and release tag builds verify and unzip-smoke the
  tag-named runtime asset before publishing.
- P8: `CHANGELOG.md` now has an Unreleased / `v0.2.0-beta - in progress`
  section, `.github/releases/v0.2.0-beta.md` now exists as a draft tag-specific
  release-note file, and README/packaging/release-note instructions now describe
  checksum verification, unzip layout, `JAVA_HOME`-unset smoke tests, PATH setup,
  and representative sample scans for the beta runtime asset.
- Remaining beta work stays tracked below, especially broader daemon/LSP/MCP
  protocol coverage beyond the hardened command paths, final release-note
  limitation links and license/provenance warnings, final artifact metadata, and
  acceptance-review coverage.

## Release objective

`v0.2.0-beta` moves RefactorKit from the first public alpha preview to a beta
that integration consumers can test against repeatedly. The release should prove
that the core safety workflow works across realistic Java project shapes:
preview, inspect diagnostics, apply, diagnose again, and rollback.

The beta builds on completed `v0.1.0-alpha` evidence:

- 15 golden cases passing in the alpha test suite;
- ADR coverage for the major architecture decisions;
- runtime release asset publication with a checksum;
- rollback-focused agent simulation tests;
- documented alpha limitations for lexical Java analysis and conservative MVP
  refactorings.

## Stability promise

The beta is not a `v1.0.0` compatibility freeze, but it must be stable enough for
scripts, editors, daemons, and local AI agents to test safely.

For `v0.2.0-beta`:

- CLI command names, option names, exit-code categories, and structured output
  fields used by documented workflows are beta compatibility contracts.
- Daemon JSON-RPC method names, request shapes, response envelopes, and error
  categories used by documented workflows are beta compatibility contracts.
- LSP and MCP surfaces remain beta/experimental, but their documented commands,
  tools, resources, and refusal behavior must have contract tests.
- Breaking changes after this beta require changelog entries and migration notes.
- Patch safety semantics remain mandatory: preview before apply, snapshot
  validation, affected-file reporting, diagnostics, transaction logging, and
  rollback metadata.
- Java analysis may remain lexical/heuristic only where explicitly documented in
  previews, warnings, release notes, and the lexical-vs-compiler decision record.

## Exit criteria

The release candidate is acceptable only when all of the following are true:

1. CI is green on the intended release commit.
2. `./gradlew build` passes locally or in CI for the release candidate.
3. `./refactorkit test-golden` passes with representative success and refusal
   coverage for every shipped patch-producing operation.
4. Golden coverage includes, at minimum:
   - `renameClass` success and refusal/risk cases;
   - `renameMember` success and refusal/risk cases;
   - `moveClass` success and refusal/risk cases;
   - `organizeImports` success, no-op, and documented limitation cases;
   - `safeDelete` success and refused-because-referenced cases;
   - limited `extractMethod` success and conservative refusal cases;
   - limited `changeSignature` rename/add/reorder/remove success cases and at
     least one refusal case;
   - external Java class import preview, conflict, license-policy, and multiple
     public type cases;
   - recipe or orchestration coverage for any recipe operation advertised as
     shipped in the beta.
5. Sample projects exercise Maven, Gradle, Spring, JPA, and multi-module scans.
6. Integration tests apply, diagnose, rollback, and verify hash restoration for
   representative mutating operations.
7. Daemon JSON-RPC, LSP, and MCP contract tests cover preview, apply, rollback,
   diagnostics, invalid input, stale snapshot, and refused operation responses.
8. The `safe-delete` documentation explains build-file, generated-code,
   reflection, framework-configuration, and external-configuration limits.
9. The external importer test suite covers provenance, license warnings, naming
   conflicts, package rewriting, multiple public types, and refusal paths.
10. CI publishes the self-contained runtime artifact and checksum, verifies the
    checksum, and release verification checks and smoke-tests the extracted
    artifact before publication.
11. The project records a decision on what remains lexical MVP in beta and what
    must become compiler-backed before `v1.0.0`.
12. CHANGELOG, release notes, installation/smoke-test instructions, known
    limitations, and migration notes are updated before tagging.

## Ordered workstreams and epics

### P0 — Release scope and compatibility baseline

Goal: prevent beta scope drift and define exactly what external consumers may
rely on.

Tasks:

- Freeze the list of operations advertised for `v0.2.0-beta`.
- Mark each CLI command as beta contract, experimental, or internal.
- Mark each daemon JSON-RPC method as beta contract, experimental, or internal.
- Review LSP commands and MCP tools/resources/prompts against the documented
  integration contract.
- Define structured error categories for refused plans, invalid input, stale
  snapshot, diagnostics failure, unsupported operation, and unsafe workspace
  paths.
- Add migration-note requirements for any CLI/JSON-RPC changes made after
  `v0.1.0-alpha`.
- Create or update the lexical-vs-compiler decision record: identify which beta
  behavior remains lexical MVP and which capabilities require compiler-backed
  analysis before `v1.0.0`.

Deliverables:

- Updated integration contract documentation.
- Compatibility table for CLI and JSON-RPC surfaces.
- Lexical-vs-compiler-backed decision note linked from release notes.

### P1 — Golden coverage for all shipped operations

Goal: make shipped behavior reviewable, repeatable, and explicit.

Tasks:

- Expand golden tests beyond the 15 alpha cases to cover representative success
  and refusal paths for every shipped patch-producing operation.
- Keep every golden case small, readable, and tied to one expected behavior.
- Include cross-file and cross-package references where the operation claims to
  update them.
- Include refusal fixtures for ambiguity, unresolved symbols, conflicts, stale or
  unsafe inputs, and unsupported lexical cases.
- Require golden expected plans to check operation status, affected-file counts,
  warnings, and summaries where practical.
- Document any operation that cannot yet support a meaningful success or refusal
  fixture and decide whether it remains shipped or is marked experimental.

Deliverables:

- Expanded `testdata/golden/` suite.
- Updated testing documentation listing current beta golden cases.
- CI failure on golden drift.

### P2 — Representative sample projects

Goal: verify that scans, symbols, diagnostics, and previews work on project
shapes beta users are likely to try.

Tasks:

- Keep `samples/java-maven-simple` as the minimal Maven smoke sample.
- Keep or add `samples/java-gradle-simple` for Gradle discovery and scan smoke
  coverage.
- Add or expand a Spring sample with annotations that should trigger preview
  warnings for string/framework references.
- Add or expand a JPA sample with entity/table/column annotations that should
  trigger preview warnings for persistence-contract risk.
- Add or expand a multi-module sample with cross-module references and clear
  module boundaries.
- Add scans, symbol listing, diagnostics, and at least one safe preview workflow
  per sample category.
- Avoid turning sample projects into large fixtures; keep each one focused on the
  release risk it demonstrates.

Deliverables:

- Maven, Gradle, Spring, JPA, and multi-module sample coverage.
- CI smoke commands for the representative samples.
- Documented sample matrix in testing or release notes.

### P3 — Apply, diagnostics, and rollback integration safety

Goal: prove that RefactorKit-managed writes are recoverable in realistic flows.

Tasks:

- Add integration tests that create a preview, inspect diagnostics, apply the
  patch, run diagnostics again, rollback the transaction, and verify snapshot or
  content restoration.
- Cover file modification, file creation, file rename/move, and refused deletion
  paths.
- Include stale snapshot rejection before apply.
- Include overlapping edit rejection and unsafe path rejection at the patch
  engine boundary.
- Verify `.refactorkit/` transaction metadata is created only inside the
  workspace root and ignored by golden comparisons where appropriate.
- Preserve rollback simulations from alpha and extend them to the beta operation
  set.

Progress:

- PatchEngine tests cover stale snapshot rejection, outside-workspace path
  refusal, overlapping-edit rejection, and rollback restoration for modify,
  create, rename, and delete edits.

Remaining deliverables:

- Representative apply/diagnose/rollback workflows across beta operations.
- Clear rollback verification output in CI logs.
- Release notes explaining rollback expectations and limits.

### P4 — Daemon, LSP, and MCP contract tests

Goal: make integration surfaces safe for editor, daemon, and local-agent pilots.

Tasks:

- Add JSON-RPC daemon tests for `project.open`, `project.summary`,
  `symbol.search`, `symbol.references`, `diagnostics`, `refactor.preview`,
  `refactor.apply`, and `patch.rollback`.
- Cover success and refusal responses for invalid params, unsupported operations,
  stale plan IDs, stale snapshots, and unsafe paths.
- Add LSP protocol tests for definition, references, prepareRename, rename,
  codeAction, documentSymbol, diagnostics, executeCommand preview/apply, and
  rollback command behavior.
- Add MCP tests for scoped resources, refusal of files outside the scanned
  workspace, preview/apply/rollback tools, external import tool, and context
  bundle generation.
- Verify that daemon/LSP/MCP errors do not leak internal exceptions as plain
  strings.

Progress:

- Daemon JSON-RPC tests cover `refactor.preview` → `refactor.apply` →
  `patch.rollback`, missing params, unknown plan, and stale plan/snapshot error
  code contracts.
- MCP tests cover `renameClass` preview/apply/rollback, invalid tool errors
  without stack traces, and outside-workspace/not-in-snapshot resource refusals.
- LSP tests cover `executeCommand` preview metadata (`refactorkitPlanId`,
  operation, status, summary, risk level, and warnings), preview → apply →
  rollback for `renameClass`, pending-plan storage for apply, transaction
  metadata rollback, unknown plan and transaction refusals, unknown commands,
  and referenced `safeDelete` `PLAN_REFUSED` behavior.
- Daemon JSON-RPC and MCP tests cover external importer preview output for the
  stable provenance/license fields.

Remaining deliverables:

- Broader daemon JSON-RPC, LSP, and MCP contract-test coverage for the full beta
  surface listed above, especially read-only LSP features and protocol edges not
  exercised by the command preview/apply/rollback flow.
- Keep integration-contract documentation aligned if command, method, tool, or
  resource contracts change before tagging.
- Migration notes for any changed method, command, tool, or resource.

### P5 — Operation documentation and risk warnings

Tasks:

- Expand `safe-delete` documentation with explicit limits for build files,

  tests, and unknown downstream consumers.
- Ensure rename, move, extract method, change signature, organize imports, and

  and rollback expectations.
- Check preview warning wording for lexical/heuristic operations.
- Document how agents should handle refused plans without falling back to global
  text replacement.
- Link relevant ADRs from release notes and operation docs.

Progress:

- Added operation documentation for safe delete, rename class/member, move class,
  and organize imports.
- Updated extract method, change signature, external importer, refactoring-engine,
  and safety-model docs with success/refusal/warning/rollback guidance.
- Safe-delete documentation now has a beta limitations section for build files,
  generated code, reflection, framework configuration, tests, unknown downstream

- Runtime planner warning wording has been reviewed against current operation

  limited extract method, limited change signature, and external Java import.
  Reviewed categories include lexical/string/framework limits, large or forced

Remaining deliverables:

- Link final operation limitations from tag-specific release notes and changelog.
- Link relevant ADRs from release notes and any operation doc section that needs

### P6 — External Java class importer hardening

Goal: keep external code assimilation safe and auditable.

Tasks:

- Add tests for markdown fence stripping, package detection, package rewriting,
  target path calculation, and organizing imports after import planning.
- Add tests for multiple public top-level types, multiple non-public top-level
  types, naming conflicts, unknown license, blocking license policy, and
  provenance metadata.
- Require preview-only behavior for unknown or risky license cases unless policy
  explicitly allows apply.
- Document license-risk and provenance fields in CLI/JSON-RPC/MCP outputs.
- Verify importer refusal messages recommend user review instead of overwrite.

Progress:

- Focused importer tests cover provenance warning details, including source kind,
  source URL, license, and source hash evidence.
- GPL-family imports now have high-risk warning/risk-level coverage.
- Type layout tests cover one public type plus package-private helper kept in one

- Markdown cleanup tests cover non-Java fenced snippets with surrounding prose.
- CLI/JSON-RPC/MCP output-contract tests and documentation now cover the stable

  `licenseDetected`, `licenseRisk`, and `originalHash`.
- Refusal-message tests now cover unknown-license blocks and naming conflicts

  license policy, or naming conflicts, and RefactorKit never overwrites existing

Remaining deliverables:

## Non-goals and deferred work

The beta must not expand into unrelated product work. The following are deferred
unless explicitly re-scoped before the release candidate:

- full Java compiler-backed semantic analysis for all operations;
- perfect unused-import removal if type analysis is still unavailable;
- perfect Spring/JPA/Jackson framework-aware rewrites;
- IDE plugins, custom UI, Tauri integration, or cloud service work;
- broad multi-language semantic refactoring beyond documented MVP surfaces;
- `v1.0.0` API freeze and long-term deprecation policy completion;
- destructive forced safe-delete behavior without explicit user override and
  documented risk;
- importing external code without provenance and license-risk handling.

## Risks and mitigations

| Risk | Impact | Mitigation | Beta status |
|------|--------|------------|-------------|
| Lexical Java analysis misses semantic references | Unsafe or incomplete previews | Explicit lexical-vs-compiler decision, preview warnings, refusal cases, and manual review guidance | Must be visible |
| Framework and reflection references are hidden in strings/config | Rename/move/safe-delete plans may miss external contracts | Spring/JPA samples, warning coverage, safe-delete docs, known-limits release notes | Must be visible |
| CLI/JSON-RPC contracts stabilize too early | Consumers may depend on weak shapes | Scope freeze, contract tests, migration notes for changes | Controlled risk |
| LSP/MCP behavior is less mature than CLI/daemon | Editor and agent pilots may see protocol gaps | Mark beta/experimental, add refusal and scoped-resource tests | Controlled risk |
| External importer license handling is incomplete | Legal/provenance risk | Preview-only default for unknown license, provenance metadata, blocking policy tests | Must be visible |
| Packaging artifact differs from local build | Users cannot reproduce smoke tests | CI artifact verification, checksum publication, packaged CLI smoke tests | Must pass |
| Test suite grows without readability | Golden failures become hard to review | Small focused fixtures and documented matrix | Controlled risk |

## Beta acceptance criteria

The beta can be accepted when a reviewer can complete this workflow from a clean
release candidate without undocumented assumptions:

1. Download or build the runtime artifact and verify its checksum.
2. Run the packaged CLI without relying on a globally installed Java runtime.
3. Scan Maven, Gradle, Spring, JPA, and multi-module sample projects.
4. Generate at least one safe preview, inspect warnings and diagnostics, apply
   the plan, rollback the transaction, and verify restoration.
5. Run the expanded golden suite and see all cases pass.
6. Exercise daemon JSON-RPC preview/apply/rollback and observe structured success
   and refusal responses.
7. Exercise representative LSP and MCP contract tests or documented smoke tests.
8. Confirm that safe-delete and external-import limitations are documented and
   visible in release notes.
9. Confirm that lexical MVP limitations are documented and the compiler-backed
   follow-up decision is recorded.
10. Confirm that CHANGELOG, release notes, migration notes, artifact names,
    checksums, and rollback guidance are complete before tagging.
