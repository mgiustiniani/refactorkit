## Capability boundary

This slice provides structural, exact-package Java compilation-unit migration.
It moves non-generated production/test units and `package-info.java`, updates
imports and fully qualified moved-type references, exact package-name literals
used by Java architecture contracts, and exact Java source-path literals naming
moved units. Subpackage declarations remain unchanged.

It does not claim compiler-authoritative package migration, recursive subpackage
migration, Maven module/artifact migration, or arbitrary changes to POMs,
Dockerfiles, CI, scripts, documentation, features, or configuration strings.

## External baseline limitation

`mvn -B --no-transfer-progress clean test` was executed. The migrated
`cluster-control-ratis-infrastructure` module passed, including the source-path
contract that exposed the first migration defect. The full reactor later failed
in `s3-reactive-api-adapter` because
`Phase3PipelineUnitSpecsCucumberTest.Example #1.1` expected `chunking` evidence
and observed only `chunk-persistence`, with an associated missing chunk file.
The same focused test fails from an untouched disposable clone at the exact
external baseline, so this is recorded as an external baseline failure rather
than package-migration regression. RefactorKit did not modify Magrathea to mask
that failure.

Fresh-checkout JDT diagnostics also retain a large unchanged baseline. Maven
reactor dependencies are modeled and focused cross-module fixtures are clean,
but JDT sourcepath lookup cannot resolve every Magrathea declaration whose
physical package path intentionally differs from its declared package until a
future bounded batch-compilation source environment is implemented. The normal
recipe policy therefore compares diagnostic deltas; strict mode remains
available and would refuse this external baseline.
