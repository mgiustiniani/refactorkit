# RefactorKit v0.3.0 Release Plan

Status: draft for the post-`v0.2.0-beta` development slice.

Target implementation version on main: `0.3.0-SNAPSHOT`.
Current beta contract API baseline: `0.2`.

## Release objective

`v0.3.0` should make RefactorKit's integration metadata and compatibility rules
clearer before the project moves toward `v1.0.0`. The release is not intended to
weaken or replace the existing safety workflow: every mutating operation remains
previewable, diagnosable, traceable, and rollbackable.

## Goals

1. Expose consistent version/API metadata across CLI, daemon JSON-RPC, LSP, and
   MCP surfaces.
2. Define the semantic versioning and deprecation policy for beta-contract
   integration surfaces.
3. Decide and prototype the compiler-backed Java analysis direction needed before
   `v1.0.0`.
4. Broaden protocol-contract documentation for version metadata, capability
   discovery, and read-only integration checks.
5. Keep the `v0.2.0-beta` safety model intact: preview before apply, snapshot
   validation, affected-file reporting, diagnostics, transaction logs, and
   rollback metadata.

## Workstreams

### P0 — Version and API metadata

- Keep centralized metadata for implementation name, implementation version, and
  API version.
- Keep `apiVersion=0.2` as the current beta contract baseline while main uses
  implementation version `0.3.0-SNAPSHOT`.
- Document and test CLI `--version`/`version`, daemon `server.version`, and
  LSP/MCP `serverInfo.version`.

### P1 — Semantic versioning and deprecation policy

Status: draft policy published in
[docs/compatibility-policy.md](../compatibility-policy.md). The integration
contract page links to the policy and keeps the detailed per-surface baseline.

- Define how implementation versions, API versions, prerelease tags, and
  snapshots relate to compatibility promises. Drafted.
- Document when a beta-contract breaking change requires a migration note,
  deprecation warning, or API-version change. Drafted.
- Define deprecation expectations for CLI commands/options, daemon methods and
  fields/errors, LSP commands/metadata, and MCP tools/resources/prompts.
  Drafted.

### P2 — Compiler-backed Java analysis planning

Status: decision docs started. ADR 0008 records Eclipse JDT as the primary
compiler-backed Java analysis prototype candidate for parsing, bindings, symbol
identity, and reference precision. Lexical planners remain a documented safety
fallback, and OpenRewrite remains a possible future recipe/transformation
backend rather than the first symbol identity source.

Remaining prototype tasks:

- Validate source-root and classpath discovery on representative Maven and Gradle
  samples.
- Prove JDT-backed symbol identity for classes and selected members.
- Prove reference search can distinguish same-name symbols in different packages
  or scopes.
- Label preview evidence as JDT-backed or lexical fallback.
- Keep warnings/refusals for unsupported overload, inheritance, framework,
  generated-code, and classpath cases.
- Avoid claiming full `v1.0.0` semantic coverage until implementation and tests
  prove it.

### P3 — Broader protocol contracts

- Expand integration-contract documentation for read-only metadata and capability
  discovery surfaces.
- Keep daemon, LSP, and MCP contract tests aligned with documented beta-contract
  behavior.
- Preserve scoped access rules: no arbitrary filesystem access through MCP and no
  workspace modification outside preview/apply flows.

## Non-goals

- No `v1.0.0` API freeze in this slice.
- No weakening of preview/apply/rollback safety requirements.
- No IDE-specific or UI-specific coupling.
- No claim that Java analysis is fully compiler-backed until the implementation,
  tests, and docs prove it.

## Exit criteria

- Version/API metadata is documented and tested on all advertised integration
  surfaces.
- The SemVer/deprecation policy is published in project docs.
- The compiler-backed Java analysis direction is recorded with prototype evidence
  or explicit next actions.
- Integration-contract docs and changelog entries are updated before tagging.
- CI, golden tests, and relevant contract tests pass for the release candidate.
