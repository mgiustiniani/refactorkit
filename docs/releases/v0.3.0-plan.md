# RefactorKit v0.3.0 Release Plan

Status: draft for the post-`v0.2.0-beta` development slice.

Target implementation version on main: `0.3.0-SNAPSHOT`.
Current beta contract API baseline: `0.2`.

## Release objective

`v0.3.0` should make RefactorKit's integration metadata and compatibility rules
clearer before the project moves toward `v1.0.0`. The release is not intended to
weaken or replace the existing safety workflow: every mutating operation remains
previewable, diagnosable, traceable, and rollbackable.

## Goals

1. Expose consistent version/API metadata across CLI, daemon JSON-RPC, LSP, and
   MCP surfaces.
2. Define the semantic versioning and deprecation policy for beta-contract
   integration surfaces.
3. Decide and prototype the compiler-backed Java analysis direction needed before
   `v1.0.0`.
4. Broaden protocol-contract documentation for version metadata, capability
   discovery, and read-only integration checks.
5. Keep the `v0.2.0-beta` safety model intact: preview before apply, snapshot
   validation, affected-file reporting, diagnostics, transaction logs, and
   rollback metadata.

## Workstreams

### P0 — Version and API metadata

- Keep centralized metadata for implementation name, implementation version, and
  API version.
- Keep `apiVersion=0.2` as the current beta contract baseline while main uses
  implementation version `0.3.0-SNAPSHOT`.
- Document and test CLI `--version`/`version`, daemon `server.version`, and
  LSP/MCP `serverInfo.version`.

### P1 — Semantic versioning and deprecation policy

Status: draft policy published in
[docs/compatibility-policy.md](../compatibility-policy.md). The integration
contract page links to the policy and keeps the detailed per-surface baseline.

- Define how implementation versions, API versions, prerelease tags, and
  snapshots relate to compatibility promises. Drafted.
- Document when a beta-contract breaking change requires a migration note,
  deprecation warning, or API-version change. Drafted.
- Define deprecation expectations for CLI commands/options, daemon methods and
  fields/errors, LSP commands/metadata, and MCP tools/resources/prompts.
  Drafted.

### P2 — Compiler-backed Java analysis planning

Status: first prototype, class/interface override relation evidence, and exact signed
`renameMember` overload/refusal evidence available. ADR 0008 records Eclipse JDT
as the primary compiler-backed Java analysis prototype candidate for parsing,
bindings, symbol identity, and reference precision.
`modules/refactorkit-java` now includes an early `JdtJavaSemanticAnalyzer` backed
by `org.eclipse.jdt.core` `3.37.0`. It parses Java source at Java 21 compliance,
configures source roots from the scanned `ProjectSnapshot`, and collects class,
interface, enum, record, method, field, and constructor declarations with
qualified names, source path and line, optional binding keys, and evidence
labels. Member qualified names now include signatures, such as
`com.acme.User#displayName()`, `com.acme.Lookup#find(java.lang.String)`, and
`com.acme.Lookup#<init>()`.
Symbols expose `ownerQualifiedName`, `memberSignature`, and `sourceRange`;
references expose `symbolSignature` and `sourceRange`.

Current analyzer tests cover method/field identity and binding-key matched
references, distinguishing same-simple-name imports (`com.acme.right.Service`
versus `com.acme.left.Service`), parsing interface/enum/record declarations,
nested type/member ownership, overloaded method disambiguation, constructor
identity and references via `ClassInstanceCreation`, source-visible
class/interface override relation evidence, unresolved external type warnings
with `JDT_PARSE` evidence, and source-root validation on representative Maven and
Gradle samples.
Override relation evidence is reported as `JdtJavaSemanticOverrideRelation`
entries for methods such as `com.acme.Child#find(java.lang.String)` overriding
`com.acme.Base#find(java.lang.String)` and `DefaultLookup#find(java.lang.String)`
implementing/overriding `LookupApi#find(java.lang.String)`, combining JDT
override/subsignature checks with scanner-visible inheritance records as a
conservative fallback. The
sample checks find expected type symbols (`samples/java-maven-simple` with
`com.example.UserManager` and `samples/java-gradle-simple` with
`com.example.UserService`) plus method symbols on both sample shapes.

Rename-member previews now include exact signed method overload selection for
the proven JDT slice plus conservative refusal for inheritance-sensitive cases.
For a signed selector such as `com.example.Lookup#find(java.lang.String)`,
`JavaRenameMemberPlanner` uses the selected JDT declaration/reference source
ranges and matching binding key to rename only that overload. The signed flow
refuses when JDT reports parse/classpath warnings, when the selector does not
resolve to exactly one JDT method candidate, when no binding key is available,
or when the selected method participates in an override/inheritance relation,
including interface implementation selectors such as
`com.example.LookupApi#find(java.lang.String)`. Unsigned member rename remains a
lexical fallback and may still rename all overloads with warnings.

Read-only Java adapter and protocol lookup now uses the same clean JDT binding
slice for exact signed member selectors and source positions. `searchSymbols`,
`findSymbol`, `findReferences`, and `resolveSymbol` can resolve IDs such as
`com.example.Lookup#find(java.lang.String)`, and daemon JSON-RPC plus MCP
search/definition/reference surfaces, CLI definition/reference surfaces, and
LSP definition/reference/rename-position resolution can expose that exact
selector without changing existing unsigned lexical symbol behavior. Limited
change-signature accepts signed selectors for single-method cases by normalizing
the selector back to the method name, but overloaded methods remain refused.
OpenRewrite remains a possible future recipe/transformation backend rather than
the first symbol identity source.

Remaining prototype tasks:

- Validate classpath discovery robustness on representative Maven and Gradle
  samples; scanner-provided source roots and unresolved-type warnings now have
  sample evidence, but broader classpath behavior is not yet proven.
- Continue hardening selected-member identity/reference coverage beyond the
  current record discovery, nested ownership, exact signed overload,
  constructor, override relation, and unresolved-type prototype tests,
  especially broader scoped cases.
- Implement and prove override-aware propagation for signed member rename before
  allowing JDT rename plans across inheritance hierarchies.
- Expand authoritative JDT planner decisions beyond the current signed-method
  mutating slice and read-only signed selector lookup only when selected cases
  prove safe.
- Keep warnings/refusals for unsupported override propagation, framework,
  generated-code, broader overload, and classpath cases.
- Avoid claiming full `v1.0.0` semantic coverage until implementation and tests
  prove it.

### P3 — Broader protocol contracts

- Expand integration-contract documentation for read-only metadata and capability
  discovery surfaces.
- Keep daemon, LSP, and MCP contract tests aligned with documented beta-contract
  behavior.
- Preserve scoped access rules: no arbitrary filesystem access through MCP and no
  workspace modification outside preview/apply flows.

## Non-goals

- No `v1.0.0` API freeze in this slice.
- No weakening of preview/apply/rollback safety requirements.
- No IDE-specific or UI-specific coupling.
- No claim that Java analysis is fully compiler-backed until the implementation,
  tests, and docs prove it.

## Exit criteria

- Version/API metadata is documented and tested on all advertised integration
  surfaces.
- The SemVer/deprecation policy is published in project docs.
- The compiler-backed Java analysis direction is recorded with prototype evidence
  or explicit next actions.
- Integration-contract docs and changelog entries are updated before tagging.
- CI, golden tests, and relevant contract tests pass for the release candidate.
