# Migrating to RefactorKit v0.7.x

Status: active migration notes for `0.7.0-SNAPSHOT`.

## Compatibility baseline

`v0.7.x` retains API version `0.2`. Existing beta CLI commands, daemon JSON-RPC
methods, MCP tools and preview/apply/rollback envelopes are not renamed or
removed. Consumers that use only the previous operation inventory require no
request-shape migration.

Check compatibility through:

```bash
refactorkit version
```

or daemon `server.version`. Both report implementation and API versions.

## Additive Java/Maven behavior

### Explicit Java platform authority

Maven projects with an effective `maven.compiler.release` use exact attested Java
platform signatures. Source-only projects no longer gain semantic mutation
authority from the VM that happens to run RefactorKit. Configure a qualified JDK
platform input or handle the typed platform-unavailable diagnostic and keep the
operation in preview/refused state.

### Maven classpath semantics

Maven dependency management now keys variants by group, artifact, type and
classifier. Scope propagation distinguishes compile, provided, runtime, test and
system visibility. Clients should not assume that every reactor edge is exported
to downstream main compilation.

These changes refine diagnostics and safety decisions without changing legacy
wire envelopes.

## Maven ownership migration

`java.moveAcrossMavenModules` is new and experimental. It does not replace
`moveSourceRoot`:

- use `moveSourceRoot` when build ownership/POMs must remain unchanged;
- use `java.moveAcrossMavenModules` only when explicitly moving ownership between
  existing effective Maven modules and authorizing exact dependency coordinates.

CLI example:

```bash
refactorkit java move-across-maven-modules \
  --from source/src/main/java \
  --to destination/src/main/java \
  --dependency-pom consumer/pom.xml \
  --source-group-id com.example --source-artifact-id source --source-version 1 \
  --destination-group-id com.example --destination-artifact-id destination \
  --destination-version 1 \
  --root /workspace
```

Preview is the default. Add `--apply` only after reviewing both the Java renames
and exact POM diff. Roll back with the returned transaction ID through the normal
`patch rollback`, daemon or MCP route.

Property-backed, inherited, profile-dependent or ambiguous dependency origins
refuse. Do not work around refusal with raw filesystem moves or XML serialization.

## Diagnostics consumers

Legacy daemon `diagnostics` remains available. Exact IDE-oriented consumers may
adopt additive `diagnostics.v2`, which includes request correlation, readiness,
snapshot/compiler authority and exact UTF-16 ranges. Adoption is optional during
API `0.2`; no legacy response field was removed.

## Rollback and recovery

Managed transaction schema remains backward-readable under the documented journal
compatibility policy. Normal rollback checks exact post-images. If startup reports
`transaction.recoveryRequired`, use explicit `patch recover`; read-only project
open never performs recovery or creates transaction metadata automatically.

## Known non-migrations

This release does not:

- promote API `1.0`;
- make named JPMS reactors authoritative;
- execute Maven/Gradle project plugins or scripts;
- silently rewrite reflection/framework/configuration strings;
- infer parent `<modules>` or Maven coordinate changes from package/source moves.

See [`v0.7.0-support-matrix.md`](v0.7.0-support-matrix.md) for qualified rows and
remaining limits.
