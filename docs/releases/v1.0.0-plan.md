# RefactorKit v1.0.0 Stable Release Plan

Status: active.

Current implementation baseline on main: `1.0.0-rc.1-SNAPSHOT` after verified
`v0.3.0` publication.
Current documented integration API baseline: `0.2`.
Stable target implementation/API baseline: `1.0.0` / `1.0`.

## Release objective

`v1.0.0` will be the first stable local RefactorKit release. It will freeze the
core patch, CLI, and daemon safety contracts; define stable LSP/MCP integration
boundaries; and ship compiler-backed Java operations only where semantic evidence
is proven. Unsupported, ambiguous, unresolved, generated, reflective, or
framework-string cases must remain explicit refusals or reviewed fallbacks.

Stable does not mean that every possible Java refactoring is implemented. It
means that advertised stable operations have versioned contracts, deterministic
preview/apply/rollback behavior, tested semantic boundaries, and documented
failure modes.

## Release sequence

| Release | Purpose | Tag gate |
|---------|---------|----------|
| `v0.3.0` | Publish the JDT analysis foundation, capability discovery, and post-beta compatibility policy | Current `v0.3.0` exit criteria, green CI, release runtime smoke |
| `v1.0.0-rc.1` | Freeze the proposed `apiVersion=1.0` contract and run stable acceptance, performance, packaging, and migration validation | All mandatory implementation workstreams complete; no known data-loss defect |
| Additional `v1.0.0-rc.N` | Correct release-blocking defects without bypassing acceptance gates | Required when an RC changes a stable contract or artifact |
| `v1.0.0` | Publish and deploy the stable runtime | Clean RC acceptance, green tag workflow, verified assets and provenance |

No release tag may be moved or recreated. A failed candidate receives a new
commit and, when already tagged, a new RC number.

## Transactionality audit gate

The open findings in [Transactionality and requirements audit](../transactionality-audit.md)
are normative release-input evidence. Current flows are preflighted compensatable
batches, not crash-safe durable transactions. `v1.0.0-rc.1` remains blocked until
critical transaction, journal, rollback-conflict, concurrency, LSP-boundary, and
same-file edit-coordinate findings are closed or the advertised contract is
narrowed precisely.

## Stable scope

### Stable product contracts

The following are stable at `v1.0.0`:

- core `ProjectSnapshot`, `TextEdit`, `FileEdit`, `WorkspaceEdit`, `PatchPlan`,
  transaction, stale-snapshot, overlap, workspace-boundary, apply, and rollback
  behavior;
- documented CLI command names, options, structured exit/error categories, and
  preview/apply/rollback workflow;
- documented daemon JSON-RPC methods, capability metadata, request/response
  fields, structured errors, and workspace-scoped mutation rules;
- runtime package layout, embedded-Java launcher behavior, checksum, SBOM, and
  release provenance metadata;
- Java scan, symbol, definition, reference, diagnostics, and the operations
  explicitly classified as stable in `docs/integration-contracts.md`.

### Stable-with-conservative-refusal Java operations

These operations may refuse unresolved or unsafe cases, but must never silently
fall back to an unsafe global rewrite:

- rename type, including class, interface, enum, record, and annotation type;
- signed method and annotation-element rename for complete proven source-visible
  families;
- exact field rename;
- move type;
- organize imports;
- safe delete;
- documented limited extract method and change signature cases.

Every operation must state whether its evidence is JDT binding-backed, lexical
fallback, or refusal. Stable acceptance will define the exact fallback policy per
operation. Lexical behavior may remain only where tests prove it cannot claim or
perform a semantic edit incorrectly.

### Experimental or deferred at `v1.0.0`

The following remain explicitly experimental unless promoted by a later plan
update and acceptance evidence:

- automatic execution of Maven or Gradle to resolve dependencies;
- generated-source rewriting and annotation-processor output rewriting;
- reflection, arbitrary string/configuration, and unknown framework convention
  rewriting;
- arbitrary annotation-element change signature or default-value editing;
- unrestricted extract method and change signature beyond documented supported
  control-flow/signature shapes;
- non-Java semantic refactoring beyond documented structural adapters;
- cloud, collaborative, or IDE-specific UI behavior.

`v0.3.0`/`ba114134be605b7b0076e7471f8111c081a8aaa3`. The downloaded runtime

- Set implementation version to `0.3.0` on the release commit while keeping

- Make release automation distinguish prerelease tags from stable tags.

### S1 — Freeze the stable safety model

Status: in progress and release-blocked by the remaining audit findings among
TX-001 through TX-010. TX-009/TX-010 are closed through coordinate normalization,
cross-entry overlap validation, complete preflight rendering, and line/character
bounds. TX-003 is closed: generated UUIDv4 transaction IDs,
contained normalized log paths, symbolic-link/non-regular-file refusal,
owner-only permissions where supported, coded corruption failures, and invalid-ID
protocol tests are implemented. Ordered file-state preflight also runs before the
first write, and paths traversing symbolic-link components are refused.
Managed first-party apply/rollback now uses a non-blocking workspace lock, and
snapshot-aware apply revalidates affected paths under that lock. The hash-only
apply overload is removed, and snapshot hashes bind module/source-root metadata,
extensions, ignore policy, language IDs, paths, and contents. Under-lock
engine-owned source rescans reject added/missing/changed/omitted files. Java
classpath entries, prospective compiled outputs, local JAR directories, and
generated declaration files are fingerprinted and revalidated before journaling
(`TX-014` closed for declared source/classpath scope).
A versioned write-ahead journal is now persisted durably before mutation, owns
lifecycle transitions through apply/rollback/recovery, retains pre/post images,
and performs conflict-aware startup compensation for interrupted managed writes.
Managed workspace results are now fully staged in same-directory temporary files,
flushed, atomically replaced, and followed by directory flushes; unsupported
workspace-root capability is reported/refused. Normal rollback now verifies exact
post-images and exposes stable conflict/recovery codes; explicit force is audited.
Daemon apply/rollback now refreshes the stored session snapshot, clears stale
pending plans, and returns the current hash (`TX-018` closed). LSP native edits
now use full-sync open-buffer overlays, monotonic/exact document
versions, explicit client-managed/no-rollback metadata, capability refusal, and
managed-write open-buffer guards (`TX-007` closed). Recipes evaluate steps against
one evolving staged snapshot and commit one recipe-wide plan/WAL transaction;
planning failure and no-op execution write no journal record (`TX-008` closed).
Integrity/quarantine, kill/disk-full/power-loss fault injection, mounted-store
capability coverage, and broader property/stress evidence remain open.

- Specify normative patch/apply/rollback invariants and stable structured errors.
- Add property/stress tests for edit ordering, overlaps, mixed create/modify/
  rename/delete plans, rollback restoration, stale snapshots, symlink/path
  traversal, interrupted apply recovery, and transaction-log corruption.
- Define atomicity guarantees by filesystem capability and make partial-failure
  behavior observable and recoverable.
- Prove that no production workspace write bypasses `PatchEngine`.

### S2 — Complete compiler-backed Java boundaries

- Upgrade Eclipse JDT to a version that supports the advertised Java source range
  through Java 25 while preserving Java 8-compatible RefactorKit bytecode.
- Detect/configure project source compliance instead of assuming one source
  level; test representative Java 8, 11, 17, 21, and 25 constructs.
- Make source roots, module relationships, JDK types, compiled outputs, local
  JARs, and explicit/generated dependency classpaths deterministic and visible.
- Define clean-evidence criteria per operation. Unresolved bindings must cause a
  documented fallback or refusal, never silent semantic authority.
- Add semantic tests for nested/member types, generics, annotations, static
  imports, records, sealed types, pattern constructs, overloads, overrides,
  package visibility, external declarations, and generated-source boundaries.
- Promote only proven planners to stable JDT authority; document remaining
  conservative limitations.

### S3 — Diagnostics and validation gates

- Provide compiler-backed parse/type diagnostics with stable severity, source
  range, evidence, and error-category fields.
- Compare diagnostics before preview, after preview simulation, after apply, and
  after rollback where the operation contract requires it.
- Refuse apply when configured validation gates regress beyond the approved plan.
- Add representative Maven, Gradle, multi-module, Spring, JPA, and annotation
  sample acceptance tests without silently executing project build scripts.

### S4 — Stable CLI and daemon API `1.0`

- Inventory every documented CLI and daemon field, method, error, and exit code.
- Publish `apiVersion=1.0`, compatibility/deprecation rules, and migration notes
  from API `0.2`.
- Add machine-readable capability schemas and contract snapshots.
- Add backward-compatibility tests for supported `0.2` requests during the
  documented transition window or explicitly version/refuse them.
- Ensure unknown fields, unknown methods, invalid params, project lifecycle,
  stale plans, and rollback errors have deterministic structured responses.

### S5 — LSP and MCP stable boundaries

- Classify each LSP capability, command, MCP tool/resource/prompt as stable or
  experimental.
- Add protocol initialization/capability, definition/reference, diagnostics,
  preview/apply/rollback, refusal, cancellation, and malformed-input tests for
  stable surfaces.
- Preserve MCP least privilege: no arbitrary filesystem tool and no write outside
  preview/apply flows.
- Document client compatibility and unsupported protocol behavior.

### S6 — Performance and resource bounds

- Define reference workspaces and measurable scan, index, preview, apply, and
  memory budgets.
- Add repeatable benchmark/stress jobs for large file counts, large files, dense
  references, and multi-module graphs.
- Bound daemon pending plans, caches, request sizes, result counts, and transaction
  metadata growth; expose deterministic limit errors.
- Verify cancellation/timeout and recovery behavior for long-running analysis.

### S7 — Supply chain, packaging, and reproducibility

- Keep the Linux x86_64 runtime self-contained and runnable without global Java.
- Produce checksum, SBOM, and GitHub artifact provenance/attestation for release
  assets; document verification commands.
- Remove avoidable timestamps/non-determinism or document reproducibility limits;
  compare two clean package builds in CI.
- Verify archive layout, executable permissions, embedded runtime, license files,
  version metadata, and `JAVA_HOME`-unset smoke commands.
- Make prerelease/stable GitHub Release classification derive from the tag.

### S8 — Documentation and migration closure

- Reconcile README, operation docs, safety model, integration contracts, ARC42,
  C4, ADRs, changelog, and release notes with actual stable behavior.
- Publish a `v0.2`/`v0.3` to `v1.0` migration guide and deprecation inventory.
- Document install, upgrade, downgrade, rollback, incident withdrawal, and local

- Remove stale statements that describe already-completed JDT work as purely
  lexical, while retaining real limitations.

## Stable acceptance matrix

Before `v1.0.0-rc.1`, acceptance must cover:

| Area | Required evidence |
|------|-------------------|
| Build | `./gradlew build` on clean checkout and CI |
| Product regression | all golden cases pass; every patch-producing stable operation has success and refusal coverage |
| Safety | stale, overlap, unsafe path/symlink, mixed edit, interrupted apply, rollback hash restoration |
| Java compatibility | representative source levels 8, 11, 17, 21, and 25 |
| Semantic precision | same-name disambiguation, overload/override families, nested/generic/annotation/static cases, external-boundary refusal |
| Protocols | CLI, daemon, stable LSP, and stable MCP contract suites |
| Samples | Maven, Gradle, multi-module, Spring, and JPA scan/diagnostic/preview workflows |
| Runtime | package, unzip, checksum, SBOM/provenance, `JAVA_HOME`-unset smoke |
| Performance | documented reference-workspace budgets pass without unbounded growth |
| Documentation | stable contracts, migration, risks, and release notes reviewed |

## Release-blocking defects

Any of the following blocks RC/stable publication:

- silent data loss, write outside workspace root, unrollbackable managed write, or
  stale-plan application;
- semantic edit to a binding other than the selected binding in a stable JDT
  operation;
- hidden lexical fallback where the preview claims compiler-backed authority;
- undocumented stable API break or inconsistent version/capability metadata;
- release artifact that requires a globally installed Java runtime;
- missing/invalid checksum, SBOM/provenance failure, or failed packaged smoke;
- red required CI, golden, contract, compatibility, or acceptance test.

## RC procedure

1. Freeze scope and classify every advertised surface.
2. Set implementation/API versions to the RC values.
3. Generate migration notes, release notes, SBOM/provenance configuration, and
   acceptance record.
4. Run all mandatory acceptance commands from a clean checkout.
5. Push the release commit and wait for green branch CI.
6. Create the immutable RC tag.
7. Verify tag CI, assets, checksum, SBOM/provenance, and packaged smoke.
8. Record findings. Fix release blockers on a new commit and issue a new RC tag.

## Stable publication procedure

1. Confirm the last RC has no unresolved release blocker and its acceptance
   evidence is reproducible on the stable commit.
2. Set implementation version to `1.0.0` and `apiVersion` to `1.0`.
3. Complete final changelog, migration, support, and release notes.
4. Run clean build, golden, contract, Java compatibility, stress, package, and
   smoke verification.
5. Push the stable commit and require green branch CI.
6. Create immutable tag `v1.0.0`.
7. Verify the GitHub Release is not marked prerelease and is the latest release.
8. Download and verify checksum, SBOM/provenance, archive layout, embedded runtime,
   and smoke workflows independently of the build tree.
9. Deploy the verified artifact locally and record the deployed hash/version.
10. Move main to the next development snapshot only after publication succeeds.

## Completion rule

`v1.0.0` is complete only when the stable tag, GitHub Release, verified runtime
assets, stable API metadata, migration documentation, CI evidence, and local
post-publication smoke/deployment record all agree. Passing unit tests alone is
not sufficient.
