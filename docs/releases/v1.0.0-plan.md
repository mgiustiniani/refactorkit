# RefactorKit v1.0.0 Supreme Multi-Language Roadmap

Status: active long-range product roadmap.

Current baseline: `v0.6.0` adds the managed TypeScript/JavaScript semantic
foundation after `v0.5.0` completed the first multi-language kernel and
cross-platform runtime band, building on the published
[`v0.4.0` hardening plan](v0.4.0-plan.md). No `v1.0.0-rc.N` is currently
scheduled. RefactorKit has no deadline-driven need to freeze an incomplete
contract.

## Supreme 1.0 objective

`v1.0.0` is the first stable, IDE-grade, deterministic multi-language
refactoring platform. It is not a Java-only API freeze and not a collection of
shallow rename wrappers. It combines:

1. a language-neutral, previewable, diagnostics-gated, atomic/compensatable,
   rollbackable patch platform;
2. deep idiomatic refactoring catalogues backed by the strongest semantic engine
   available for each language;
3. cross-language and ecosystem-aware project models;
4. explicit capability/evidence matrices with conservative refusal where proof
   is incomplete;
5. bounded library, CLI, daemon, LSP, and MCP integrations;
6. verifiable, self-contained or explicitly toolchain-qualified distributions.

Java remains the first and reference adapter and is expected to have the widest
catalogue. Kotlin, Scala, Groovy, Clojure, C#, TypeScript/JavaScript, C/C++,
Objective-C, Swift, Go, Rust, and Python are not intentionally second-class. They target equivalent
IDE-grade quality and semantic safety through language-idiomatic operations,
not artificial command-for-command parity.

Clojure is the final planned language-family integration before the `1.0`
alpha/beta/RC stabilization cycle. Its initial arrival does not itself authorize
stable publication: all language and platform gates must pass together.

## Product principles

### Pre-1.0 scope admission

Every language or product capability explicitly requested and approved by the
product owner before stable publication is assigned to a documented `0.x` or
`1.0` development milestone and completed before `v1.0.0`. New ideas do not
silently bypass architecture, safety, evidence, resource, migration, or
acceptance gates, and may extend the pre-1.0 roadmap. There is no deadline-based
scope cut that moves approved functionality behind `1.0`; any exception requires
an explicit superseding product decision and ADR update.

### Equivalent quality, idiomatic capability

Every adapter targets equivalent standards for project discovery, semantic
identity, diagnostics, preview, managed apply, rollback, generated-code policy,
timeout/cancellation, evidence, testing, and documentation. Operation catalogues
remain language-specific:

- JVM OO languages expose hierarchy, signature, member, package/module, extract,
  inline, migration, and framework operations;
- Clojure exposes namespace/var/form/protocol/macro-aware structural operations;
- C exposes component, header, include, build-target, API-prefix, and ABI-aware
  operations;
- C++ additionally exposes namespace, class hierarchy, template, module, and
  Clang-backed operations;
- Objective-C exposes selector, protocol, category, property, header/module,
  nullability, ARC, framework and Swift-interop-aware operations;
- Swift exposes module/type/member/protocol/extension, async/actor, macro,
  SwiftPM/Xcode, Objective-C interoperability and source-compatible migration
  operations;
- Rust exposes module/trait/Cargo-aware operations;
- Python and dynamic Groovy/JavaScript expose evidence-qualified behavior rather
  than false static certainty.

### Semantic edits never bypass the core

Compiler APIs and language servers are proposal providers, not filesystem
writers:

```text
semantic backend -> normalized WorkspaceEdit -> PatchPlan -> preview
-> diagnostics/evidence/approval -> PatchEngine -> transaction/rollback
```

Every external edit is treated as untrusted until paths, ranges, overlap,
snapshot freshness, capability authority, diagnostics, and workspace scope pass.

### Capability truth over marketing parity

Every language/operation cell is classified as:

```text
STABLE | EXPERIMENTAL | STRUCTURAL | REFUSED | NOT_APPLICABLE
```

The runtime reports backend, backend version, evidence kind, completeness,
project/toolchain configuration, and known blind spots. Hidden lexical fallback
is never stable authority.

## Shared platform milestones

### P1 — Hardened foundation (`v0.4.x`)

- Publish the completed managed transaction, Java semantic-boundary, and staged
  compiler-diagnostics work.
- Complete beta contract inventory, protocol hardening, resource limits,
  SBOM/provenance, reproducibility analysis, and documentation.
- Keep API pre-stable; do not issue an early `1.0` release candidate.

### P2 — Cross-platform runtime and multi-language kernel (`v0.5.x` target)

- Publish natively built and accepted self-contained Linux x86_64, Windows
  x86_64, macOS x86_64, and macOS arm64 runtimes with per-platform checksums,
  SBOMs, attestations, managed transaction smoke, and filesystem qualification.
- Add an adapter registry and route files/projects/requests by language.
- Support mixed-language snapshots and aggregated diagnostics.
- Generalize evidence beyond `JDT_BINDING` while retaining compatibility detail:
  compiler binding, language server, parse tree, structural, lexical fallback.
- Implement a production external semantic-process manager: lifecycle,
  initialization, capability negotiation, framing, overlays, document versions,
  timeout, cancellation, restart, stderr/output bounds, provenance, shutdown.
- Implement real Tree-sitter bindings for structural parsing where appropriate.
- Normalize external workspace edits into core coordinates and validation.
- Add per-language capability schemas and contract snapshots.
- Add toolchain discovery without silently executing project build scripts.

Delivered early by the `0.5.x` Java reference slice: offline Maven effective
models, per-module main/test/generated environments, hash-bound parent/BOM/local-
artifact/module-edge evidence, typed root-cause cascade suppression, and
rename-only source-root ownership relocation. P2B generalizes and productionizes
that slice; it does not mark the full kernel complete.

### P3 — Shared ecosystem models

#### JVM family

Create internal reusable JVM components without extracting a separate product.
ADR 0010 makes this an explicit Build Model SPI gate before deep JVM adapter
expansion.

Completed reference evidence:

- Maven parent/property/release/dependency-management/BOM/reactor modeling;
- independent main/test/generated Java environments and typed missing-input roots;
- hash-bound build inputs and local artifacts;
- package/FQCN identity separated from source-root/module ownership;
- first language-neutral Build Model SPI contracts, validated source-set graph,
  deny-by-default discovery policy, hash-bound snapshot projection, and redacted
  daemon/MCP summary.

Remaining shared model work:

- API-versioned retirement of compatibility `Module` projection for external
  library consumers, completion of capability schemas, and production hardening
  of the explicit Maven/Gradle providers;
- Maven production edge coverage and credential/repository policy;
- Gradle effective model with explicit executable-build-code policy;
- sbt/BSP metadata and cross-language JVM identity.

Target components:

- module/source-set graph;
- Maven, Gradle, Gradle Kotlin/Groovy DSL, sbt/BSP metadata;
- JDK, target bytecode, classpath, JAR/output/generated-source evidence;
- JVM binary names and method/field descriptors;
- cross-language declaration/reference identity;
- Java/Kotlin/Scala/Groovy/Clojure visibility and incomplete-adapter refusal.

A cross-language operation is applicable only when every relevant adapter and
boundary is available or the missing surface is explicitly excluded and approved.

#### Clang family

Create shared C/C++/Objective-C infrastructure:

- compilation database and per-translation-unit flags;
- include graph, public/private header model, source/header ownership;
- CMake/Meson/Bazel/Make/Visual Studio build-manifest adapters as individually
  advertised capabilities;
- preprocessor configuration evidence;
- target/toolchain/sysroot identity;
- ABI/API risk reporting;
- C, C++, Objective-C and Objective-C++ operation matrices kept separate;
- Objective-C selector/message-send, protocol/category, ARC/nullability,
  framework/module-map, Xcode target and Swift bridging-header evidence.

#### Apple Swift ecosystem

Create shared Swift infrastructure without coupling core to Xcode:

- Swift compiler and SourceKit-LSP version/toolchain provenance;
- SwiftSyntax source-preserving rewrite support;
- SwiftPM package/target/product/plugin/macro graph;
- bounded declarative Xcode project/workspace, scheme, configuration and SDK
  metadata with no implicit build-script phase execution;
- module, target triple, SDK, deployment target, conditional-compilation and
  generated-source evidence;
- Swift/Objective-C declaration identity, generated bridging surfaces and
  incomplete-boundary refusal.

## Language delivery sequence

Minor versions are planning bands, not deadlines. A language may require more
than one `0.x` release; it is promoted only after its own acceptance evidence.
Java depth advances in every band rather than pausing while breadth expands.

| Planning band | New language/ecosystem focus | Concurrent Java/platform depth |
|---|---|---|
| `0.4.x` | Harden current Java and core foundation | Existing operations, diagnostics, safety, importer |
| `0.5.x` | Multi-language kernel and real structural/LSP infrastructure | Complete rename catalogue and reusable refactoring primitives |
| `0.6.x` | TypeScript + JavaScript | Advanced signature and import/move work |
| `0.7.x` | Kotlin and Java/Kotlin interoperability | Cross-language JVM identity; extract/inline expansion |
| `0.8.x` | Python | Package/module and dynamic-evidence policies |
| `0.9.x` | Go | Hierarchy/data-flow/refactoring framework work |
| `0.10.x` | Scala 2/3 and JVM interoperability | JVM family transactions and migrations |
| `0.11.x` | C, C++ and Objective-C through shared Clang infrastructure | Framework-aware Java and large-workspace performance |
| `0.12.x` | Swift, SwiftPM/Xcode model and Objective-C interoperability | Native cross-language evidence and Java performance |
| `0.13.x` | Groovy, Gradle Groovy DSL, Spock | Mixed static/dynamic JVM evidence |
| `0.14.x` | C# and Roslyn solution model | Advanced recipes and ecosystem migrations |
| `0.15.x` | Rust and Cargo/rust-analyzer model | Cross-adapter acceptance hardening |
| `1.0` development | Clojure plus global stabilization | Final Java IDE-grade catalogue and all-language acceptance |

The sequence may change based on backend maturity and user demand, but removing a
language from the `1.0` target requires an explicit roadmap decision and ADR.
ADR 0010 and the Maven/source-ownership delivery do not reorder this language
sequence; they strengthen the shared `0.5.x` foundation required before it.

## Deep adapter targets

### Java — reference and widest catalogue

Backend: Eclipse JDT compiler/search/rewrite/refactoring facilities, selected
OpenRewrite recipes, and RefactorKit-owned risk/evidence analysis.

Required catalogue families:

- rename every source-visible type/member/parameter/local/type-parameter/record/
  annotation/package/module shape where semantic proof exists;
- move types, members, packages, modules, and source-set/module ownership;
- safe delete for types, members, parameters, packages, and modules;
- comprehensive signature changes across overload/override/source-visible family;
- deterministic whole-file/range/changed-lines formatting using project style
  evidence, with idempotence and minimal-diff guarantees;
- extract method/variable/constant/field/parameter/interface/superclass/delegate/
  class/parameter-object where control/data flow permits;
- inline local/field/constant/method/parameter with evaluation-order proof;
- pull up, push down, hierarchy/interface/superclass/delegation operations;
- encapsulation, constructor/factory, static/instance, lambda/anonymous/method-
  reference, class/record, and related modernization operations;
- imports, package, JPMS, API migration, Java-version and ecosystem recipes;
- Spring, Jakarta/JPA, Jackson, testing, serialization, ServiceLoader,
  reflection/configuration and generated-code risk awareness.

The `moveSourceRoot` slice now satisfies deterministic same-package main/test
source ownership relocation when build metadata already defines the post-image.
Descriptor-editing ownership migrations, package/module moves, and broader Java
catalogue families remain roadmap work.

### TypeScript and JavaScript

Backend: TypeScript compiler API/tsserver with an LSP bridge where useful.
Target solution-aware rename, move-file/import updates, organize imports,
extract/inline, signature, module/export and JS/TS migration operations. Dynamic
JavaScript certainty is reported separately from typed TypeScript evidence.

### Kotlin

Backend: Kotlin compiler/Analysis API; language server only as a bootstrap layer.
Target declaration/property/function/package/move/signature/extract/inline/
hierarchy operations, extension/suspend/companion/data/sealed/expect-actual and
multiplatform awareness, plus Java interoperability and Android/Spring risks.

### Python

Backend: Pyright plus a concrete-syntax rewrite engine such as LibCST and
project/import graph support. Target module/symbol rename, move/import updates,
extract/inline and idiomatic transformations with explicit dynamic/reflection
confidence boundaries.

### Go

Backend: gopls plus Go parser/type APIs where needed. Target module/package,
rename, move, imports, extract/inline, signature, interface and safe-delete
operations with go.mod/workspace awareness.

### Scala

Backend: Metals, BSP, SemanticDB, Scalameta/Scalafix, and Scala 3 TASTy where
required. Target Scala 2/3 rename/move/extract/inline/signature/hierarchy,
trait/object/companion/given/using/implicit/extension/opaque-type operations and
Java/Kotlin interoperability.

### C and C++

Backend: clangd for navigation/common edits and Clang tooling for deep rewrite
and analysis. C++ has language namespaces; C does not. Directory layout is never
silently equated with a namespace.

Required independent operations include:

- `renameCppNamespace` for declarations, qualified uses, aliases, using
  directives, ADL-sensitive/template/specialization and forward-declaration cases;
- `renameCppModule` for C++20 module names, imports, partitions, and build metadata;
- `renameCSymbolPrefix` for proven functions/types/variables/enums and separately
  evidenced macros/include guards;
- `moveSourceTree` and `movePublicHeader` with include and build-manifest updates;
- `relocateComponent`, a composite plan covering source/header paths, include
  paths, build target, optional C API prefix, optional C++ namespace/module,
  forwarding headers, API/ABI warnings, and affected translation units;
- extract/inline/signature/hierarchy/template/modernization operations where
  Clang evidence and configuration coverage permit.

A logical component consists of source/header paths, include roots, build target,
optional C symbol prefix, optional C++ namespace, and optional C++ module. Path,
namespace, module, target, and prefix may be moved together or independently.
Every preview states which compilation configurations were validated.

### Objective-C and Objective-C++

Backend: clangd plus Clang tooling/indexing, sharing compilation-database and
header/module infrastructure with C/C++ while retaining separate language
capabilities. Target selector and symbol rename, method/property/signature
changes, protocol/category/class moves, header/import/module-map updates,
extract/inline, nullability/ARC modernization, framework/Xcode target migration,
and Objective-C/Swift interoperability. Dynamic dispatch, runtime selector
construction, KVC/KVO, Interface Builder/storyboard references, preprocessor
configurations, generated headers and external framework consumers require
explicit evidence, warning or refusal. Objective-C++ is never treated as plain
Objective-C or plain C++ when either semantic surface is relevant.

### Swift

Backend: Swift compiler facilities, SourceKit-LSP and SwiftSyntax, with SwiftPM
and bounded declarative Xcode project models. Target solution-wide symbol/type/
member/module rename, file/type moves with imports, signature changes, extract/
inline, protocol/extension/conformance operations, async-await/actor and
ownership modernization, package/target migrations, and API evolution recipes.
Macros, result builders, conditional compilation, generated Objective-C headers,
C/Objective-C bridging, Apple SDK availability, binary frameworks and build
plugins are configuration-qualified evidence or conservative refusal.

### Groovy

Backend: Groovy compiler AST and mature language-server/refactoring facilities.
Target class/method/property/move/extract/inline/signature/closure/trait and JVM
interoperability operations, Gradle DSL and Spock awareness, with separate
`CompileStatic`, `TypeChecked`, and dynamic evidence.

### C#

Backend: Roslyn compiler, workspace, solution, code-fix and refactoring APIs.
Target solution-wide rename/move/signature/extract/inline/hierarchy/encapsulation,
constructor/factory, async, LINQ, nullable, record/class and ecosystem operations,
with MSBuild/NuGet/source-generator evidence. C# is a top-tier deep adapter, not a
baseline-only integration.

### Rust

Backend: rust-analyzer and compiler/Cargo facilities where stable. Target item/
module rename/move, imports, extract/inline, trait/signature and idiomatic match/
ownership-aware transformations. Macro/procedural-macro/configuration coverage is
explicit evidence or refusal.

### Clojure — final planned language integration

Backend: clojure-lsp, clj-kondo, rewrite-clj and namespace/project tooling. Target
var/namespace rename, move form, extract/inline, thread/unthread, introduce let,
destructuring, require/import cleanup, protocol/multimethod, reader-conditional,
macro-boundary and structural form operations. IDE-grade means idiomatic Clojure
capability, not imitation of Java classes.

## Universal stable acceptance gate

Every advertised stable adapter must demonstrate:

- deterministic project/toolchain discovery and version/provenance evidence;
- semantic symbol identity, definition, references, diagnostics and generated-
  source policy;
- deterministic project-style-aware formatting exposed through previewable
  managed operations and correctly classified native editor edits;
- a documented idiomatic advanced-refactoring catalogue with success, refusal,
  ambiguity, external-boundary and framework/build cases;
- exact source ranges and normalized non-overlapping workspace edits;
- preview, affected files, evidence, risk, warnings and staged diagnostics;
- explicit authorization, managed `PatchEngine` apply, transaction log, rollback,
  stale snapshot and conflict behavior;
- bounded requests/results/memory/process output, cancellation, timeout and crash
  recovery;
- representative single/multi-module and mixed-language samples;
- unit, property, golden, integration, protocol, agent-simulation, stress and
  packaged-runtime/toolchain acceptance;
- stable capability schema, migration notes, limitations and support policy.

Operations without this evidence remain experimental, structural, refused, or
not applicable; they do not inherit stability merely because another operation
for the language is stable.

## Cross-language acceptance

At minimum, `1.0` validates:

- Java/Kotlin;
- Java/Kotlin/Scala;
- Java/Groovy;
- Java/Clojure at JVM-visible boundaries where source semantics permit;
- TypeScript/JavaScript;
- C/C++/Objective-C/Objective-C++ mixed Clang targets;
- Swift/Objective-C mixed frameworks or applications;
- C# multi-project solutions.

A semantic operation refuses a completeness claim when a relevant language
adapter, external binary boundary, generated source, framework string, macro
configuration, consumer project, or build configuration is unavailable.

## 1.0 stabilization sequence

After Clojure reaches its target catalogue:

1. freeze scope and capability matrices;
2. complete all-language and cross-language acceptance;
3. set the proposed API to `1.0` and publish migration/deprecation inventories;
4. issue immutable `1.0.0-alpha.N` builds for integration feedback;
5. issue immutable `1.0.0-beta.N` builds after feature completion;
6. issue immutable `1.0.0-rc.N` builds only after API and artifact freeze;
7. verify clean builds, benchmarks, checksums, SBOM, provenance/attestation,
   reproducibility limits, embedded/managed toolchains, archive layout and
   independent downloaded-artifact smoke;
8. publish `v1.0.0` only when no release-blocking safety, semantic, protocol,
   resource, packaging, migration or documentation defect remains.

Tags are immutable. Failed candidates receive new commits and version numbers.

## Definition of supreme 1.0 done

`v1.0.0` is complete only when the stable tag, release assets, API metadata,
capability matrices, all-language acceptance, Java reference catalogue,
idiomatic deep catalogues for every target adapter, cross-language evidence,
transaction safety, performance limits, migration/support documentation, and
independently verified distribution records agree.

Passing unit tests, adding Clojure, or exposing an LSP rename alone is not
sufficient.
