# RefactorKit v0.7.x Kotlin and JVM Interoperability Release Plan

Status: active development.

Baseline: published `v0.6.2`; current development version `0.7.0-SNAPSHOT`;
API `0.2` remains active until an explicit compatibility migration is accepted.

The `0.7.x` band may use multiple patch releases. A language capability is
promoted only when its own semantic, transaction, native-platform and packaged
acceptance evidence is complete.

## Objective

Deliver the first deep Kotlin semantic adapter and prove that the language-neutral
Build Model SPI and adapter kernel can support a second JVM language without
encoding Java-only assumptions. The band must establish conservative Java/Kotlin
interoperability while continuing Java depth and completing the explicitly
deferred advanced TypeScript/JavaScript operations from the `0.6.x` plan.

Kotlin support is not considered deep or stable because parsing, outline, or an
LSP rename works. Managed mutation requires hash-bound toolchain and project
evidence, semantic identity, exact diagnostics, normalized proposals, preview,
explicit authorization, `PatchEngine` apply, WAL, recovery and rollback.

## K1 — Kotlin toolchain and backend boundary

- [x] Add `refactorkit-kotlin` behind the existing language adapter registry; core
  does not depend on Kotlin compiler or IDE classes. The initial descriptor owns
  `.kt`/`.kts` routing but advertises only typed `REFUSED`/`NONE` capabilities
  until the semantic backend is qualified.
- [x] Select and document versioned provider `kotlin-compiler-explicit-v1` with
  backend `kotlin-compiler-embeddable-k2`. The first declarative qualification row
  is JDK 21 plus Kotlin 2.0.21; no semantic authority is implied. An external
  language server may bootstrap read operations but is never sufficient stable
  mutation authority by itself.
- [x] Discover configured JDK and Kotlin compiler/backend inputs explicitly, bind
  executable/JAR/version/classpath evidence with SHA-256 and a deterministic
  projection hash, double-read critical evidence, and refuse unsupported,
  workspace-local-by-default or drifting toolchains.
- [x] Execute no Java, Kotlin compiler, Gradle tasks/plugins, Maven lifecycle,
  compiler plugins, annotation processors, KSP, kapt or project scripts during
  default discovery.
- [ ] Define bounded process/in-process compiler lifecycle, cancellation, timeout,
  memory/output limits, shutdown and crash recovery for the selected backend.
- [ ] Publish qualified Kotlin/JDK/Gradle/Maven support rows and license/SBOM
  boundaries before any managed capability is advertised as supported.

## K2 — Kotlin and shared JVM project model

- [ ] Project Kotlin main/test/generated source sets through the existing Build
  Model SPI with exact module ownership, target JDK/bytecode, classpath, outputs
  and hash-bound descriptor evidence.
- [ ] Extend bounded declarative Maven and Gradle providers only where metadata is
  provable without executing build code. Plugin-derived or executable-only model
  data remains `PARTIAL`, `EXECUTION_REFUSED`, or unavailable.
- [ ] Model JVM binary names, Kotlin source declarations and callable/property
  identity without exposing compiler-internal handles in API `0.2`.
- [ ] Establish cross-language Java/Kotlin declaration and reference identity for
  supported JVM-visible types and members.
- [ ] Keep Kotlin/JVM, Kotlin Multiplatform, Android, generated code, compiler
  plugins, `expect`/`actual`, and script sources as separate capability/evidence
  cases. Unsupported boundaries must refuse rather than silently degrade.
- [ ] Validate the Build Model SPI against Kotlin so any remaining Java/Maven
  assumptions are removed from shared contracts rather than copied into the new
  adapter.

## K3 — Semantic read operations

- [ ] Provide bounded project symbol search, document symbols, definition,
  references and exact compiler diagnostics for qualified Kotlin/JVM projects.
- [ ] Define stable opaque Kotlin symbol IDs from durable semantic identity rather
  than source offsets or backend object identity.
- [ ] Preserve exact UTF-16/API coordinate conversion, source hashes, document
  versions and immutable workspace overlays where an external backend is used.
- [ ] Aggregate Java and Kotlin diagnostics without duplicate cascades; missing
  build/toolchain inputs produce concise typed roots.
- [ ] Expose backend, version, evidence, completeness and refusal reasons through
  library, CLI, daemon, LSP capability metadata and MCP.

## K4 — Managed Kotlin rename foundation

- [ ] Qualify semantic rename for an initial Kotlin/JVM catalogue covering classes,
  interfaces, objects, functions, properties, parameters and type parameters where
  backend proof is exact.
- [ ] Update constructors, callable/property references, imports, aliases and file
  names only when the semantic backend proposes and RefactorKit validates the
  complete operation.
- [ ] Add Java-to-Kotlin and Kotlin-to-Java reference coverage for at least one
  public type/member rename. If either side lacks exact evidence, return a typed
  cross-language refusal.
- [ ] Detect or warn about reflection, serialization names, Spring/Jakarta/Jackson
  strings, Android resources, generated sources, compiler plugins and unknown
  external consumers.
- [ ] Require preview, affected files, risk/evidence, before/staged diagnostics,
  explicit authorization, WAL, rollback, stale-snapshot refusal and lock-time
  toolchain/build-evidence revalidation.

## K5 — Idiomatic Kotlin depth

These capabilities may span later `0.7.x` patch releases but remain part of the
Kotlin band:

- [ ] Move declaration/file/package with import and Java interoperability updates.
- [ ] Organize imports and deterministic project-style-aware formatting.
- [ ] Change signature across overrides, overloads, default/named arguments and
  Java callers.
- [ ] Add bounded extract/inline operations with control/data-flow proof.
- [ ] Model extension receivers, suspend functions, companion/data/sealed/value
  classes, delegated properties and JVM annotations.
- [ ] Add explicit `expect`/`actual`, multiplatform, Android and framework-aware
  capability matrices before claiming support for those shapes.

## T5 — Deferred TypeScript/JavaScript advanced operations

The following work was explicitly deferred by the `0.6.x` plan and proceeds in
parallel without weakening the Kotlin release gates:

- [ ] Move file/type/module with import/export and project-reference updates.
- [ ] Organize imports and deterministic project-style-aware formatting.
- [ ] Change signature with overload/override/call-site coverage.
- [ ] Add bounded extract/inline operations where control/data-flow proof exists.
- [ ] Add TypeScript/JavaScript migration recipes without executing framework or
  package build code.

Each operation retains explicit-toolchain provenance, exact compiler diagnostics,
preview, approval, `PatchEngine`, WAL and rollback. Dynamic or mixed JavaScript
never inherits typed TypeScript certainty.

## J1 — Concurrent Java and shared-platform depth

- [ ] Continue Java change-signature, move/package/module, extract/inline and
  hierarchy catalogue expansion using JDT-backed evidence.
- [ ] Reuse shared JVM identity and Build Model improvements rather than adding
  Kotlin-specific branches to Java planners.
- [ ] Expand Spring, Jakarta/JPA, Jackson, reflection/configuration and generated
  code risk evidence where cross-language JVM operations expose those boundaries.
- [ ] Retire compatibility projections only through an explicit API migration;
  API `0.2` behavior remains supported throughout this band.

## I1 — Integrations and native acceptance

- [ ] Route Kotlin files and mixed Java/Kotlin snapshots through library, CLI,
  daemon and MCP. Preserve editor-native Kotlin LSP ownership where applicable.
- [ ] Add representative Maven, Gradle, mixed Java/Kotlin and refusal fixtures;
  no sample may require implicit network access or build-script execution.
- [ ] Add unit, property, golden, protocol, agent-simulation, hostile-workspace,
  stress, timeout and crash/restart coverage.
- [ ] Qualify at least one non-trivial Kotlin or Java/Kotlin semantic mutation on
  Linux x86-64, Windows x86-64, macOS x86-64 and macOS arm64 using the packaged
  RefactorKit runtime and explicitly qualified external toolchain inputs.
- [ ] Verify preview, exact staged diagnostics, authorization, apply, WAL,
  kill-during-write recovery, rollback and byte-identical restoration natively.
- [ ] Publish support matrix, limitations, migration notes, checksums, SPDX SBOMs,
  attestations and independent downloaded-asset verification.

## Explicit non-goals for the first `0.7.0` slice

- Claiming all Kotlin Multiplatform, Android, compiler-plugin, kapt/KSP or script
  projects as supported.
- Treating LSP rename alone as managed Kotlin authority.
- Executing Gradle or Maven project code to obtain a model.
- Freezing API `1.0` or retiring compatibility models without migration evidence.
- Completing every K5, T5 and J1 operation in one oversized release commit.

## First implementation sequence

1. Activate `0.7.0-SNAPSHOT` metadata and this plan.
2. Add the isolated `refactorkit-kotlin` module and capability descriptor with all
   unsupported semantic operations explicitly refused.
3. Implement hash-bound Kotlin toolchain discovery and bounded project-model
   projection.
4. Deliver semantic reads and exact diagnostics on a committed Kotlin/JVM sample.
5. Deliver one managed Kotlin rename end to end.
6. Add exact Java/Kotlin interoperability evidence and refusal boundaries.
7. Expand K5/T5/J1 operations in small independently accepted slices.
8. Run full native packaged acceptance before publishing any `v0.7.x` tag.

## Release gate

The first `v0.7.x` release requires a real Kotlin compiler/Analysis API-backed
mutation—not structural or lexical fallback—to pass semantic resolution, exact
normalized preview, staged diagnostics, explicit authorization, managed apply,
WAL, recovery and rollback on every supported native platform. At least one mixed
Java/Kotlin boundary must either succeed with exact evidence or refuse with a
stable typed reason. Existing Java and TypeScript capabilities must remain green.
