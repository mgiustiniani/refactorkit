# Migrating from v0.4.0 to v0.5.0

## Compatibility

The API compatibility baseline remains `0.2`. Existing documented Java CLI,
daemon, LSP, and MCP workflows remain available. `v0.5.0` adds schemas and
capabilities rather than promoting the API to stable `1.0`.

## New runtime assets

Select the asset matching the host:

- `linux-x86_64`;
- `windows-x86_64`;
- `macos-x86_64`;
- `macos-aarch64`.

Each archive contains its own Java runtime. Do not combine launchers or runtime
files from different assets. Verify the adjacent checksum before extraction.
macOS and Windows assets remain unsigned; see `docs/distribution-trust.md`.

## Capability discovery

Automation may use:

```bash
refactorkit capabilities
```

The same language-kernel schema v1 is available at:

- daemon `server.capabilities.languageKernel`;
- LSP initialize `capabilities.experimental.refactorkitLanguageKernel`;
- MCP initialize `refactorkitLanguageKernel`.

Consumers must branch on `schemaVersion`, operation stability, evidence, and
mutation authority. TypeScript/JavaScript `outline` and `identifierSearch` use
native structural evidence; they do not imply stable semantic rename authority.

## Build models

Maven discovery is offline by default and does not load Maven settings or
credentials. Anonymous Maven Central resolution remains explicit and SHA-256
verified. Gradle discovery is declarative and does not execute Wrapper, settings,
scripts, plugins, tasks, or Tooling API.

Private/authenticated Maven and executable Gradle require future provider IDs;
there is no compatibility flag that enables them in current providers.

## Source-root relocation

`java move-source-root` changes source ownership with rename-only file edits. It
preserves package declarations, imports, FQCNs, and file bytes. It does not edit
Maven or Gradle descriptors. The post-image build model must already recognize
the destination.

## External semantic providers

External process and LSP APIs are internal/experimental. Accepted workspace edits
remain proposals and cannot be written directly. Callers integrating these APIs
must retain preview, diagnostics, approval, managed apply, WAL, and rollback.

## No migration action required

No transaction journal conversion is required from `v0.4.0`. Existing journals
retain their schema and recovery behavior. As always, finish or recover pending
transactions before switching executable versions.
