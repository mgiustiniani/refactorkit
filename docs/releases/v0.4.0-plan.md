# RefactorKit v0.4.0 Hardening Release Plan

Status: active.

Current baseline: published `v0.3.0`, implementation development version
`0.4.0`, integration API baseline `0.2`.

## Objective

`v0.4.0` publishes the safety and Java-semantic hardening completed after
`v0.3.0` without prematurely claiming the final multi-language `1.0` contract.
It preserves the qualified managed-file transaction boundary, Java 8–25 JDT
analysis, structured evidence, exact staged diagnostics, and self-contained CLI
packaging while preparing the adapter platform needed by later `0.x` releases.

The final `v1.0.0` target is defined separately in
[`v1.0.0-plan.md`](v1.0.0-plan.md). API `1.0` is deliberately deferred; API
`0.2` remains active until a later pre-1.0 contract migration is explicitly
planned and tested.

## Completed workstreams

### H1 — Qualified transaction safety

Status: complete.

- TX-001 through TX-018 are closed for cooperative writers and filesystems that
  honor file force, atomic move, and directory force.
- Managed apply is lock-scoped, snapshot-aware, previewed, authorized,
  diagnostics-gated, WAL-backed, recoverable, and rollbackable.
- Journals are versioned, checksummed, quarantined on corruption, and preserve
  supported content and filesystem metadata.
- Fault, torn-write, process-kill, distinct-store, recovery, rollback-conflict,
  UTF-8/BOM, symlink, overlap, and workspace-boundary acceptance is present.
- Native client-applied LSP edits and standalone directory operations remain
  explicitly outside the managed transaction guarantee.

Normative evidence: [`../transactionality-audit.md`](../transactionality-audit.md).

### H2 — Java semantic boundary

Status: complete for the currently advertised operation matrix.

- Eclipse JDT Core 3.44/JLS25 supports configured Java source levels 8–25.
- Maven/Gradle module relationships, source roots, source levels, classpaths,
  outputs, local JARs, generated declarations, and build descriptors are bound
  into snapshot evidence.
- Analysis visibility is limited to the owning module and declared transitive
  dependencies.
- Generated Java remains analyzable but never receives stable rewrite authority.
- `JDT_BINDING` and documented `STRUCTURAL` plans may apply; explicit
  `LEXICAL_FALLBACK` plans are review-only.
- The normative operation matrix is [`../java-capabilities.md`](../java-capabilities.md).

### H3 — Compiler diagnostics and validation

Status: complete.

- JDT diagnostics expose stable syntax/type codes, severity, exact source range,
  compiler evidence, and category.
- Staged post-images are analyzed through isolated exact source overlays rather
  than stale workspace source roots.
- Managed apply compares current and staged error multisets before WAL and blocks
  unavailable providers or every unapproved regression.
- Maven, Gradle, multi-module, Spring, and JPA acceptance does not execute build
  scripts; lifecycle tests re-diagnose after apply and rollback.

## Remaining v0.4 workstreams

### H4 — Deterministic Java single-file formatting

Status: complete.

- [x] Implement Eclipse JDT compilation-unit formatting with project style discovery
  and a documented deterministic RefactorKit fallback.
- [x] Bind formatter identity, version, options, and relevant project configuration
  into planning/snapshot evidence.
- [x] Emit a normal `PatchPlan` with preview, exact staged diagnostics, approval,
  managed apply, transaction log, and rollback; never write directly from the
  formatter.
- [x] Expose CLI `format-file`, daemon/MCP preview/apply flows, and both native
  client-managed LSP `textDocument/formatting` and managed
  `refactorkit.formatFile` ownership modes.
- [x] Preserve strict UTF-8/BOM and line-ending policy, refuse generated sources,
  avoid unrelated import changes, and add idempotence/minimal-diff acceptance.
- [x] Keep range and changed-lines formatting as later additive capabilities;
  whole-file formatting is the `v0.4.0` contract.

Normative design: [`../formatting.md`](../formatting.md).

### H5 — Beta contract inventory

Status: complete for API `0.2`; `docs/api-0.2-contract.json` is verified against
daemon capability discovery.

- [x] Inventory CLI, daemon, LSP, and MCP fields, commands, methods, errors, and exit
  behavior without declaring final API `1.0` stability.
- [x] Add contract snapshots and deterministic malformed/unknown input behavior.
- [x] Correct boolean-option parsing and other discovered CLI ambiguities.
- [x] Document API `0.2` additions or deprecations and decide whether a later
  pre-1.0 release introduces API `0.3`.

### H6 — Protocol hardening

Status: complete for advertised `v0.4.0` surfaces. In-flight cancellation is not
advertised by the synchronous API `0.2` servers and is explicitly deferred to
the asynchronous `v0.5.x` semantic-process kernel.

- [x] Complete initialization, capability, refusal, malformed-input,
  workspace-switch, stale-plan, and lifecycle suites; document cancellation as
  unadvertised rather than claiming a false synchronous guarantee.
- [x] Preserve MCP least privilege and the split between client-managed LSP edits and
  RefactorKit-managed patch plans.
- [x] Classify each exposed capability as beta-contract, experimental, or internal.

### H7 — Resource bounds

Status: complete for the qualified local API `0.2` baseline. Deeper external
semantic-process and aggregate-snapshot limits remain `v0.5.x` promotion gates.

- [x] Define reference workspaces and measurable scan/index/preview/apply budgets.
- [x] Bound stdio request/frame size, result presentation, and pending plans for
  the current in-process API `0.2` surfaces; file/snapshot, cache, subprocess, and
  aggregate journal limits are mandatory before corresponding `v0.5.x` features.
- [x] Add reference stress ceilings and deterministic protocol limit behavior;
  do not advertise unsupported cooperative cancellation.

### H8 — Supply chain and packaging

Status: implementation complete; tag workflow verification remains part of H9.
Two repeated local archives produced identical SHA-256 values.

- [x] Preserve the Linux x86_64 self-contained jlink runtime and `JAVA_HOME`-unset
  smoke suite.
- [x] Add SBOM and artifact provenance/attestation.
- [x] Compare repeat package builds and document unavoidable reproducibility limits.
- [x] Verify archive layout, permissions, embedded runtime modules, checksum,
  and version metadata; SBOM records packaged dependency licenses.

### H9 — Documentation and release acceptance

Status: local release-candidate acceptance complete. Immutable-tag workflow and
post-publication downloaded-artifact verification remain the publication step.

- [x] Reconcile README, operation docs, integration contracts, ARC42, ADRs,
  changelog, packaging, and safety documentation with `v0.4.0` behavior.
- [x] Publish migration notes from `v0.3.0` and an acceptance record.
- [x] Run clean build, all golden cases, packaged smoke, contract suites, stress
  checks, checksum/SBOM/provenance verification, and downloaded-artifact smoke.

## Non-goals

`v0.4.0` does not freeze API `1.0`, claim final IDE-grade Java catalogue
completion, or add production semantic adapters for every roadmap language.
Those are staged through subsequent `0.x` milestones and the final `1.0`
acceptance program.
