= ADR 0008: Compiler-Backed Java Analysis Strategy
:toc:
:sectnums:

Date: 2026-07-10

Status: Accepted; stable semantic capability promotion is evidence-gated

== Context

ADR 0007 kept the beta Java adapter lexical/structural where conservative and visible, but required a compiler-backed follow-up before RefactorKit can claim v1.0.0-level semantic guarantees.

`v0.3.0` is the post-beta slice for publishing that analysis foundation. After `v0.3.0`, immutable `v1.0.0-rc.N` candidates freeze and validate proposed stable contracts before `v1.0.0`. The analysis strategy must improve parsing, binding-aware symbol identity, diagnostics, and reference precision without weakening the existing preview/apply/rollback safety model.

Stable does not require every Java refactoring to be semantic or supported. It requires each advertised stable operation to have a versioned contract, explicit evidence classification, tested boundaries, and conservative refusal where semantic authority is incomplete.

== Decision

Use Eclipse JDT as the primary compiler-backed source of Java parsing, project/classpath-aware bindings, symbol identity, reference search, and semantic diagnostics. Preserve Java 8-compatible RefactorKit bytecode as decided by ADR 0001, but upgrade JDT to a version that supports the advertised Java source range through Java 25.

Configure source compliance per project or source set instead of assuming one level. Make source roots, module relationships, JDK types, compiled outputs, local JARs, and explicit or generated dependency classpaths deterministic and visible in evidence and diagnostics. Do not silently execute Maven or Gradle to obtain that evidence.

A planner has clean semantic evidence only when the selected binding is exact, required references and source-visible family members are resolved, relevant source/classpath diagnostics do not invalidate the proof, and every planned semantic edit derives from that evidence. Missing, unresolved, ambiguous, generated, external, reflective, framework-string, or otherwise incomplete evidence produces a documented refusal unless the operation contract defines a separately tested non-semantic fallback.

Lexical analysis may support scanning, discovery, warnings, or a narrowly classified transformation only when tests prove that it cannot claim or perform semantic authority incorrectly. A preview must label JDT-backed, lexical, and refused outcomes. Stable operations must never hide a lexical edit behind a compiler-backed claim.

Keep OpenRewrite as an optional future recipe and transformation backend, not as the primary source of symbol identity.

Promote individual planners and operation shapes to stable only after semantic, diagnostic, compatibility, and refusal tests prove their advertised boundaries. Do not infer full Java-refactoring coverage from choosing JDT.

== Recorded Prototype Evidence

The following repository evidence was recorded for the `v0.3.0` prototype. It describes the prototype baseline, not stable acceptance. In particular, Java 21 compliance and JDT 3.37.0 do not by themselves establish the advertised Java 8 through Java 25 range.

The first `v0.3.0` P2 JDT prototype adds dependency `org.eclipse.jdt:org.eclipse.jdt.core:3.37.0` in `modules/refactorkit-java` and introduces `JdtJavaSemanticAnalyzer`. The analyzer parses Java files with Eclipse JDT at Java 21 compliance, configures source roots from the scanned `ProjectSnapshot`, and currently collects class, interface, enum, method, field, and constructor declarations with qualified names, source path and line, optional binding key, and evidence labels `JDT_BINDING`, `JDT_PARSE`, or `LEXICAL_FALLBACK`.

Member qualified names now include signatures, for example `com.acme.User#displayName()`, `com.acme.Lookup#find(java.lang.String)`, and `com.acme.Lookup#<init>()`. `JdtJavaSemanticSymbol` exposes `ownerQualifiedName`, `memberSignature`, and `sourceRange`; `JdtJavaSemanticReference` exposes `symbolSignature` and `sourceRange`. Source ranges preserve JDT's zero-based columns as `SourceRange` characters so planner edits can be generated from exact JDT declaration/reference evidence. The analyzer visits `ClassInstanceCreation` so constructor references can resolve by JDT binding key, skips declaration-name `SimpleName`s instead of line-based filtering, and emits `JdtJavaSemanticWarning` entries from JDT compilation errors with `JDT_PARSE` evidence for unresolved external type or classpath issues.

The prototype returns `JdtJavaSemanticReference` entries when JDT binding keys match collected symbols. `JdtJavaSemanticAnalyzer` also exposes resolved binding uses, including occurrences inside import declarations, and marks whether each use is in import context. To keep declaration and use identity stable for generic code, it normalizes parameterized `ITypeBinding`, generic `IMethodBinding`, and `IVariableBinding` keys to their declaration bindings. It also reports `JdtJavaSemanticOverrideRelation` entries for source-visible method override/inheritance evidence, combining JDT `IMethodBinding.overrides`/subsignature checks with scanner-visible extends/interface inheritance records as a conservative fallback. Analyzer tests cover method/field identity and references, same-simple-name import disambiguation for `com.acme.right.Service` versus `com.acme.left.Service`, interface/enum parsing, nested type/member ownership, overloaded method disambiguation, constructor identity/references, a child method overriding its base method, interface implementation evidence where `DefaultLookup#find(java.lang.String)` implements/overrides `LookupApi#find(java.lang.String)`, unresolved type warnings, and scanner-provided source roots on representative samples (`samples/java-maven-simple` expecting `com.example.UserManager` and `samples/java-gradle-simple` expecting `com.example.UserService`) with method-symbol discovery on both sample shapes.

The recorded `v0.3.0` planner integration includes an exact signed-method `renameMember` slice with override-aware propagation. For a signed selector such as `com.example.Lookup#find(java.lang.String)`, `JavaRenameMemberPlanner` resolves exactly one JDT method candidate, requires a binding key, and computes the transitive source-visible class/interface override family through JDT binding-key relations. It generates `sourceRange` edits for every scanned declaration in that family and for every binding-matched call site. Static and private methods are excluded from override relations, so static method hiding does not expand a rename family. If the family includes a declaration outside the scanned source workspace, such as `java.lang.Object#toString()`, the signed rename is refused because source-only propagation would be unsafe.

Planner tests cover a transitive `Base` -> `Mid` -> `Child` class family, an interface and its implementation, an unrelated method with the same signature remaining unchanged, static hiding exclusion, and refusal for an external family declaration. Exact overload tests also verify that `find(String)` is renamed while `find(int)` remains unchanged. The signed flow remains conservative: it refuses when JDT reports parse/classpath warnings, when the selector does not resolve to exactly one JDT method candidate, when no JDT binding key is available, or when the complete override family cannot be represented by scanned source declarations. Arbitrary unsigned `METHOD` rename remains a lexical fallback and can still rename all overloads with warnings.

The recorded `v0.3.0` field-rename slice recognizes an unambiguous `FIELD` selector such as `com.acme.Stats#count` in `JavaRenameMemberPlanner`. With clean global JDT evidence, the planner selects the exact field binding and generates edits from the field declaration and binding-matched reference `sourceRange` values. Apply-test evidence renames the declaration, `this.count`, and `stats.count` while preserving a shadowing local variable named `count` and the unrelated `com.other.Stats#count` field. If the owner already declares the requested target field, the planner refuses before attempting semantic fallback, so unclean evidence cannot bypass collision safety. Unclean or unresolved JDT evidence retains the explicit lexical member fallback. This field-specific evidence does not broaden semantic guarantees for arbitrary unsigned method selectors.

The recorded `v0.3.0` class-rename slice integrates `JavaRenameClassPlanner` with JDT evidence. When JDT analysis is clean, the planner resolves the exact target type binding and emits source-range edits for the type declaration, constructor declarations, and binding-matched imports, type references, and constructor calls, followed by a source-file rename. Binding identity leaves same-simple-name types in other packages unchanged. The planner refuses an unchanged requested name and conflicts with an existing target symbol or target file. JDT parse or classpath warnings, or failure to resolve the exact target binding, produce an explicit lexical-fallback warning. Existing reflection, framework, and generated-code warnings remain in the preview. Planner tests cover constructor rename and apply, import updates, same-simple-name disambiguation, conflict refusal, and lexical-fallback warning behavior.

The recorded `v0.3.0` move-class slice integrates `JavaMoveClassPlanner` with JDT evidence. When JDT analysis is clean, the planner resolves the exact target type binding and derives the set of files containing binding-matched references. Package/import/fully-qualified-name edits are restricted to that set, which avoids adding imports to unrelated files in the old package and preserves references to unrelated types with the same simple name. The package-declaration edit and source-file move remain `PatchPlan` operations. Direct-import ranges are treated as covered when fully qualified occurrences are processed, preventing overlapping edits, and a direct import is removed rather than rewritten when the referencing file is already in the target package. The planner refuses invalid target packages and conflicts with an existing target FQN or target file. JDT parse or classpath warnings, or unavailable exact target-binding evidence, produce an explicit lexical-file-scoping warning. Planner tests cover apply, used and unused files in the old package, target-package import removal, preservation of an unrelated same-simple-name type, invalid-package and target-conflict refusal, and lexical-fallback warning behavior.

The recorded `v0.3.0` safe-delete slice uses exact JDT type-binding references when semantic analysis reports no parse or classpath warnings and the exact target type binding is available. Binding-key matching distinguishes same-simple-name types in different packages, so references to an unrelated type do not block deletion of the selected type. If semantic warnings exist or the exact target binding is unavailable, `JavaSafeDeletePlanner` explicitly uses its existing lexical reference scan and adds a lexical-fallback warning to the preview. Forced-delete behavior and framework warnings are unchanged. Planner tests cover refusal for an imported target reference with JDT binding evidence, deletion remaining available when only an unrelated same-simple-name type is referenced, and a lexical-fallback warning when clean JDT evidence is unavailable.

The recorded `v0.3.0` annotation-type slice represents annotation declarations as `ANNOTATION` in both core `Symbol.Kind` and the JDT semantic kind. Lexical top-level indexing recognizes `@interface` declarations. The JDT analyzer visits `AnnotationTypeDeclaration` nodes and records binding identity, declaration ranges, nesting ownership, and binding-matched annotation usage references. Annotation types therefore participate in read-only symbol lookup and in the existing class-style rename, move, and safe-delete planners, as well as recipe and type-owner validation. An exact rename test renames the `com.acme.Audited` declaration, source file, and usages to `Tracked` while preserving `com.other.Audited` and its usage, demonstrating binding-based same-simple-name disambiguation.

The recorded `v0.3.0` annotation-element slice supersedes the earlier annotation-type-only limitation. `JdtJavaSemanticAnalyzer` visits `AnnotationTypeMemberDeclaration` nodes and emits each annotation element as a `METHOD` symbol with a zero-parameter signed identity such as `com.acme.Route#path()`. It stores method-binding evidence and excludes the declaration name from references. Read-only `findSymbol` and `findReferences` resolve annotation elements. When semantic analysis is clean, the existing signed `renameMember` flow updates the exact annotation-element declaration and binding-matched named usages such as `@Route(path=...)`. Implicit single-element annotation values have no element name to edit. Apply-test evidence renames `path()` to `value` while preserving an unrelated `com.other.Route#path()` declaration and usage. This evidence does not claim annotation-element change-signature support or default-value editing.

The recorded `v0.3.0` organize-imports slice integrates `JavaOrganizeImportsPlanner` with per-file clean JDT evidence. For an exact import, the planner removes it only when the imported binding has no non-import use in that file. Tests demonstrate that exact imports used by `List<Map<...>>` and a used generic static `emptyList` import remain, while unused `Set` and `singletonList` imports are removed. Wildcard imports are preserved. Files with unresolved JDT evidence explicitly retain the existing lexical behavior for sorting, deduplication, and same-package import removal. Existing preview, apply, and rollback behavior is unchanged.

Broader classpath robustness, framework and reflection references, generated code, annotation-processor output, and other planner coverage remain to be proven on representative projects.

== Consequences

* The Java adapter can prototype compiler-backed identity and reference search without changing the PatchPlan safety model.
* Documentation and previews must distinguish JDT-backed evidence, narrowly classified lexical behavior, and refusal.
* OpenRewrite remains available for future large-scale recipes or transformations, but JDT drives the first compiler-backed identity experiment.
* `v0.3.0` publishes prototype evidence and remaining gaps; immutable RC acceptance, not the prototype label, determines `v1.0.0` readiness.
* Build, classpath, annotation-processing, framework-string, reflection, and generated-code boundaries remain explicit risks until tested.

== Alternatives Considered

=== Eclipse JDT

Chosen as the primary prototype candidate because it provides mature Java parsing, bindings, and reference-oriented infrastructure suitable for IDE-like analysis on JVM projects.

=== javac APIs

Useful for compiler-faithful parsing and diagnostics, but less convenient as the first refactoring-oriented symbol/reference API. It remains a possible supplement for diagnostics or source-level checks.

=== OpenRewrite

Strong candidate for future recipe execution and transformations. Not chosen as the first symbol identity source because the prototype needs binding-aware lookup and reference precision before broad recipe execution.

=== Spoon

Useful for Java model analysis and transformation experiments, but not selected as the first candidate because JDT better matches the immediate IDE-like binding/reference goals.

=== Lexical-only

Rejected as the strategy for semantic authority. Lexical analysis remains useful for discovery, warnings, and separately classified non-semantic behavior, but semantic guarantees require clean compiler/classpath-aware evidence or refusal.

== Prototype Acceptance Criteria

The `v0.3.0` prototype is acceptable when it demonstrates, with tests or documented evidence:

* source-root and classpath discovery sufficient for representative Maven and Gradle samples;
* JDT-backed symbol identity for classes and selected members without relying on plain text matches;
* reference search that distinguishes at least same-name symbols in different packages or scopes;
* preview output that labels whether evidence is JDT-backed or lexical fallback;
* conservative warnings or refusals for unsupported overload, external override-family, framework, generated-code, or classpath cases;
* no regression of preview/apply/rollback, affected-file reporting, diagnostics, and transaction safety requirements.

== Stable Promotion Criteria

A Java capability is stable at `apiVersion=1.0` only when acceptance evidence demonstrates:

* an Eclipse JDT version and source-compliance configuration that support representative Java 8, 11, 17, 21, and 25 projects;
* deterministic, visible source-root, module, JDK, output, JAR, and explicit/generated classpath inputs;
* binding-precise behavior and refusal coverage for the operation's documented nested/member type, generic, annotation, static import, record, sealed type, pattern, overload, override, visibility, external declaration, and generated-source boundaries;
* stable compiler-backed diagnostics with severity, source range, evidence, and error category, including validation before preview, after simulation, after apply, and after rollback where required by the operation contract;
* no semantic edit when clean evidence is absent, and no lexical behavior presented as compiler-backed authority;
* bounded scan, analysis, preview, and result behavior on the release reference workspaces; and
* preservation of the versioned PatchPlan, CLI, and daemon safety contracts.

Capabilities that do not meet these criteria remain experimental, explicitly lexical within a proven narrow contract, or refused. Their absence does not block stable release unless they are advertised as stable.
