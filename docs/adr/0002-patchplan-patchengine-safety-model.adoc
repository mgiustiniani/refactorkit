= ADR 0002: Use PatchPlan and PatchEngine Preview-Apply-Rollback Safety Model
:toc:
:sectnums:

Date: 2026-07-10

Status: Accepted; stable contract implementation and evidence pending

== Context

RefactorKit exists to replace fragile text edits with deterministic, reviewable refactoring operations. Code changes may rename, move, create, delete, or modify files, so direct writes would make failures difficult to inspect and recover from.

All integration surfaces need one explicit safety model. For `apiVersion=1.0`, that model is a versioned product contract rather than an implementation convention.

== Decision

Represent each planned change as a `PatchPlan` containing a workspace edit, affected files, diagnostics, warnings, confidence, evidence classification, and snapshot information. Apply changes through `PatchEngine` only after preview, validation, and user or caller approval. Record transaction metadata sufficient to diagnose an interrupted apply and to roll back every managed write.

At `apiVersion=1.0`, edit ordering, stale-snapshot rejection, overlap rejection, workspace and symlink boundaries, atomicity by filesystem capability, partial-failure reporting, transaction-log handling, apply, and rollback behavior are normative versioned contracts. Production workspace writes must not bypass `PatchEngine`.

Before the first write, `PatchEngine` validates structural paths and overlapping edits, then simulates the ordered create, modify, rename, and delete operations. The simulation tracks file existence and type transitions so that a predictable later refusal caused by a missing or existing file does not occur after earlier edits have already been applied.

Every symbolic-link path component below the workspace root is rejected with diagnostic code `path.symbolicLink`, including links that resolve to a location inside the workspace. This conservative rule complements the existing `path.outsideWorkspace` rejection.

Where an operation contract requires diagnostics, compare diagnostics before preview, after preview simulation, after apply, and after rollback. Validation regression beyond the approved plan causes refusal rather than an unreported or best-effort write.

This decision records the intended stable safety model. It does not assert that the current implementation satisfies that model. At the audited baseline, server-managed and CLI apply flows are previewable, preflighted, compensatable batches: they write sequentially and return compensation only after a completely successful apply. They are not ACID transactions and are not crash-safe or durably atomic.

== Consequences

* CLI, daemon, LSP, and MCP clients can present shared preview semantics, but each integration must accurately classify who owns the write and recovery record.
* The engine can reject a supplied stale snapshot hash, overlapping edits within one modify operation, and unsafe workspace or symbolic-link paths before writing.
* Compensation metadata gives callers a recovery path only after a completely successful managed apply in the tested no-concurrency, no-interruption case; it is not evidence of crash recovery.
* Refactoring implementations must produce structured edits instead of mutating files directly.
* Complex operations may need additional diagnostics before they can be safely accepted.
* Stable acceptance requires property, stress, corruption, interruption, unsafe-path, concurrency, and hash-restoration evidence; the accepted architecture does not by itself prove those behaviors complete.
* Preflight validation reduces predictable partial writes but does not provide filesystem atomicity, isolation, durability, or all-or-nothing behavior after I/O or process failure.
* The stable contract must define atomicity, consistency validation, isolation, durability, compensation, rollback, crash recovery, transaction boundaries, failure models, approval, and degraded filesystem guarantees separately.
* Closure requires a versioned write-ahead journal and lifecycle, safe staging and replacement according to filesystem capability, one-writer locking, affected-file preconditions, conflict-safe rollback, complete edit rendering, stable error categories, diagnostics gates, and recovery reporting. These are required consequences of this decision, not completed work.

== Evidence

The late first v1 S1 `PatchEngine` hardening slice includes test evidence for:

* rejecting a symbolic link to a location outside the workspace without writing;
* rejecting a create followed by a delete of a missing file without leaving the earlier create applied; and
* accepting a valid modify-then-rename sequence through ordered state simulation.

The link:../transactionality-audit.md[transactionality audit] at commit `5f47ad0` limits that evidence to predictable preflight failures and successful compensation in controlled cases. It does not prove durable atomic transactions.

== Audited Limitations

The following open findings qualify the accepted direction and must not be read as implemented fixes:

* `TX-001` is implementation-closed on supported filesystems: apply renders and stages complete post-images before mutation, catches failure, performs compatible compensation, and durably reports `RECOVERY_REQUIRED`; destructive fault evidence remains a release gate.
* `TX-002` is closed: `PatchEngine` durably persists and advances the versioned write-ahead lifecycle before and through workspace mutation; integrations no longer save metadata afterward.
* `TX-003` is closed after the audited baseline: transaction IDs enforce the generated UUIDv4 grammar; normalized contained log paths reject symbolic links and non-regular records; owner-only permissions are applied where supported; and malformed records produce coded failures. Durable journal lifecycle/integrity work remains under `TX-002`, `TX-011`, and `TX-012`.
* `TX-004` is closed for versioned journal transactions: normal rollback verifies exact affected-path post-images and refuses `rollback.conflict` without writes; explicit force restores pre-images and records the destructive override; legacy records without images require manual recovery.
* `TX-005` is closed for RefactorKit-managed writes after the audited baseline: `PatchEngine` holds a non-blocking OS file lock across validation/apply/rollback, snapshot-aware apply revalidates affected paths under that lock, the hash-only overload is removed, and all first-party/test callers use the protected contract.
* `TX-006` is closed for reported supported filesystems: apply/rollback/recovery use same-directory staging, file force, required atomic replacement, delete/parent-directory force, durable directory creation, permission preservation, temp cleanup, and explicit capability refusal.
* `TX-009` is closed: modify entries are normalized per path/structural segment into one original-content coordinate space, with overlap validation across the merged set and one normalized journal edit.
* `TX-010` is closed: line and character-within-line bounds are enforced, exact line-end positions remain valid, and preflight renders complete results before journal creation or mutation.
* `TX-011` is partially closed: retained records now include schema, operation, forward/compensation edits, pre/post images, lifecycle, timestamps, and failure detail; version metadata, validation/attempt history, and post-snapshot remain; schema-v2 SHA-256 integrity checksums are implemented.
* `TX-012` is partially closed: create/update/delete persistence durably flushes files/directories and lifecycle replacement requires a same-directory atomic move; schema-v2 checksums and v1-on-update migration are implemented; quarantine, capability reporting, and fault evidence remain.
* `TX-013`: rollback restores text rather than complete filesystem metadata or directory state, and the edit model does not represent all advertised directory operations.
* `TX-014` is closed for declared source and Java classpath scope: snapshot hashes bind module/source-root metadata, extensions, ignore policy, language IDs, paths, source content, and classpath evidence. Under lock the engine independently walks and rehashes sources plus active/prospective classpath entries, local JAR directories, and generated declaration files, refusing changed evidence before journaling.
* `TX-015` is closed: a core gate compares diagnostics on current and exact staged post-image snapshots under lock before WAL, refusing new errors/provider failure. All direct callers must provide `DiagnosticsGate`; ungated apply overloads were removed before API freeze.
* `TX-016` is closed: explicit managed apply invocation is the approval event; approval-required plans refuse missing authorization, and the transaction persists approval kind, surface, actor, and timestamp. Legacy records are marked `LEGACY_UNRECORDED`.

== Residual Risks and Open Scope

* At the audited `5f47ad0` baseline, `TX-001` through `TX-006` and `TX-009` through `TX-016` were open; subsequent implementation closes `TX-001` through managed-write `TX-006` (with filesystem qualification), plus `TX-009`, `TX-010`, and `TX-016`, closes source/classpath-scope `TX-014`, and partially closes `TX-011` and `TX-012`.
* Startup recovery now handles compatible interrupted lifecycle states and blocks conflicts; kill/power-loss, torn workspace writes, journal corruption, and incomplete-compensation fault evidence remain open.
* Managed snapshot-aware writes close the normal RefactorKit-writer scan-to-mutation race and independently detect newly appeared declared-scope sources and changed Java classpath artifacts; native editor-applied LSP edits and hostile concurrent filesystem namespace replacement remain outside that evidence.
* Stable `1.0` transactionality must not be claimed until the required guarantees are either evidenced or narrowed to precise degraded guarantees.
