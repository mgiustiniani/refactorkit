= ADR 0011: Credential and Executable Build Provider Boundaries
:status: accepted
:date: 2026-07-12

== Context

The Build Model SPI has deny-by-default policy switches for anonymous network,
credential access, and explicit build-code execution. A policy allowance is a
caller ceiling, not evidence that a provider implements the capability safely.
Maven private repositories require secret custody and repository provenance;
Gradle effective models normally require executing settings/build scripts,
plugins, convention logic, and potentially arbitrary project code.

Treating `ALLOW_EXPLICIT` as permission for the current Maven or Gradle providers
to read host credentials or execute builds would invalidate the offline,
reproducible, redacted discovery boundary proven by P2B.

== Decision

. `maven-effective-v1` remains credential-free and plugin/lifecycle-free. It may
  use only anonymous, checksum-verified HTTPS Maven Central when network access is
  explicitly allowed. Maven settings, environment credentials, credential
  helpers, private repositories, mirrors, servers, proxies, and decrypted secrets
  are not loaded.
. Any future authenticated Maven provider uses a new provider identity and a
  separately reviewed capability. Requests carry opaque process-local credential
  handles, never secret values. Handles are scoped to an allowlisted HTTPS
  repository origin, expire, are non-serializable, and never enter snapshots,
  diagnostics, logs, protocol responses, WAL, transaction metadata, or hashes.
. Authenticated transports require redirect refusal, origin pinning, bounded
  time/size, checksum/signature policy, cache provenance, redaction tests, and
  explicit user approval. Host Maven settings are not an implicit credential
  source.
. `gradle-declarative-v1` never executes Gradle even when the generic request
  policy allows explicit build code. It returns bounded `PARTIAL` metadata or
  `EXECUTION_REFUSED`.
. Any future executable Gradle provider uses a new provider identity and a
  separate local worker process. It requires an explicit request, declared Gradle
  distribution/wrapper provenance, clean bounded environment, workspace/read
  scopes, network and credential policy, timeout/cancellation, stdout/stderr
  limits, process-tree termination, output schema validation, and audit evidence.
. An executable provider's result is still an untrusted metadata proposal. It
  grants no write authority and cannot bypass preview, diagnostics,
  `PatchEngine`, approval, WAL, or rollback.
. Generic `BuildModelDiscoveryPolicy` allowances are ceilings. Unsupported
  allowances are ignored/refused by a provider and never upgrade its capability.

== Consequences

=== Positive

* P2B can complete without silently introducing host-secret access or arbitrary
  build execution.
* Existing provider identities keep stable, testable security semantics.
* Future privileged providers have explicit identities, provenance, and review
  boundaries instead of hidden mode switches.
* Snapshots and protocol responses remain secret-free.

=== Costs and limitations

* Private/authenticated Maven repositories remain unsupported.
* Gradle models that require executable semantics remain partial or refused.
* Some valid enterprise projects need explicit future provider work.
* Opaque credential custody and worker sandboxing require platform-specific
  acceptance before support can be claimed.

== Acceptance guardrails

* Tests request credential/build execution allowances and prove current providers
  still report `credentialsAccess=denied` and `buildCodeExecution=denied`.
* No secret-shaped value appears in Build Models, diagnostics, daemon/MCP output,
  journals, or transaction logs.
* No current provider invokes Maven/Gradle lifecycle code, wrappers, scripts,
  tasks, plugins, Tooling API, credential helpers, or Maven settings.
* Future provider support is not considered complete until native process-kill,
  timeout, redaction, provenance, and hostile-workspace acceptance passes.
