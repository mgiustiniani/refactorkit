= ADR 0005: Package the CLI with a Self-Contained jlink Runtime
:toc:
:sectnums:

Date: 2026-07-10

Status: Accepted; stable release evidence pending

== Context

RefactorKit should be easy to run from terminals, CI jobs, editor integrations, and local agent workflows. Requiring users to install or select a compatible global Java runtime increases setup friction and support risk.

The implementation is JVM-based, so a runtime must still be present at execution time.

== Decision

Provide build tasks that assemble a self-contained CLI distribution with an embedded Java runtime generated by `jlink`. The packaged launcher resolves Java from the distribution directory rather than from the system `PATH` or `JAVA_HOME`. The default runtime module contract includes `java.base`, `java.compiler`, `java.logging`, `java.xml`, and `jdk.unsupported`; `java.compiler` is required by JDT-backed signed selectors using `javax.lang.model.SourceVersion`.

For stable releases, the runtime archive is a versioned product contract. Publish its checksum, SBOM, and provenance or attestation, and document verification commands. Compare clean package builds, remove avoidable nondeterminism, and document any remaining reproducibility limits. Verify archive layout, executable permissions, embedded runtime, license files, version metadata, and an execution smoke test with `JAVA_HOME` unset before promoting an immutable release candidate to `v1.0.0`.

== Consequences

* Users can run the packaged CLI without installing Java globally.
* Distribution artifacts are larger than plain application archives.
* jlink packaging is OS/architecture-specific, so release automation builds on native Linux x86_64, Windows x86_64, macOS Intel, and macOS Apple Silicon runners. Cross-platform launcher success alone is insufficient: managed transaction/durability semantics must pass natively.
* A JDK remains required at build time to create the runtime image.
* Runtime module selection must be maintained as dependencies evolve; packaged-runtime smoke tests must check required modules and execute JDT-backed behavior rather than only `--help` or scans.
* Release automation must classify prerelease and stable artifacts from immutable tags and must not move or recreate a failed tag.
* Self-containment, checksum, SBOM, provenance, and reproducibility claims require evidence from the published artifact; successful assembly alone is insufficient.

== Implementation Evidence

The immutable `v0.3.0` tag points to commit `ba114134be605b7b0076e7471f8111c081a8aaa3`. Its GitHub release at https://github.com/mgiustiniani/refactorkit/releases/tag/v0.3.0 is published as the latest, non-prerelease release with a self-contained runtime ZIP and its `.sha256` asset. The downloaded archive verified as SHA-256 `0cf054a7eb99a86734f663ca0966da2b47259b8a4658bca97f9044bc2d21ff10`, passed version, help, and sample-project scan checks with `JAVA_HOME` unset, and was deployed locally.

A later signed-selector smoke found that the published `v0.3.0` image omitted `java.compiler`, causing `NoClassDefFoundError: javax/lang/model/SourceVersion`. The development packaging contract now requires that module and runs signed `definition`/`references` against the packaged runtime with source-integrity checks in CI and release workflows.

The published `v0.4.0` Linux x86_64 artifact corrected `java.compiler`, passed signed-selector and managed formatter checks, added a checksum and SPDX SBOM, published build/SBOM attestations, was independently downloaded and verified, and was deployed locally. `v0.5.0` expands the native matrix to Windows x86_64, macOS Intel, and macOS Apple Silicon. Each new platform remains acceptance-in-progress until its filesystem, apply, WAL, recovery, rollback, archive, SBOM and attestation evidence passes; README support labels follow evidence rather than workflow presence.
