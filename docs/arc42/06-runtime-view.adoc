== Runtime Scenario: Safe Java Refactoring through CLI

[cols="1,3",options="header"]
|===
|Step |Interaction

|1
|Developer runs a CLI command such as `rename`, `move-class`, `organize-imports`, `safe-delete`, `extract-method`, or `change-signature`.

|2
|CLI scans or loads the target workspace and builds a `ProjectSnapshot`.

|3
|CLI delegates Java-specific analysis to `JavaLanguageAdapter` / planner classes.

|4
|Planner returns a `PatchPlan` with affected files, workspace edits, diagnostics, warnings, confidence, risk level, and an explicit JDT-backed, lexical, or refusal evidence classification.

|5
|User reviews the preview. If applying, `PatchEngine` acquires the workspace writer lock, checks the derived current snapshot hash, revalidates initially affected paths, and preflights path, symbolic-link, overlap, and ordered file-state rules before sequential direct writes.

|6
|Only after every write succeeds does the current flow return compensating rollback edits; the CLI then persists transaction metadata. A write or process failure can therefore leave partial state without a durable recovery record.

|7
|Rollback replays compensation without validating post-apply file state and can overwrite later edits. Central diagnostics gates, conflict-safe rollback, workspace locking, durable journaling, and crash recovery are planned requirements, not current behavior.
|===

Status: implementation-informed from CLI, Java adapter, patch engine, testkit docs, and `docs/transactionality-audit.md`. This is a preflighted compensatable batch, not an ACID or crash-safe transaction.

== Runtime Scenario: AI-assisted Refactoring through MCP

[cols="1,3",options="header"]
|===
|Step |Interaction

|1
|Local LLM agent calls `project_scan` for a workspace root.

|2
|Agent asks for `project_summary`, symbol resources, definitions, references, diagnostics, or `generate_context_bundle`.

|3
|Agent calls `preview_refactoring` or `import_external_java_class` instead of editing files directly.

|4
|MCP server delegates to core, Java adapter, and/or web importer to produce a preview.

|5
|Agent/user inspects affected files, warnings, and diagnostics.

|6
|Agent explicitly calls `apply_refactoring`; the current engine treats this invocation as sufficient even though `requiresUserApproval` is advisory metadata rather than a separately enforced approval state.

|7
|After a completely successful apply, compensation is available through `rollback_refactoring`. The transaction log is saved after mutation, and rollback is not conflict-safe against later workspace edits.
|===

Status: implementation-informed from `docs/ai-workflow.md`, `docs/integration-contracts.md`, and the transactionality audit; stable MCP status remains capability-by-capability and acceptance-gated. MCP managed apply is a compensatable batch, not a durable transaction.

== Runtime Scenario: Editor Operation through LSP

[cols="1,3",options="header"]
|===
|Step |Interaction

|1
|External editor starts `refactorkit-lsp` over stdio.

|2
|Editor sends LSP requests such as definition, references, prepareRename, rename, codeAction, documentSymbol, semanticTokens, or diagnostics.

|3
|LSP server maps requests to Java/core model operations.

|4
|For native rename and code actions, the server returns a standard LSP `WorkspaceEdit`; the editor applies it outside `PatchEngine`. This is a client-managed edit with no RefactorKit transaction or rollback record.

|5
|Current planning can use disk snapshots rather than unsaved buffer content and emits no document version, creating stale-buffer application risk.

|6
|For the custom `refactorkit.applyPlan` command, RefactorKit owns sequential writes and post-apply compensation logging, with the same non-crash-safe limitations as other managed apply paths.
|===

Status: implementation-informed from `docs/integration-contracts.md` and `docs/transactionality-audit.md`; stable LSP status remains capability-by-capability and acceptance-gated. Native edits and RefactorKit-managed apply have distinct transaction boundaries.

== Runtime Scenario: Daemon Capability Discovery Before Project Open

[cols="1,3",options="header"]
|===
|Step |Interaction

|1
|A daemon client starts the local daemon over stdio and sends JSON-RPC method `server.capabilities` before `project.open`.

|2
|The daemon returns RefactorKit name, version, API version, protocol `json-rpc-2.0`, and transport `stdio` without creating a project session or accessing a workspace. The stable target value is API version `1.0`; the current baseline remains pre-stable until contract acceptance.

|3
|The response lists documented daemon methods with stability, `requiresProject`, and `writesWorkspace` descriptors. `java.importExternalClass` is explicitly classified as experimental.

|4
|The response also advertises preview-before-apply, snapshot validation, transaction rollback, and workspace-scoped-write safety flags.

|5
|The client uses the metadata to select compatible calls and still opens a project before invoking methods whose descriptor requires one. Discovery itself grants no filesystem access.
|===

Status: implementation-informed from `DaemonSession`, its daemon contract test, `docs/integration-contracts.md`, and the transactionality audit. Current apply/rollback rescans used for validation are not assigned back to the daemon session, so later queries can observe stale metadata (`TX-018`).

== Runtime Scenario: Recipe Apply Saga

[cols="1,3",options="header"]
|===
|Step |Interaction

|1
|The recipe engine plans and applies each step separately rather than staging one recipe-wide workspace state.

|2
|After each successful step it saves that step's transaction metadata; a later step sees a workflow that may already have changed the workspace.

|3
|If a later logical step fails, the engine attempts earlier compensations in reverse order.

|4
|A crash, log failure, I/O failure, concurrent edit, or compensation conflict can leave a partially completed recipe. Multiple same-file edits may also retain original-snapshot coordinates across sequential modifications.
|===

Status: implementation-informed from `docs/transactionality-audit.md`. Recipe execution is a saga with best-effort compensation, not one atomic transaction.

== Runtime Scenario: External Java Class Import Preview

[cols="1,3",options="header"]
|===
|Step |Interaction

|1
|User or agent provides Java source from clipboard, file, URL, snippet, or LLM response.

|2
|Importer strips Markdown fences and records provenance/license warnings.

|3
|Importer validates target package, detects top-level public types, splits multiple public types when needed, rewrites package declaration, and organizes imports.

|4
|Importer checks naming conflicts in the scanned snapshot or filesystem.

|5
|Importer returns a preview-only `PatchPlan`; apply is delegated to the patch engine after user approval.
|===

Status: implementation-informed from `docs/external-class-importer.md`.
