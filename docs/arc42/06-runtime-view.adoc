== Runtime Scenario: Safe Java Refactoring through CLI

[cols="1,3",options="header"]
|===
|Step |Interaction

|1
|Developer runs a CLI command such as `rename`, `move-class`, `organize-imports`, `safe-delete`, `extract-method`, or `change-signature`.

|2
|CLI scans or loads the target workspace and builds a `ProjectSnapshot`.

|3
|CLI delegates Java-specific analysis to `JavaLanguageAdapter` / planner classes.

|4
|Planner returns a `PatchPlan` with affected files, workspace edits, diagnostics, warnings, confidence, risk level, and an explicit JDT-backed, lexical, or refusal evidence classification.

|5
|User reviews the preview. If applying, `PatchEngine` validates the current snapshot hash, checks edit safety, applies the workspace edit, and creates rollback metadata.

|6
|Configured stable validation compares diagnostics before preview, after simulation, and after apply; an unapproved regression refuses apply. Rollback uses the recorded `Transaction` and, where required, verifies post-rollback diagnostics and restoration.
|===

Status: implementation-informed from CLI, Java adapter, patch engine, and testkit docs.

== Runtime Scenario: AI-assisted Refactoring through MCP

[cols="1,3",options="header"]
|===
|Step |Interaction

|1
|Local LLM agent calls `project_scan` for a workspace root.

|2
|Agent asks for `project_summary`, symbol resources, definitions, references, diagnostics, or `generate_context_bundle`.

|3
|Agent calls `preview_refactoring` or `import_external_java_class` instead of editing files directly.

|4
|MCP server delegates to core, Java adapter, and/or web importer to produce a preview.

|5
|Agent/user inspects affected files, warnings, and diagnostics.

|6
|Agent calls `apply_refactoring` only after approval; rollback remains available through `rollback_refactoring`.
|===

Status: implementation-informed from `docs/ai-workflow.md` and `docs/integration-contracts.md`; stable MCP status remains capability-by-capability and acceptance-gated.

== Runtime Scenario: Editor Operation through LSP

[cols="1,3",options="header"]
|===
|Step |Interaction

|1
|External editor starts `refactorkit-lsp` over stdio.

|2
|Editor sends LSP requests such as definition, references, prepareRename, rename, codeAction, documentSymbol, semanticTokens, or diagnostics.

|3
|LSP server maps requests to Java/core model operations.

|4
|For workspace edits, the server returns standard LSP edit structures, including `changes` and `documentChanges` where applicable.

|5
|For advanced patch commands, editor can use advertised `workspace/executeCommand` hooks.
|===

Status: implementation-informed from `docs/integration-contracts.md`; stable LSP status remains capability-by-capability and acceptance-gated.

== Runtime Scenario: Daemon Capability Discovery Before Project Open

[cols="1,3",options="header"]
|===
|Step |Interaction

|1
|A daemon client starts the local daemon over stdio and sends JSON-RPC method `server.capabilities` before `project.open`.

|2
|The daemon returns RefactorKit name, version, API version, protocol `json-rpc-2.0`, and transport `stdio` without creating a project session or accessing a workspace. The stable target value is API version `1.0`; the current baseline remains pre-stable until contract acceptance.

|3
|The response lists documented daemon methods with stability, `requiresProject`, and `writesWorkspace` descriptors. `java.importExternalClass` is explicitly classified as experimental.

|4
|The response also advertises preview-before-apply, snapshot validation, transaction rollback, and workspace-scoped-write safety flags.

|5
|The client uses the metadata to select compatible calls and still opens a project before invoking methods whose descriptor requires one. Discovery itself grants no filesystem access.
|===

Status: implementation-informed from `DaemonSession`, its daemon contract test, and `docs/integration-contracts.md`.

== Runtime Scenario: External Java Class Import Preview

[cols="1,3",options="header"]
|===
|Step |Interaction

|1
|User or agent provides Java source from clipboard, file, URL, snippet, or LLM response.

|2
|Importer strips Markdown fences and records provenance/license warnings.

|3
|Importer validates target package, detects top-level public types, splits multiple public types when needed, rewrites package declaration, and organizes imports.

|4
|Importer checks naming conflicts in the scanned snapshot or filesystem.

|5
|Importer returns a preview-only `PatchPlan`; apply is delegated to the patch engine after user approval.
|===

Status: implementation-informed from `docs/external-class-importer.md`.
