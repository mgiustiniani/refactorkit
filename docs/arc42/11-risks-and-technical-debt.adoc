= 11. Risks and Technical Debt
:toc:
:sectnums:

Status: implementation-informed, needs human review for prioritization.

[cols="1,3,2,2,1",options="header"]
|===
|ID |Risk / debt |Impact |Mitigation / next action |Status

|R-001
|Compiler-backed Java analysis is implemented only for selected JDT-backed slices; other operations and explicit fallback paths still rely on lexical or structural evidence.
|Refactorings outside the supported signed-method and unambiguous-field `renameMember` slices (including exact annotation-element rename), type-level rename (including annotation types), move-class file-scoping, safe-delete type-reference, annotation-type/element indexing and reference, and exact-import use slices may miss reflection, generated code, annotation processors, external callers, and some framework/string references. Arbitrary unsigned methods retain lexical behavior. Type-level rename, move class, and safe delete can regain lexical false positives when JDT analysis is not clean or the exact target binding is unavailable; unresolved organize-imports files retain lexical cleanup only.
|Follow ADR 0008; classify every stable path as JDT-backed, narrowly proven lexical behavior, or refusal. Do not promote prototype lexical fallbacks that can perform an incorrect semantic edit; retain external-family and collision refusals.
|known risk

|R-002
|`organize-imports` removes unused exact imports only when per-file JDT evidence is clean; wildcard imports are preserved and unresolved files retain lexical sorting, deduplication, and same-package removal.
|Users may expect IDE-equivalent wildcard analysis and unused-import cleanup even with incomplete classpaths or unresolved source.
|Keep wildcard and unresolved-file behavior explicit and conservative; broaden classpath robustness and wildcard-use analysis before claiming complete IDE-equivalent cleanup.
|known limitation

|R-003
|`extract-method` MVP only supports no-argument `private void` extraction and refuses many cases.
|Limited usefulness for real-world extraction scenarios.
|Keep refusals explicit; expand only with robust variable/control-flow analysis.
|known limitation

|R-004
|Change-signature is conservative and refuses hierarchy/public/protected/generated/method-reference cases.
|Limited support for public API evolution.
|Add compiler-backed hierarchy/call-site analysis or retain the documented limited shape and conservative refusal at stable.
|known limitation

|R-005
|Safe delete uses exact JDT bindings only for clean source analysis and does not inspect build files, generated code, reflection, external config, or all framework references.
|Possible unsafe delete if non-source references exist; lexical fallback may also conflate same-simple-name types when semantic evidence is unavailable.
|Maintain framework and forced-delete warnings; for stable classification, prove any narrow lexical behavior safe or refuse when exact evidence is unavailable. Add build/config/framework analysis before lowering risk.
|known risk

|R-006
|External importer license detection is heuristic.
|Unknown or incompatible code could be imported if user ignores warnings.
|Keep preview-only unknown-license behavior; consider policy enforcement and SPDX scanning.
|known risk

|R-007
|External LSP-backed multi-language support is currently stub/future work.
|Users may overestimate non-Java semantic refactoring capability.
|Keep docs explicit; expose only structural/local operations as MVP.
|implementation-informed

|R-008
|ADR status and stable capability classification can drift from contracts and implementation evidence.
|Reviewers or clients may mistake implemented prototype behavior for an accepted stable guarantee.
|Keep chapter 9 synchronized with ADR bodies and require contract snapshots plus RC acceptance before stable classification.
|known risk

|R-009
|The stable plan freezes core patch, CLI, and daemon API `1.0` contracts, but the in-process library export boundary is not yet inventoried.
|Embedding applications may depend on internals not covered by the stable integration contract.
|Publish an explicit stable library surface or classify in-process APIs as non-stable before RC acceptance.
|planned release review

|R-010
|The confirmed stable artifact is Linux x86_64; additional OS and artifact-repository targets remain undefined.
|Consumers may assume portability beyond the verified self-contained archive.
|Publish the exact artifact matrix and verification commands; classify other targets as unsupported or experimental.
|known limitation

|R-011
|Security threat model is incomplete.
|Trust boundaries around MCP clients, external code sources, external language servers, and filesystem edits need review.
|Create threat model and security ADR/quality scenarios.
|needs-review

|R-012
|Runtime/dynamic and C3 diagrams are missing.
|Architecture documentation may be less useful for maintainers working on internal interactions.
|Add scenario-driven dynamic views and C3 component views when needed.
|actionable

|R-013
|Class rename uses exact JDT bindings only when source analysis is clean and the exact target binding is available; reflection, generated code, framework strings/configuration, and unknown external callers remain outside binding-matched source edits.
|A rename may require manual follow-up outside analyzed source, while lexical fallback may conflate same-simple-name types.
|Retain reflection/framework/generated-code warnings, conflict refusals, and same-simple-name binding tests; stable classification must prove any narrow fallback safe or refuse without clean binding evidence.
|known risk

|R-014
|Move class uses exact JDT bindings only to select referencing files when source analysis is clean; edits inside selected files remain lexical, and reflection, generated code, framework strings/configuration, annotation-processor output, and unknown external callers remain outside this evidence.
|A move may require manual follow-up outside analyzed source. Parse/classpath warnings or unavailable target binding trigger broader lexical file scoping, which can reintroduce unrelated old-package edits or same-simple-name confusion.
|Retain non-source warnings, target conflict/invalid-package refusals, overlap prevention, and binding-scoping tests; stable classification must prove lexical file scoping safe for its exact contract or refuse without clean evidence.
|known risk

|R-015
|Daemon capability metadata is maintained alongside the request dispatcher and can drift from implemented method behavior.
|Clients could make incorrect compatibility, project-lifecycle, or workspace-write assumptions from stale descriptors or safety flags.
|Keep the capability contract test aligned with every documented daemon method; prefer deriving dispatch and discovery from one method registry if the surface grows.
|known risk

|R-016
|Annotation-element support is limited to JDT-backed read-only lookup/references and clean exact signed rename of the zero-parameter element declaration and named usages. Implicit single-element values have no name to edit; annotation-element change-signature and default-value editing are not implemented.
|Users could infer broader annotation-member or annotation-value editing from successful signed element rename.
|Keep the exact supported boundary explicit; retain binding-based same-name disambiguation tests and add dedicated semantic evidence before advertising change-signature or default-value editing.
|known limitation

|R-017
|Exact field rename depends on an unambiguous `Owner#field` selector and clean global JDT evidence; unclean or unresolved prototype analysis retains lexical member fallback, and non-source references remain outside binding ranges.
|Fallback may rename shadowing or unrelated same-named identifiers, and callers could overgeneralize field evidence to arbitrary unsigned methods.
|For stable classification, prove a narrow lexical contract cannot perform an incorrect semantic edit or replace it with refusal; preserve collision and binding-disambiguation tests.
|known risk

|R-018
|The current JDT baseline and fixed Java 21 compliance do not prove the advertised Java 8 through Java 25 source range.
|Stable parsing, bindings, diagnostics, or refactorings may fail or misclassify newer or older constructs.
|Upgrade JDT, configure compliance per project/source set, and pass representative Java 8, 11, 17, 21, and 25 acceptance suites.
|planned release blocker

|R-019
|Stable diagnostics and before/simulation/after/rollback validation gates are not yet evidenced as one versioned contract.
|An apply could introduce unapproved regressions or clients could interpret diagnostics inconsistently.
|Freeze diagnostic fields/categories and enforce operation-specific regression gates before RC promotion.
|planned release blocker

|R-020
|No accepted numeric performance/resource budgets are recorded.
|Large workspaces or hostile protocol inputs could cause unbounded latency, memory, pending plans, caches, results, or transaction metadata.
|Define reference workspaces and limits; add repeatable stress, cancellation, timeout, and deterministic limit-error evidence.
|planned release blocker

|R-021
|The published `v0.3.0` asset has checksum and downloaded-artifact smoke evidence, but SBOM, provenance/attestation, clean-build reproducibility, and promoted stable-RC evidence are not yet recorded.
|The `v0.3.0` package is independently checksum-verifiable and locally runnable, but a later stable package could still be non-reproducible or inconsistent with release metadata.
|Verify two clean builds and the downloaded immutable-RC asset, documenting unavoidable reproducibility limits before stable publication.
|planned stable-release blocker; `v0.3.0` checksum/smoke path implementation-informed

|R-022
|LSP and MCP surfaces are implemented but not yet completely classified as stable or experimental.
|Clients may depend on untested protocol behavior, and MCP privilege or cancellation boundaries could be overstated.
|Inventory every capability, command, tool, resource, and prompt and pass stable protocol boundary suites.
|planned release blocker

|R-023
|PatchEngine preflight rejects selected structural/path/overlap errors, symbolic-link traversal, and predictable invalid ordered file-state transitions. Managed writes hold a workspace lock, revalidate affected paths, durably journal pre/post intent, stage all content, require atomic replacement, and flush files/directories (`TX-001`, `TX-002`, `TX-005`, supported-filesystem `TX-006` closed; `TX-014` narrowed).
|Unsupported root capability is refused and per-target atomic-move failure is recoverable, but mounted-store variation and destructive kill/disk-full/power-loss evidence remain; caller-provided scope and editor-managed LSP writes remain outside complete evidence.
|Make scan scope engine-owned; add journal checksum/quarantine and fault/kill/restart/mounted-filesystem acceptance evidence.
|planned release blocker

|R-024
|Transaction-log identifier/path hardening is implemented (`TX-003` closed), while schema, integrity, atomic durable persistence, quarantine, and retention remain open (`TX-011`, `TX-012`).
|Validated UUIDv4 IDs, normalized containment, symbolic-link/non-regular-file refusal, owner-only permissions where supported, and coded corrupt-record failures prevent the audited direct traversal path; truncated or incomplete records can still block recovery and audit.
|Add atomic durable journal persistence, version schemas/lifecycle, verify integrity, quarantine corruption, and retain appropriate history.
|remaining journal integrity and release blocker

|R-025
|Versioned normal rollback now verifies exact post-images and explicit force is audited (`TX-004` closed), but restoration remains content/POSIX-permission focused and complete scan scope is caller-provided (`TX-013`, `TX-014`).
|Legacy image-less records require manual recovery; ownership, timestamps, ACLs, extended attributes, encoding metadata, and directory state may not be restored; omitted newly appeared sources can weaken library freshness scope.
|Define full metadata/directory restoration and engine-owned scan scope; keep forced/manual recovery separate and explicit.
|critical data-loss risk; planned release blocker

|R-026
|Native LSP rename/code-action edits are client-managed outside `PatchEngine`, can be planned from stale disk content for unsaved buffers, and carry no document version (`TX-007`).
|Editors can apply stale edits without RefactorKit-owned transaction or rollback, while clients may infer guarantees from server-managed apply documentation.
|Classify native edits explicitly, require document-version checks or a client-apply acknowledgement/journal design, and never advertise RefactorKit rollback for editor-owned writes without evidence.
|planned release blocker

|R-027
|Recipe execution remains a per-step durable saga (`TX-008`); same-file coordinate merging, global overlap validation, bounds checks, and complete preflight rendering are implemented (`TX-009`, `TX-010` closed).
|A later logical failure can still require multi-record reverse compensation rather than one recipe-wide commit.
|Choose and fully specify either the retained durable saga boundary or one staged recipe-wide commit, with crash/cancellation acceptance evidence.
|planned release blocker

|R-028
|Diagnostics, approval, error mapping, and daemon state refresh are not transactionally enforced (`TX-015`, `TX-016`, `TX-017`, `TX-018`).
|Apply can commit unapproved diagnostic regressions; approval lacks an auditable meaning; clients cannot distinguish refusal/recovery causes; daemon queries can observe stale post-write metadata.
|Define authoritative diagnostic gates and approval semantics, publish stable error categories, compensate or report diagnostic failures deterministically, and refresh session state after apply/rollback.
|planned release blocker
|===

== Technical Debt from Release Plan

The active stable plan defines the remaining release-hardening workstreams:

* `v0.3.0` publication is complete from immutable tag `v0.3.0`; main now advances toward `1.0.0-rc.1-SNAPSHOT`;
* keep API `0.2` active until normative patch/apply/rollback and API `1.0` CLI/daemon contracts are frozen and accepted;
* prove Eclipse JDT source compliance and semantic boundaries for Java 8 through Java 25;
* close or explicitly narrow `TX-001` through `TX-010` before stable transactionality claims, and resolve `TX-011` through `TX-018` contract/recovery gaps;
* define managed apply as a durable transaction only after evidence; until then retain the accurate compensatable-batch, client-managed LSP edit, and recipe-saga classifications;
* complete diagnostic regression gates and representative project acceptance;
* classify and test stable LSP/MCP boundaries;
* establish bounded performance and resource behavior;
* verify self-contained assets with checksum, SBOM, provenance/attestation, and reproducibility evidence;
* promote only an RC with no unresolved release blocker.

Status: confirmed target from `docs/releases/v1.0.0-plan.md`; `v0.3.0` publication and downloaded-runtime smoke evidence are implementation-informed, while RC/stable acceptance evidence remains pending.
