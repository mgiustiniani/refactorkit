== Level 1: RefactorKit System

RefactorKit provides deterministic code-intelligence and refactoring capabilities with preview, validation, managed apply, and compensation semantics. Managed apply is a WAL-backed, recoverable, staged per-path transaction rather than one filesystem-wide ACID commit; native LSP edits are version-checked and explicitly client-managed/no-rollback, and recipes reduce staged steps to one managed transaction. RefactorKit exposes capabilities through CLI, daemon, LSP, MCP, and library integration modes. For `v1.0.0`, the core patch, documented CLI, and documented daemon surfaces become stable API `1.0` contracts only after acceptance; LSP and MCP surfaces are stable only capability by capability where explicitly classified.

|Kotlin/JVM library
|Java scan/symbol/reference/diagnostic/refactoring capability: offline plugin-free Maven effective-reactor modeling with separate main/test JDT environments and hash-bound parent/BOM/artifact evidence; lexical and JDT indexing of classes, interfaces, enums, records, and annotation types; JDT binding-backed class-style rename for types (including annotation types), signed method rename, and unambiguous `Owner#field` field rename; JDT binding-scoped move class; structural byte-identical whole-source-root relocation; per-file JDT binding-aware exact-import cleanup; JDT binding-backed safe delete; extract method MVP; change signature MVP; recipes; framework-aware risk detection; and explicit lexical fallback where defined. The semantic analyzer visits annotation-type declarations and records their binding identity, source ranges, nesting owner, and usage references. It visits annotation elements as zero-parameter signed `METHOD` symbols, records method-binding evidence, and excludes declaration names from reference results. Read-only lookup/reference queries and clean signed member rename therefore resolve exact annotation elements and binding-matched named usages. For a uniquely selected `FIELD` and clean global evidence, `JavaRenameMemberPlanner` uses exact field-binding declaration/reference ranges; owner-field collision validation occurs before semantic fallback. Unclean or unresolved evidence uses the explicit lexical member fallback, and arbitrary unsigned methods have no equivalent semantic guarantee. The analyzer also exposes import-context and non-import binding uses normalized to declaration bindings for generic type, method, and variable identities. Annotation-element change-signature and default-value editing are not implemented.
|`modules/refactorkit-java`, Java docs, ADR 0008.

== Key Dependencies

* CLI depends on core, Java and Kotlin adapters, web importer, tree-sitter foundation, and testkit.
* Daemon and LSP depend on core plus Java and Kotlin adapters.
* MCP depends on core, Java and Kotlin adapters, and web importer.
* Java adapter, Kotlin adapter, web importer, tree-sitter foundation, and testkit depend on core.
* Kotlin compiler classes are compile-only for the isolated bridge and are supplied explicitly only to the external worker; they are not loaded into the packaged RefactorKit process.
* Web importer reuses Java source-root/package/import knowledge.
