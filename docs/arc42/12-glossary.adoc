= 12. Glossary
:toc:
:sectnums:

Status: implementation-informed. Terms are sourced from repository docs and source names.

[cols="1,3,2,2",options="header"]
|===
|Term |Definition |Scope |Evidence

|RefactorKit
|Deterministic refactoring and code-intelligence engine for safe, previewable, rollbackable code transformations.
|System
|README, AGENTS

|PatchPlan
|Structured preview/apply plan for a code modification, including operation, snapshot hash, affected files, workspace edit, diagnostics, warnings, risk, and confidence.
|Core
|`Models.kt`

|WorkspaceEdit
|Collection of file edits to modify, create, delete, or rename files in a workspace.
|Core
|`Models.kt`

|TextEdit
|A replacement over a source range.
|Core
|`Models.kt`, `TextEdits.kt`

|PatchEngine
|Core service that locks, preflights, journals, stages, durably commits per-path images, recovers interrupted managed writes, and performs conflict-safe or explicitly forced rollback. Multi-path batches are recoverable rather than one filesystem-wide ACID commit.
|Core
|`PatchEngine.kt`, ADR 0002, transactionality audit

|Transaction
|Domain identity and compensation view embedded in the durable journal record. Journal pre/post images and lifecycle—not the `Transaction` object alone—provide recovery and conflict-safe rollback evidence.
|Core
|`Models.kt`, `docs/transactionality-audit.md`

|Compensatable Batch
|Legacy classification for previewed/preflighted sequential writes with post-success compensation only. Current managed paths have advanced to locked, write-ahead-journaled, durably staged recoverable batches.
|Core / integrations
|ADRs 0002 and 0004, transactionality audit

|Transaction Log
|Versioned write-ahead recovery metadata stored under `.refactorkit/transactions`, durably created before mutation and atomically advanced through apply/rollback/recovery states. Schema-v2 SHA-256 integrity, v1-on-update migration, and blocking owner-only corruption quarantine are implemented; full metadata and complete fault evidence remain planned.
|Core / workspace
|`TransactionLog`, transactionality audit

|Saga
|Multi-step workflow that commits steps separately and attempts later compensation. RefactorKit recipes no longer use this boundary; they stage one recipe-wide transaction.
|Recipes
|ADR 0004, transactionality audit `TX-008`

|ProjectSnapshot
|Immutable snapshot of workspace modules and source files with a content hash.
|Core
|`Models.kt`

|LanguageAdapter
|Contract implemented by language-specific adapters for parse, symbols, references, diagnostics, and refactoring application.
|Core / adapters
|`Models.kt`

|Java Language Adapter
|Java-specific implementation providing source scanning, symbols, references, diagnostics, and refactoring planners.
|Java module
|`JavaLanguageAdapter.kt`

|Symbol
|Named code element such as class, interface, enum, record, annotation type, method, field, constructor, package, or unknown.
|Core / Java
|`Models.kt`

|Annotation Type
|Java `@interface` declaration represented as an `ANNOTATION` symbol. RefactorKit indexes its declaration, nesting owner, binding identity, and usages for type-level lookup and existing class-style operations.
|Core / Java
|`Models.kt`, `JavaLanguageAdapter.kt`, `JdtJavaSemanticAnalyzer.kt`, ADR 0008

|Annotation Element
|Zero-parameter member declared inside a Java annotation type, represented by JDT analysis as a signed `METHOD` symbol such as `com.acme.Route#path()`. Read-only queries and clean exact signed rename cover its declaration and binding-matched named usages; implicit values have no element name to edit. Change-signature and default-value editing are outside this support.
|Java
|`JdtJavaSemanticAnalyzer.kt`, `JavaRenameMemberPlannerTest.kt`, ADR 0008

|Reference
|Source location that references a symbol.
|Core / Java
|`Models.kt`

|Diagnostic
|Structured message with severity and optional location/code.
|Core
|`Models.kt`

|RiskLevel
|Risk classification for a patch or refactoring: LOW, MEDIUM, HIGH.
|Core / safety
|`Models.kt`

|Stable Contract
|An API `1.0` behavior or capability with a versioned schema/semantics, deterministic failure modes, compatibility policy, and passing RC acceptance evidence. Stability does not imply support for every refactoring shape.
|Release / integration
|`docs/releases/v1.0.0-plan.md`, ADRs 0002 and 0004

|Evidence Classification
|Per-operation outcome classified as JDT binding-backed, narrowly proven lexical behavior, or refusal. Lexical behavior must not be presented as semantic authority.
|Java / safety
|ADR 0008

|Release Candidate (RC)
|Immutable `v1.0.0-rc.N` tag used only after the deep language roadmap, Clojure integration, alpha/beta validation, and API freeze to validate proposed stable contracts and artifacts; a failed tagged candidate is replaced by a new commit and RC number.
|Release
|`docs/releases/v1.0.0-plan.md`

|SBOM
|Software Bill of Materials published with a stable runtime asset as supply-chain verification evidence.
|Packaging / release
|ADR 0005

|CLI
|Command-line RefactorKit application.
|Integration
|`refactorkit-cli`

|JSON-RPC Daemon
|Long-running local stdio JSON-RPC process for project sessions and preview/apply/rollback operations.
|Integration
|`refactorkit-daemon`, integration docs

|LSP Server
|Language Server Protocol integration for editor-native operations.
|Integration
|`refactorkit-lsp`

|MCP Server
|Model Context Protocol interface exposing deterministic tools/resources/prompts to local LLM agents.
|Integration
|`refactorkit-mcp`

|External Java Class Importer
|External code assimilation module that creates preview plans for imported Java classes with provenance/license/conflict checks.
|Web importer
|`refactorkit-web-importer`, importer docs

|Recipe
|YAML-defined multi-step refactoring workflow whose steps are previewed against one evolving staged snapshot and reduced to one recipe-wide managed apply/rollback transaction.
|Java / CLI
|`docs/recipe-engine.md`, `recipes/java/rename-package.yml`, transactionality audit

|Golden Test
|Test case with before/after trees, request JSON, and expected plan metadata.
|Testkit
|`docs/testing.md`, `testdata/golden`

|Agent Simulation Test
|Test that simulates local LLM agent workflow: scan, preview, inspect, apply, verify, rollback.
|Testkit
|`docs/testing.md`

|Tree-sitter Foundation
|Current structural multi-language foundation with regex-backed scanner and extension point for real Tree-sitter/native support.
|Multi-language
|`refactorkit-tree-sitter`, multi-language docs

|ExternalLspAdapter
|Adapter intended to launch external language servers and bridge future LSP-backed language support.
|Multi-language
|`docs/multi-language.md`

|Framework-aware Risk Detection
|Lexical detection of Spring, JPA/Jakarta Persistence, and Jackson annotations to raise warnings/risk during selected Java refactorings.
|Java
|`docs/framework-awareness.md`
|===
