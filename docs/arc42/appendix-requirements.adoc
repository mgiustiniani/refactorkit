== Transactionality Requirement Clarifications

These requirements capture the audit's required contract decisions and acceptance gaps. They are not implementation claims.

[cols="1,4,1",options="header"]
|===
|ID |Requirement |Status

|REQ-TX-001
|Define atomicity, consistency validation, isolation, durability, compensation, rollback, and crash recovery separately; do not describe current preflighted compensatable batches as ACID or crash-safe transactions.
|needs-review; release-blocking terminology

|REQ-TX-002
|Define the transaction boundary for one plan, recipe, native LSP edit, custom managed LSP apply, diagnostics, and direct library use.
|needs-review

|REQ-TX-003
|Cover process kill, power loss, disk full, permission change, non-atomic filesystems, log corruption, concurrent writers, and editor-buffer divergence in the failure model.
|planned acceptance

|REQ-TX-004
|Normal rollback must detect post-apply conflicts and refuse rather than silently overwrite later changes; forced and manual recovery require explicit contracts.
|planned release blocker

|REQ-TX-005
|Define a durable versioned journal lifecycle, integrity and permissions, retention, startup recovery, and observable incomplete-recovery states before managed writes are called crash-safe.
|planned release blocker

|REQ-TX-006
|Define snapshot scope, engine-verified affected-file preconditions, per-path expected hashes, metadata scope, case sensitivity, generated files, and symbolic-link behavior.
|planned release blocker

|REQ-TX-007
|Define authoritative diagnostics and regression gates before, during simulation, after apply, and after rollback, including deterministic recovery when validation fails.
|planned release blocker

|REQ-TX-008
|Define one-writer workspace locking, timeout, stale-lock recovery, cancellation, and multi-process behavior.
|planned release blocker

|REQ-TX-009
|Classify native LSP edits as client-managed/no-rollback, overlay full-sync buffers, require monotonic versions, emit exact open-document versions, refuse incapable clients, and guard managed disk writes from open-buffer divergence.
|implemented and tested (`TX-007` closed)

|REQ-TX-010
|Stage every recipe step against one evolving immutable snapshot, expose one recipe-wide plan, and perform at most one managed WAL transaction; planning failure and no-op execution must not write journal records.
|implemented and tested (`TX-008` closed)

|REQ-TX-011
|Use one coordinate space per file/plan, merge all same-file edits, validate complete line/character bounds and cross-edit overlap, and render every final file before mutation.
|planned release blocker

|REQ-TX-012
|Treat explicit managed apply as auditable approval, refuse missing authorization, persist approval identity, map managed refusal categories deterministically, and refresh integration state after mutation.
|implemented and tested (`TX-016`, `TX-017`, `TX-018` closed)
|===
