== Strategy Summary

RefactorKit is structured as a local, deterministic refactoring engine around a language-independent patch core and pluggable language/integration adapters. The primary strategy is to make changes previewable and diagnosable, with managed compensation after completely successful apply, for humans, CI, editors, and local AI agents. The audited current flows are preflighted compensatable batches, not ACID or crash-safe transactions; stronger rollback, isolation, durability, and recovery properties remain planned.

The stable-release strategy has published the JDT foundation as `v0.3.0` and now advances main toward `1.0.0-rc.1-SNAPSHOT`. Proposed API `1.0` contracts will be frozen in immutable release candidates before the accepted candidate is promoted to `v1.0.0`; the active API remains `0.2` until those contract gates pass. Stability is evidence-gated per contract and capability; intentionally unsupported refactoring shapes remain conservative refusals rather than release gaps.

== Fundamental Decisions

[cols="1,3,2,1",options="header"]
|===
|Topic |Strategy |Evidence |Status

|Patch-first architecture
|RefactorKit-managed operations produce `PatchPlan` / `WorkspaceEdit` previews and are applied by the core patch engine. Managed writes use WAL-backed staged per-path atomic replacement; native LSP edits remain client-applied outside this boundary, while recipes stage all steps into one managed transaction.
|`modules/refactorkit-core`, `docs/patch-engine.md`, ADRs 0002/0004, `docs/transactionality-audit.md`.
|implementation-informed limitation

|Language adapter boundary
|The core owns generic models and safety mechanics; language-specific intelligence lives behind `LanguageAdapter` implementations.
|`LanguageAdapter`, `JavaLanguageAdapter`, tree-sitter foundation.
|implementation-informed

|Java-first MVP
|Java is the first semantic language target, with scanner, symbols, references, diagnostics, and Java refactoring planners.
|`refactorkit-java`, Java docs.
|implementation-informed

|Multiple local integration modes
|Expose the same deterministic capabilities through CLI, JSON-RPC daemon, LSP, MCP, and library use.
|Module layout, README, integration docs.
|implementation-informed

|AI-safe workflow
|Local agents should use symbol/context/refactoring tools and review previews rather than directly editing source files.
|`docs/ai-workflow.md`, `refactorkit-mcp`.
|implementation-informed

|Incremental compiler-backed Java analysis with classified evidence
|Proven Java slices use exact Eclipse JDT binding evidence; prototype operations currently retain explicit lexical behavior or conservative refusal when semantic evidence is unavailable or not clean. Stable promotion requires that every fallback be separately narrow and proven safe; otherwise the stable outcome is refusal. No lexical path may claim compiler-backed semantic authority. Class rename and safe delete use exact type-binding evidence when possible. Move class uses exact target-type binding to scope package/import/FQN edits to actual referencing files. Java annotation types are represented as first-class symbols by lexical and JDT indexing and reuse the existing type-level lookup, rename, move, safe-delete, recipe, and owner-validation paths. Annotation elements are represented as zero-parameter signed `METHOD` symbols; read-only lookup/reference queries resolve them, and clean signed member rename updates exact declarations and binding-matched named usages. An unambiguous `FIELD` selector `Owner#field` uses exact field-binding declaration/reference ranges when global JDT evidence is clean, with owner-field collision refusal taking precedence over semantic fallback; unclean or unresolved evidence retains explicit lexical member fallback. This does not imply semantic support for arbitrary unsigned methods. Organize imports uses per-file clean binding-use evidence to remove an exact import only when its imported declaration binding has no non-import use; unresolved files retain lexical cleanup behavior.
|ADR 0008; signed method and unambiguous field member-rename, class-rename, move-class, safe-delete, annotation-type, annotation-element, and organize-imports planners/analyzers and tests.
|implementation-informed

|Framework risk awareness
|Spring, JPA/Jakarta Persistence, and Jackson annotations raise risk and warnings for selected Java refactorings.
|`JavaFrameworkDetector`, `docs/framework-awareness.md`.
|implementation-informed

|Multi-language path
|Provide structural, regex-backed multi-language outline/search/local rename now; leave full semantic support to native bindings or external LSP-backed adapters later.
|`docs/multi-language.md`, `refactorkit-tree-sitter`.
|implementation-informed / planned
|===

== Bounded Contexts / Building Blocks

RefactorKit is not documented as domain-driven business bounded contexts, but its architecture has clear technical bounded contexts:

* Core Patch and Model Engine
* Java Language Adapter
* CLI Application
* JSON-RPC Daemon
* LSP Server
* MCP Server
* External Java Class Importer
* Multi-language Structural Foundation
* Test Harness

These are shown as containers/building blocks in `docs/c4/workspace.dsl`.
