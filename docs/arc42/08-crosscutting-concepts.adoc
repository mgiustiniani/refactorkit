== Patch Safety

All mutating operations are represented as `PatchPlan`s containing operation name, status, snapshot hash, confidence, required approval flag, summary, affected files, `WorkspaceEdit`, diagnostics, warnings, and risk level.

`PatchEngine` validates that:

* only preview plans can be applied;
* current snapshot hash matches the preview hash;
* edit paths stay inside the workspace root, retaining the `path.outsideWorkspace` diagnostic;
* every symbolic-link path component below the workspace root is refused with `path.symbolicLink`, conservatively even when the link resolves inside the workspace;
* text edits do not overlap; and
* ordered create, modify, rename, and delete operations have valid file existence/type transitions.

The ordered state simulation runs before the first write. It prevents a predictable later missing-file or existing-file refusal from leaving an earlier edit applied, while still allowing valid sequences such as modify then rename. Tests cover an outside-pointing symlink with no write, create followed by a missing-file delete with no partial create, and a successful modify-then-rename sequence.

Status: implementation-informed for structural/overlap validation, conservative symlink refusal, ordered state simulation, and snapshot-aware managed-write isolation. `PatchEngine` now holds a non-blocking OS workspace lock across validation/apply/rollback and revalidates initially affected paths under that lock for first-party managed integrations. The hash-only overload is removed, all callers use the snapshot-aware contract, and snapshot hashes bind declared source scope that the engine independently walks/rehashes under lock. Classpath evidence covers active entries, prospective compiled outputs, local JAR directories, and generated declarations; all are recomputed before journaling (`TX-005` and source/classpath-scope `TX-014` closed). Modify entries are normalized into one original-content coordinate space per path/structural segment, with merged overlap checks, complete preflight rendering, and line/character-within-line validation (`TX-009`, `TX-010` closed). Full post-images are rendered before mutation, staged in same-directory temporary files, forced, atomically replaced, and followed by directory force; unsupported durability capability is reported/refused (`TX-001`, `TX-006` implementation-closed on supported filesystems). Managed flows provide filesystem-qualified WAL recovery with per-path atomic commits; they do not claim an impossible filesystem-wide atomic rename across independent paths. No RefactorKit-managed production write may bypass `PatchEngine`; native LSP client writes are outside this boundary.

== Compensation, Journal, and Recovery

Before every managed write, `PatchEngine` durably creates a versioned `PREPARED` journal record containing transaction identity, operation, forward/compensation edits, and affected-path pre/post images, then advances to `APPLYING`. Apply, rollback, and startup recovery retain lifecycle states through `APPLIED`, `ROLLING_BACK`, `ROLLED_BACK`, or `RECOVERY_REQUIRED`; integrations no longer persist metadata after mutation (`TX-002` closed, `TX-001` narrowed).

Transaction identifiers now enforce the generated UUIDv4 grammar; normalized transaction-log paths are contained, reject symbolic links and non-regular records, apply owner-only permissions where supported, and report coded corruption/path failures (`TX-003` closed after the audit baseline). Records now retain a versioned lifecycle, forward/compensation edits, pre/post images, failure detail, and atomically/durably replaced history. Schema-v2 canonical SHA-256 checksums, v1-on-update migration, and blocking owner-only corruption quarantine are implemented; schema-v3 adds checksummed implementation/API versions, pre/post scope hashes, and append-only validation/lifecycle/recovery events; v2 checksum verification remains backward compatible; schema-v8 adds independently verified per-image content hashes; TX-011/TX-012 are closed for the reported durability contract. Schema-v8 compensation restores text, POSIX permissions, timestamps, owner/group identity, user-defined attributes, and exposed platform ACLs and removes transaction-created parents deepest-first; external directory contents trigger conflict refusal. The v1 text contract is strict UTF-8 with exact BOM preservation and malformed-input refusal before WAL creation. Standalone directory operations are outside the stable v1 contract (`TX-013` closed by precise scope narrowing).

Normal rollback verifies exact retained post-images under the workspace lock and refuses conflicts without writes; explicit force restores pre-images and records the override (`TX-004` closed for versioned records). Snapshot-aware apply independently rescans hash-bound source roots/extensions/ignore policy under the workspace lock, so stale library snapshots cannot omit newly appeared declared-scope sources. Java classpath artifacts and discovery manifests are likewise fingerprinted and revalidated (`TX-014` closed for source/classpath scope). Recipe steps are evaluated against one evolving immutable snapshot and reduced to one recipe-wide `PatchPlan`; apply creates one WAL transaction, while planning failure writes nothing (`TX-008` closed).

Status: implementation-informed from `docs/transactionality-audit.md`. Write-ahead lifecycle, compatible startup compensation, explicit `RECOVERY_REQUIRED`, conflict-safe rollback, durable supported-filesystem replacement, and engine-owned source/classpath scope are implemented. Versioned checksums, independent image hashes, migration, quarantine, metadata policy, deterministic faults, real kills, truncation, and cross-store evidence close the qualified transactionality requirements.

== Diagnostics Model

Diagnostics use structured severity (`INFO`, `WARNING`, `ERROR`), optional source location, and optional code. Java diagnostics currently include duplicate symbol detection, package/path mismatch warnings, and public type/file-name mismatch errors.

The stable target extends the contract with deterministic source range, evidence, and error-category fields and with comparison before preview, after preview simulation, after apply, and after rollback where an operation requires those gates. A configured regression beyond the approved plan causes refusal.

Status: implementation-informed. Managed Java surfaces supply JDT diagnostics to a core gate that compares current and exact staged post-image snapshots under lock before WAL, refusing new errors or unavailable diagnostics. Direct library apply requires an explicit gate (`TX-015` closed).

== Risk Model

Refactorings classify risk as `LOW`, `MEDIUM`, or `HIGH`. Examples:

* low risk: simple class/member rename where references are understood;
* medium risk: move class, safe delete, limited extract method, limited change signature;
* high risk: framework-managed types or external/import/license uncertainty.

Status: implementation-informed.

== Java Analysis and Refactoring

The Java adapter combines lexical/source-level scanning with selected Eclipse JDT-backed semantic-analysis slices. The following paragraphs describe recorded `v0.3.0` prototype evidence, not automatic stable classification. Core and JDT semantic symbol kinds include `ANNOTATION`; lexical top-level indexing recognizes `@interface`, while JDT visits `AnnotationTypeDeclaration` nodes and records binding identity, declaration ranges, nesting ownership, and binding-matched usage references. Annotation types participate in read-only lookup and reuse the existing class-style rename, move, safe-delete, recipe, and type-owner validation paths.

JDT also visits `AnnotationTypeMemberDeclaration` nodes and emits annotation elements as `METHOD` symbols with zero-parameter signed identities such as `com.acme.Route#path()`. Method-binding evidence supports read-only symbol/reference queries and, when analysis is clean, the existing signed `renameMember` path updates the exact declaration and binding-matched named usages such as `@Route(path=...)`. Declaration names are excluded from reference results. Implicit single-element values have no name to edit; annotation-element change-signature and default-value editing are not supported.

Signed `renameMember` uses JDT binding keys to rename transitive source-visible class/interface override families, including their scanned declarations and binding-matched call sites. Static and private methods do not form override relations, and a family containing an external declaration is refused.

For an unambiguous `FIELD` selector such as `com.acme.Stats#count`, `JavaRenameMemberPlanner` uses the exact field binding when global JDT evidence is clean and emits edits from its declaration and binding-matched reference ranges. This distinguishes qualified field access from shadowing locals and unrelated same-named fields owned by other types. If the owner already declares the requested target field, collision validation refuses the operation before semantic fallback; unclean or unresolved evidence therefore cannot bypass that safety check and instead retains the explicit lexical member fallback only when no collision exists. This field-specific behavior does not extend semantic guarantees to arbitrary unsigned method selectors.

Class rename resolves the exact target type binding when JDT analysis is clean. It generates source-range edits for the type declaration, constructor declarations, and binding-matched imports, type references, and constructor calls before renaming the source file. Binding identity leaves same-simple-name types in other packages unchanged. An unchanged requested name or an existing target symbol/file is refused. JDT parse/classpath warnings or an unavailable exact target binding trigger lexical behavior with an explicit fallback warning; reflection, framework, and generated-code warnings remain.

Move class resolves the exact target type binding when JDT analysis is clean and restricts package/import/FQN edits to files containing binding-matched references. This prevents imports in unrelated old-package files and preserves unrelated same-simple-name types. The package-declaration edit and source-file move remain patch-plan operations. A direct import is removed when the referencing file is in the target package, and direct-import ranges are excluded from nested FQN replacement to prevent overlapping edits. Invalid target packages and existing target FQNs/files are refused. Parse/classpath warnings or unavailable exact binding evidence trigger explicit lexical file scoping.

Safe delete uses exact JDT type-binding references when semantic analysis is clean and the exact target binding is available. Binding identity distinguishes types that share a simple name across packages. JDT parse/classpath warnings or an unavailable exact target binding trigger the existing lexical scan and an explicit fallback warning. Forced-delete behavior and framework warnings remain unchanged.

For organize imports, the semantic analyzer exposes resolved binding uses from imports and ordinary code and marks import context. Parameterized type, generic method, and variable binding keys are normalized to declaration bindings so generic uses match imported declarations. With clean evidence for a file, the planner removes an exact import only when that binding has no non-import use. Wildcard imports remain conservative. When a file is unresolved, the planner explicitly retains lexical sorting, deduplication, and same-package removal rather than claiming unused-import evidence. Preview, apply, and rollback continue through the common patch safety model.

Other conservative planners refuse ambiguous cases instead of guessing, especially extract method and structural change-signature operations. Known limitations include no annotation-element change-signature or default-value editing, incomplete compiler-backed coverage outside supported signed-method and unambiguous-field member rename (including exact annotation-element rename), type-level rename, move-class, safe-delete, annotation-type/element indexing and reference, and organize-imports slices, and limited handling of reflection, generated code, external configuration, framework strings, annotation-processor output, and unknown external callers.

For stable promotion, the JDT dependency and per-project source compliance must support representative Java 8, 11, 17, 21, and 25 source evidence while RefactorKit's documented Java 8-compatible bytecode constraint remains separate. Source roots, module relationships, JDK types, outputs, local JARs, and explicit/generated dependency classpaths must be deterministic and visible. Any semantic operation without clean exact evidence must refuse unless a separately classified lexical behavior is proven incapable of incorrect semantic edits; previews must label JDT-backed, lexical, and refused outcomes.

Status: implementation-informed prototype; stable capabilities are evidence-gated.

== Integration Transaction Boundaries

CLI `--apply`, daemon `refactor.apply`, MCP `apply_refactoring`, and custom LSP `refactorkit.applyPlan` use the managed WAL-backed flow. Native LSP rename and WorkspaceEdit commands are explicitly client-managed/no-rollback. Full-sync open-buffer overlays feed planning, open-document edits carry exact versions, unsupported versionable edits refuse, and managed LSP disk writes reject dirty or affected-open buffers (`TX-007` closed).

Explicit managed apply invocation is the approval event; missing authorization refuses before journaling, while successful transactions persist approval kind/surface/actor/time (`TX-016` closed). Daemon/LSP/MCP use a core-owned deterministic refusal mapper for snapshot, recovery, validation, lock, filesystem, unsafe-path, file-conflict, approval, and apply/journal categories (`TX-017` closed). Daemon apply/rollback refreshes stored session state, clears stale plans, and returns the new hash (`TX-018` closed).

Status: implementation-informed. LSP ownership/document versions and daemon state refresh are implemented; approval semantics and managed apply/rollback error categories are implemented; central diagnostics gates are implemented and required for direct apply.

== Protocol Capability Discovery

The stdio JSON-RPC daemon exposes read-only `server.capabilities` before `project.open`. Its response identifies the implementation and API version, fixes protocol and transport to `json-rpc-2.0` over `stdio`, and describes each documented method by stability, project prerequisite, and workspace-write behavior. The current external Java import preview method is marked `experimental`; the documented daemon baseline remains `beta-contract` until the planned API `1.0` inventory, migration policy, schemas, snapshots, and compatibility tests pass.

The response advertises the cross-cutting safety properties `previewBeforeApply`, `snapshotValidation`, `transactionRollback`, and `workspaceScopedWrites`. These flags describe server behavior but must not be interpreted as one filesystem-wide ACID commit or protection from hostile writers mutating classpath artifacts after preflight validation; conflict-safe normal rollback applies only to versioned journal records with pre/post images. Capability discovery neither opens a project nor authorizes filesystem access. A daemon contract test exercises response metadata on a fresh session.

For stable API `1.0`, documented core patch, CLI, and daemon fields, methods, errors, exit categories, lifecycle behavior, and capability metadata are versioned contracts. Unknown methods/fields, invalid parameters, unsupported versions, stale plans, and rollback failures must have deterministic outcomes.

Status: implementation-informed baseline; API `1.0` freeze planned.

== Agent and MCP Safety

MCP exposes deterministic tools, resources, and prompts. File resources are scoped to the scanned workspace snapshot and arbitrary filesystem access is refused. Agent workflow expects project scan, focused context, preview, inspection, apply, diagnostics, and rollback.

Every MCP tool/resource/prompt and LSP capability/command must be classified as stable or experimental. Stable paths require protocol initialization/capability, malformed-input, refusal, cancellation, and compatibility evidence. MCP remains least-privilege and provides no arbitrary filesystem tool or mutation path outside preview/apply.

Status: implementation-informed baseline; stable boundary classification planned.

== Security and Trust Boundaries

RefactorKit operates locally on user workspaces. Main trust boundaries are:

* external code sources for importer inputs;
* local LLM agents/MCP clients;
* editor/LSP clients;
* target workspace filesystem;
* optional external language servers.

Security controls currently visible in the repository are workspace path validation, MCP resource scoping, provenance/license warnings, and overwrite refusals.

Status: implementation-informed, needs-review for broader threat model.
