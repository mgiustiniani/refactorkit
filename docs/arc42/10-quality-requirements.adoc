= 10. Quality Requirements
:toc:
:sectnums:

Status: implementation-informed, needs-review for priorities and measurable targets.

== Quality Tree

* Safety
** Preview before apply
** Reject stale or unsafe edits
** Compensating rollback after a completely successful managed apply
** Implemented managed-write recovery/isolation/durability and versioned conflict-safe rollback; destructive acceptance evidence remains
** Warnings for high-risk Java/framework/import cases
* Correctness
** Symbol-aware Java operations where supported
** Conservative refusal instead of guessing
** Diagnostics before/after preview where supported
* Extensibility
** Language-independent core
** Language adapter contract
** Structural multi-language foundation and optional future LSP-backed adapters
* Usability
** CLI, daemon, LSP, MCP, and library integration modes
** Focused MCP resources and prompts for agents
* Portability and supply-chain integrity
** Java 8-compatible RefactorKit bytecode, separately verified from Java source support
** Self-contained CLI package with embedded runtime
** Checksum, SBOM, provenance/attestation, and reproducibility evidence
* Performance and resource control
** Measurable reference-workspace budgets
** Bounded requests, results, caches, pending plans, transaction metadata, memory, and execution time
* Testability
** Unit tests, golden tests, sample projects, agent simulation tests, CI

|===
|ID |Quality |Scenario |Evidence / validation |Status

|QS-SAFE-001
|Safety
|When a user applies a preview after the workspace has changed, RefactorKit refuses apply and asks for plan regeneration.
|`PatchEngine.validate` checks snapshot hash mismatch.
|implementation-informed

|QS-SAFE-002
|Safety
|When a patch contains overlapping text edits, RefactorKit refuses apply.
|`PatchEngine.validateNoOverlaps`; core tests.
|implementation-informed

|QS-SAFE-003
|Safety
|When a patch attempts to edit outside the workspace root, RefactorKit refuses it.
|`PatchEngine.validateInsideWorkspace`.
|implementation-informed

|QS-SAFE-004
|Safety
|When an external Java class import would overwrite an existing class/file, the importer refuses by default.
|`docs/external-class-importer.md`, importer tests.
|implementation-informed

|QS-SAFE-006
|Safety
|When any edit path contains a symbolic-link component below the workspace root, RefactorKit refuses it with `path.symbolicLink` before writing, even when the link resolves inside the workspace; ordinary path escapes continue to use `path.outsideWorkspace`.
|`PatchEngineTest.refusesSymbolicLinkTraversalWithoutWritingOutsideWorkspace`; `PatchEngine.validateNoSymbolicLinkTraversal`.
|implementation-informed

|QS-SAFE-007
|Safety
|Before the first write, RefactorKit simulates ordered create, modify, rename, and delete existence/type transitions. A later predictable missing-file or existing-file refusal leaves earlier edits unapplied, while a valid modify-then-rename sequence succeeds.
|`PatchEngineTest.preflightRefusalLeavesEarlierEditsUnapplied`, `PatchEngineTest.preflightSupportsSequentialModifyThenRename`; `PatchEngine.validateRuntimeState`.
|implementation-informed; interrupted I/O and TOCTOU races remain open

|QS-SAFE-008
|Safety and recoverability
|When a managed apply completes without interruption or concurrency, RefactorKit returns reverse compensation and the integration saves it afterward. The contract describes this as a preflighted compensatable batch and does not promise ACID, crash-safe, durable, or all-or-nothing execution.
|`docs/transactionality-audit.md`, ADR 0002.
|implementation-informed limitation

|QS-SAFE-009
|Data integrity
|If an affected file differs from its expected post-apply state, normal rollback refuses rather than overwriting the later edit and exposes conflict/manual-recovery information.
|Transactionality audit `TX-004`; future fault/concurrency acceptance suite.
|planned release blocker

|QS-SAFE-010
|Security and recovery
|Malformed transaction IDs, traversal, transaction-directory symlinks, truncated/corrupt journal records, or integrity failures produce structured refusal without reading, deleting, or overwriting data outside the protected journal boundary.
|Transactionality audit `TX-003`, `TX-011`, `TX-012`; future security/corruption suite.
|planned release blocker

|QS-SAFE-011
|Isolation and durability
|Under concurrent writers, I/O failure, process kill, power loss, disk full, permission changes, or filesystems without atomic move, managed apply either preserves the prior state or records a durable, deterministic recovery state and declared degraded guarantee.
|Transactionality audit `TX-001`, `TX-002`, `TX-005`, `TX-006`; future fault-injection/restart/filesystem suite.
|planned release blocker

|QS-CORR-001
|Correctness
|When clean JDT analysis resolves the safe-delete target binding, RefactorKit refuses deletion for binding-matched references without conflating unrelated same-simple-name types in other packages. If JDT parse/classpath warnings or the exact target binding prevent semantic evidence, it uses the lexical scan and warns explicitly.
|`JavaSafeDeletePlannerTest`: imported target refusal with JDT evidence, unrelated same-simple-name type allowed, and lexical fallback warning.
|implementation-informed

|QS-CORR-002
|Correctness
|When extract-method selection has control-flow or variable-dependency ambiguity, planner refuses rather than guessing parameters/returns.
|`docs/extract-method.md`, tests.
|implementation-informed

|QS-CORR-003
|Correctness
|When change-signature would affect overloaded, generated, method-reference, public/protected hierarchy, or unsafe cases, planner refuses or escalates risk.
|`docs/change-signature.md`, tests.
|implementation-informed

|QS-CORR-004
|Correctness
|When clean JDT analysis resolves the exact class-rename target binding, RefactorKit renames the type declaration, constructors, binding-matched imports/type references/constructor calls, and source file without changing same-simple-name types in other packages. It refuses unchanged names and existing target symbol/file conflicts; unavailable clean binding evidence produces an explicit lexical-fallback warning.
|`JavaRenameClassPlannerTest`: constructor rename/apply, imports, same-simple-name disambiguation, target conflict refusal, and fallback warning.
|implementation-informed

|QS-CORR-005
|Correctness
|When clean JDT analysis resolves the move-class target binding, RefactorKit scopes package/import/FQN edits to actual referencing files while retaining package-declaration and file-move patch operations. It avoids unrelated old-package imports, preserves unrelated same-simple-name types, removes now-redundant target-package imports, and avoids overlapping direct-import/FQN edits. Invalid target packages and existing target FQNs/files are refused; unavailable clean binding evidence produces an explicit lexical-file-scoping warning.
|`JavaMoveClassPlannerTest`: apply, used versus unused old-package files, target-package import removal, unrelated same-simple-name preservation, invalid/conflict refusal, and fallback warning.
|implementation-informed

|QS-CORR-006
|Correctness
|For a file with clean JDT evidence, organize imports removes an exact import only when its declaration binding has no non-import use, including normalized generic type, method, and variable uses. It preserves wildcard imports. For an unresolved file, it retains lexical sorting, deduplication, and same-package removal without claiming semantic unused-import cleanup.
|`JavaOrganizeImportsPlannerTest`: used `List<Map<...>>` and generic static `emptyList` imports retained; unused `Set` and `singletonList` imports removed; wildcard preservation; unresolved-file lexical behavior; existing preview/apply/rollback coverage.
|implementation-informed

|QS-CORR-007
|Correctness
|When two annotation types share the simple name `Audited`, an exact rename of `com.acme.Audited` to `Tracked` changes its declaration, source file, and binding-matched usages while preserving `com.other.Audited` and its usage. Annotation types also remain available to read-only lookup and existing type-level move, safe-delete, recipe, and owner-validation paths.
|`JavaRenameClassPlannerTest`: exact annotation-type declaration/file/usage rename with same-simple-name package disambiguation; lexical and JDT annotation indexing/analyzer evidence.
|implementation-informed

|QS-CORR-008
|Correctness
|Given the signed annotation-element selector `com.acme.Route#path()`, read-only lookup/reference queries resolve the zero-parameter `METHOD` symbol and its binding-matched named usages. With clean semantic evidence, rename to `value` updates the exact declaration and `@Route(path=...)` name while preserving an unrelated `com.other.Route#path()` declaration and usage. Implicit single-element values contain no name to edit.
|`JavaRenameMemberPlannerTest`: annotation-element lookup/reference and applied signed rename with package-disambiguation evidence.
|implementation-informed

|QS-CORR-009
|Correctness
|Given an unambiguous `FIELD` selector `com.acme.Stats#count` and clean global JDT evidence, rename uses exact field-binding ranges so the declaration, `this.count`, and `stats.count` change while a shadowing local `count` and `com.other.Stats#count` remain unchanged. If the owner already declares the target field, the planner refuses before semantic fallback; unclean or unresolved evidence otherwise retains explicit lexical member fallback. This scenario does not cover arbitrary unsigned methods.
|`JavaRenameMemberPlannerTest`: exact field declaration/qualified-reference apply with local-shadow and other-owner preservation, owner-field target-collision refusal, and fallback behavior.
|implementation-informed

|QS-AI-001
|AI safety
|A local LLM can get focused context through MCP without arbitrary filesystem access and request preview/refactoring tools.

|implementation-informed

|QS-SAFE-005

|QS-CORR-010
|Correctness

|QS-COMPAT-001

|Stable Java compatibility matrix; ADRs 0001 and 0008.
|confirmed target; planned evidence

|QS-DIAG-001
|Validation
|Stable compiler-backed diagnostics have deterministic severity, source range, evidence, and category, and configured before/simulation/after/rollback gates refuse unapproved regressions.
|Stable diagnostics acceptance matrix; ADRs 0002 and 0008.
|confirmed target; planned evidence

|QS-API-002
|Interoperability
|API `1.0` CLI and daemon clients receive versioned capability metadata and deterministic outcomes for unknown methods/fields, invalid parameters, stale plans, rollback failures, and supported `0.2` transition behavior or explicit version refusal.
|Stable protocol contract suite; ADR 0004.
|confirmed target; planned evidence

|QS-PROTO-001
|Interoperability and security
|Every exposed LSP capability/command and MCP tool/resource/prompt is classified stable or experimental; stable paths pass initialization, malformed-input, refusal, cancellation, and workspace-boundary tests without arbitrary MCP filesystem access.
|Stable LSP/MCP contract suites; ADRs 0004 and 0006.
|confirmed target; planned evidence

|QS-PROTO-002
|Consistency and interoperability
|A native LSP edit is identified as client-managed/no-rollback, plans from full-sync open-buffer overlays, carries exact open-document versions, and refuses incapable clients; managed LSP writes reject dirty or affected-open buffers.
|Transactionality audit `TX-007`; ADR 0004; future LSP contract tests.

|QS-RECIPE-001
|Recoverability

|QS-EDIT-001
|Correctness
|All modifications to one file use one original coordinate space, are merged and checked together for overlap and complete line/character bounds, and every final file is rendered before the first workspace mutation.

|planned release blocker

|QS-PERF-001
|Performance
|Reference workspaces remain within documented scan, index, preview, apply, and memory budgets, while daemon requests, results, caches, pending plans, and transaction metadata fail with deterministic limit/timeout/cancellation behavior rather than unbounded growth.

== Open Quality Gaps

[cols="1,3,1",options="header"]
|===
|Gap |Impact |Status

|No confirmed quality-goal priorities
|Chapter 1 quality goals may not reflect business priorities.
|needs-review

|Compiler-backed Java analysis covers selected slices only
|Signed method rename (including exact annotation-element declarations and named usages), unambiguous `Owner#field` field rename, type-level rename (including annotation types), move-class file scoping, safe-delete type references, annotation-type/element indexing and reference analysis, and exact-import use checks use selected JDT binding evidence, but lexical/structural behavior and false positives/negatives remain elsewhere and on explicit fallback paths. Field evidence does not imply semantic support for arbitrary unsigned methods. Annotation-element change-signature and default-value editing are not supported. Organize imports preserves wildcard imports and unresolved files do not receive semantic unused-import removal.
|known risk

|Performance targets not yet accepted
|The stable plan now requires measurable reference-workspace and resource budgets, but numeric budgets and passing evidence remain pending.
|planned release blocker

|No full threat model
|Security properties for MCP/importer/external language-server boundaries need formal review; stable MCP least-privilege and workspace-boundary tests reduce but do not eliminate this gap.
|needs-review

|Transactionality guarantees exceed current evidence
|Managed writes now have one-writer locking, affected-file revalidation, durable write-ahead lifecycle, compatible startup compensation, explicit recovery-required blocking, and supported-filesystem staged/forced/atomic replacement. Declared source scope is hash-bound and independently rescanned under lock; Java classpath entries, prospective outputs, local JAR directories, and declaration files are content-fingerprinted and revalidated before journaling. Same-file coordinates, cross-entry overlaps, complete preflight rendering, and line/character bounds are now enforced. Current flows still lack central diagnostics gates, enforced approval semantics, complete apply error mapping, and current daemon state. Native LSP writes are version-checked and explicitly client-managed; recipes use one staged managed transaction.
|release blocker; remaining and partially evidenced `TX` findings require closure plus destructive fault acceptance

|Stable acceptance evidence incomplete
|API `1.0`, Java 8–25, diagnostics, protocol classification, packaging, supply-chain, reproducibility, and narrowed-or-evidenced transactionality claims cannot be marked stable until the immutable RC acceptance matrix passes.
|planned release blocker
|===
