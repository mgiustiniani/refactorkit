= 10. Quality Requirements
:toc:
:sectnums:

Status: implementation-informed, needs-review for priorities and measurable targets.

== Quality Tree

* Safety
** Preview before apply
** Reject stale or unsafe edits
** Rollback after apply
** Warnings for high-risk Java/framework/import cases
* Correctness
** Symbol-aware Java operations where supported
** Conservative refusal instead of guessing
** Diagnostics before/after preview where supported
* Extensibility
** Language-independent core
** Language adapter contract
** Structural multi-language foundation and optional future LSP-backed adapters
* Usability
** CLI, daemon, LSP, MCP, and library integration modes
** Focused MCP resources and prompts for agents
* Portability
** Java 8-compatible bytecode where configured
** Self-contained CLI package with embedded runtime
* Testability
** Unit tests, golden tests, sample projects, agent simulation tests, CI

|QS-SAFE-003
|Safety
|When a patch attempts to edit outside the workspace root, RefactorKit refuses it.
|`PatchEngine.validateInsideWorkspace`.
|implementation-informed

|QS-SAFE-004
|Safety
|When an external Java class import would overwrite an existing class/file, the importer refuses by default.
|`docs/external-class-importer.md`, importer tests.
|implementation-informed

|QS-CORR-001
|Correctness
|When clean JDT analysis resolves the safe-delete target binding, RefactorKit refuses deletion for binding-matched references without conflating unrelated same-simple-name types in other packages. If JDT parse/classpath warnings or the exact target binding prevent semantic evidence, it uses the lexical scan and warns explicitly.
|`JavaSafeDeletePlannerTest`: imported target refusal with JDT evidence, unrelated same-simple-name type allowed, and lexical fallback warning.
|implementation-informed

|QS-CORR-002
|Correctness
|When extract-method selection has control-flow or variable-dependency ambiguity, planner refuses rather than guessing parameters/returns.
|`docs/extract-method.md`, tests.
|implementation-informed

|QS-CORR-003
|Correctness
|When change-signature would affect overloaded, generated, method-reference, public/protected hierarchy, or unsafe cases, planner refuses or escalates risk.
|`docs/change-signature.md`, tests.
|implementation-informed

|QS-CORR-004
|Correctness
|When clean JDT analysis resolves the exact class-rename target binding, RefactorKit renames the type declaration, constructors, binding-matched imports/type references/constructor calls, and source file without changing same-simple-name types in other packages. It refuses unchanged names and existing target symbol/file conflicts; unavailable clean binding evidence produces an explicit lexical-fallback warning.
|`JavaRenameClassPlannerTest`: constructor rename/apply, imports, same-simple-name disambiguation, target conflict refusal, and fallback warning.
|implementation-informed

|QS-CORR-005
|Correctness
|When clean JDT analysis resolves the move-class target binding, RefactorKit scopes package/import/FQN edits to actual referencing files while retaining package-declaration and file-move patch operations. It avoids unrelated old-package imports, preserves unrelated same-simple-name types, removes now-redundant target-package imports, and avoids overlapping direct-import/FQN edits. Invalid target packages and existing target FQNs/files are refused; unavailable clean binding evidence produces an explicit lexical-file-scoping warning.
|`JavaMoveClassPlannerTest`: apply, used versus unused old-package files, target-package import removal, unrelated same-simple-name preservation, invalid/conflict refusal, and fallback warning.
|implementation-informed
