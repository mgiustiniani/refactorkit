== Technical Constraints

[cols="1,3,2,1",options="header"]
|===
|ID |Constraint |Evidence |Status

|C-TECH-001
|The main implementation language is Kotlin/JVM.
|Gradle Kotlin modules and source layout.
|implementation-informed

|C-TECH-002
|Java bytecode compatibility is configured for Java 8 where possible.
|Root `build.gradle.kts` configures Java compile release 8 and Kotlin JVM target 1.8.
|implementation-informed

|C-TECH-003
|The build itself uses a JDK 21 toolchain for Kotlin/JVM modules and CI.
|Module build scripts use `jvmToolchain(21)`; CI uses Temurin 21.
|implementation-informed

|C-TECH-004
|Core engine must stay language-independent and must not depend on Java-specific logic.
|`refactorkit-core` contains `LanguageAdapter`; Java-specific planners live in `refactorkit-java`.
|implementation-informed

|C-TECH-005
|Every mutating operation must produce a previewable `PatchPlan` before application.
|`AGENTS.md`; patch model and planner classes.
|implementation-informed

|C-TECH-006
|Patch application must reject stale snapshot hashes, overlapping edits, paths outside the workspace root, and any symbolic-link component below the workspace root. Before its first write, it must simulate ordered file existence/type transitions so predictable missing-file or existing-file refusals cannot leave earlier edits applied.
|`PatchEngine.validate`, `PatchEngine.validateEdits`, and `PatchEngine.validateRuntimeState`; ADR 0002; core tests.
|implementation-informed; interrupted I/O and TOCTOU safety remain open

|C-TECH-007
|Current managed apply returns compensation only after all sequential writes succeed, and integrations then store transaction logs under `.refactorkit/transactions`. This is a preflighted compensatable batch, not an ACID, crash-safe, durable, or isolated transaction.
|`TransactionLog`, ADR 0002, `docs/transactionality-audit.md`.
|implementation-informed limitation

|C-TECH-008
|External Java class imports must remain preview-based, warn about unknown license/provenance, and avoid overwriting by default.
|`docs/external-class-importer.md`, `ExternalJavaClassImporter`.
|implementation-informed

|C-TECH-009
|The CLI distribution must be able to include an embedded Java runtime created with `jlink`.
|`docs/packaging.md`, CLI Gradle packaging tasks.
|implementation-informed

|C-TECH-010
|The stdio daemon must expose read-only capability discovery before a project is opened. Discovery reports protocol and transport identity, method stability and project/write requirements, and the safety properties clients may rely on.
|`DaemonSession.serverCapabilities`, `DaemonSessionTest`, `docs/integration-contracts.md`.
|implementation-informed

|C-TECH-011
|Stable Java source support must use an Eclipse JDT version and explicit project compliance configuration proven on representative Java 8, 11, 17, 21, and 25 constructs. Java 8-compatible RefactorKit bytecode is a separate constraint and is not source-support evidence.
|`docs/releases/v1.0.0-plan.md`; ADRs 0001 and 0008.
|confirmed target; planned acceptance

|C-TECH-012
|A stable Java operation must classify its evidence as JDT binding-backed, narrowly proven lexical behavior, or refusal. Missing or unclean semantic evidence must never become hidden lexical semantic authority.
|`docs/releases/v1.0.0-plan.md`; ADRs 0003, 0007, and 0008.
|confirmed target; planned acceptance

|C-TECH-013
|Stable diagnostics and validation must provide deterministic severity, source range, evidence, and error categories and must refuse configured regressions beyond the approved plan.
|`docs/releases/v1.0.0-plan.md`; ADRs 0002 and 0008.
|confirmed target; planned acceptance

|C-TECH-014
|Any stable transactionality claim must define and evidence its mutation boundary, filesystem capability/degradation, write-ahead recovery lifecycle, one-writer isolation, affected-file preconditions, conflict-safe compensation, journal security/integrity, complete edit rendering, diagnostics gates, approval semantics, and recovery/error states. Until then, documentation must use the narrower compensatable-batch classification.
|ADRs 0002 and 0004; `docs/transactionality-audit.md` (`TX-001` through `TX-018`).
|planned release constraint; needs-review
|===

== Architectural Boundary Constraints

[cols="1,3,2,1",options="header"]
|===
|ID |Constraint |Evidence |Status

|C-BOUND-001
|Frontend/UI technologies are out of scope for this repository.
|`AGENTS.md` scope and non-goals.
|implementation-informed

|C-BOUND-002
|LSP is for editor-native features; advanced AI workflows use JSON-RPC or MCP instead of overloading LSP. Native LSP `WorkspaceEdit`s are version-checked, client-managed, and have no RefactorKit transaction/rollback; custom `refactorkit.applyPlan` is a distinct managed path that refuses dirty/affected-open buffers.
|`AGENTS.md`, `docs/integration-contracts.md`, ADR 0004, transactionality audit `TX-007`.
|implementation-informed limitation

|C-BOUND-003
|MCP must expose deterministic tools and focused resources, not arbitrary filesystem access.
|`docs/integration-contracts.md`, `docs/ai-workflow.md`.
|implementation-informed

|C-BOUND-004
|C2 containers in the C4 model represent runtime entry points or intentionally separable building blocks; current library modules are not necessarily standalone deployables.
|`docs/c4/workspace.dsl`.
|implementation-informed

|C-BOUND-005
|At `v1.0.0`, core patch, documented CLI, and daemon contracts use API version `1.0`; every LSP capability/command and MCP tool/resource/prompt must be classified independently as stable or experimental.
|`docs/releases/v1.0.0-plan.md`; ADRs 0002, 0004, and 0006.
|confirmed target; planned acceptance
|===

== Process and Quality Constraints

[cols="1,3,2,1",options="header"]
|===
|ID |Constraint |Evidence |Status

|C-PROC-001
|CI must run Gradle build, golden tests, self-contained CLI packaging, and a packaged CLI smoke test.
|`.github/workflows/ci.yml`.
|implementation-informed

|C-PROC-002
|Features are done only with tests, preview support, rollback when writing files, diagnostics, and documentation.
|`AGENTS.md` Definition of Done.
|planned, needs-review for enforcement coverage

|C-PROC-003
|Semantic Java refactoring must not be implemented as global search-and-replace.
|`AGENTS.md` hard rules.
|implementation-informed as design constraint; needs ongoing review

|C-PROC-004
|ADRs must be stored separately and chapter 9 must remain an index rather than rewriting ADR bodies.
|Installed arc42 agent rule; `docs/adr/`.
|implementation-informed

|C-PROC-005
|Release order is published immutable `v0.3.0`, `v0.4.0` hardening, incremental deep multi-language `0.x` bands, Clojure-complete `1.0` development, then immutable alpha/beta/RC tags and `v1.0.0`; tags must not be moved or recreated. Main reports `0.4.0-SNAPSHOT`, while API `0.2` remains active until an explicit migration passes.
|`docs/releases/v1.0.0-plan.md`; ADRs 0004, 0005, and 0008.
|confirmed; `v0.3.0` step implementation-informed

|C-PROC-006
|Stable acceptance must define and pass bounded scan, index, preview, apply, memory, request, result, cache, pending-plan, and transaction-metadata budgets with deterministic timeout/cancellation/limit behavior.
|`docs/releases/v1.0.0-plan.md`; ADRs 0004, 0006, and 0008.
|confirmed target; planned acceptance

|C-PROC-007
|Stable self-contained release assets require checksum, SBOM, provenance/attestation, clean-build reproducibility comparison, archive-layout verification, and `JAVA_HOME`-unset smoke evidence.
|`docs/releases/v1.0.0-plan.md`; ADR 0005.
|confirmed target; planned acceptance
|===
