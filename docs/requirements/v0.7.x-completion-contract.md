# RefactorKit 0.7.x completion contract

Status: active release-blocking contract.

## Decision

The `0.7.x` band is not complete at K4. Every still-open K5, T5, J1 and I1 row
listed in `docs/releases/v0.7.0-plan.md` is release-blocking for closure of the
band. Work remains split into small requirement-first, independently previewable,
tested and qualified commits; this scope decision does not authorize one broad
rewrite or weaken refusal behavior.

API remains `0.2` throughout the band. Compatibility projections may be retired
only by an additive/versioned migration with executable compatibility evidence.

## Mandatory workstreams

### Shared JVM authority

Before broad public Kotlin or Java/Kotlin mutation, RefactorKit shall establish a
shared JVM declaration/reference identity mapping for at least one public type
and one member shape. Kotlin K2 and Java JDT evidence must map independently to
the same binary identity and exact source tokens. If either side, a generated
boundary, an external consumer or the build/classpath model is incomplete, the
operation shall return a stable typed refusal.

### Kotlin K5

Every K5 catalogue row shall have:

1. an English requirement and executable refusal/success tests;
2. compiler/Analysis API evidence rather than lexical authority;
3. normalized exact edits and staged diagnostics;
4. CLI/daemon/MCP transport where the operation is user-visible;
5. managed apply, WAL, recovery and rollback for writes;
6. native packaged qualification for each promoted mutation family.

Capability matrices may classify unsupported multiplatform, Android, plugin or
framework shapes as `REFUSED`; completion requires truthful detection and stable
reasons, not fabricated support.

### TypeScript/JavaScript T5

All deferred T5 operations are included: move, imports/formatting, change
signature, bounded extract/inline and migration recipes. TypeScript compiler or
language-service evidence must remain toolchain/snapshot bound. Dynamic
JavaScript never inherits TypeScript completeness.

### Java/shared-platform J1

All J1 rows are included, including bounded JDT catalogue expansion and
transactional `java.moveAcrossMavenModules`. Maven reactor edits must be offline,
plugin-free, cycle-safe, comment/unknown-element preserving and committed with
Java edits in one `PatchEngine` transaction. Compatibility behavior for API
`0.2` remains covered.

### Integration and release I1

Closure requires representative Maven, Gradle, mixed and refusal fixtures;
unit/property/golden/protocol/agent/hostile/stress/timeout/crash tests; complete
native packaged mutation acceptance; support matrices and limitations; checksums,
SPDX SBOMs, provenance/attestations and independent downloaded-asset smoke.

## Sequencing

The product-owner priority from 2026-07-17 is:

1. close every J1 Java catalogue and Maven ownership-migration slice;
2. close every T5 TypeScript/JavaScript slice;
3. resume and close the remaining K5 Kotlin slices;
4. close I1 global regression, native packaging and immutable release publication.

The already-qualified shared JVM boundary remains a prerequisite and the completed
Kotlin import row remains frozen green while J1/T5 proceed. A dependency may
require a narrowly scoped shared-kernel correction, but unrelated Kotlin feature
expansion shall not displace the ordered priority. Work remains requirement-first
and independently accepted; this ordering does not authorize broad rewrites.

## Closure gate

The band closes only when every K5, T5, J1 and I1 checkbox is complete with linked
acceptance evidence, all existing Java/TypeScript/Kotlin behavior remains green,
the working tree is clean, artifacts are independently verified, and the final
immutable `v0.7.x` tag is published. Until then the version remains a snapshot and
no roadmap text may describe the complete `0.7.x` band as released.
